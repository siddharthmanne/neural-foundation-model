Assistant2026-05-29T05:52:17.964Z

num_heads is for the shared backbone, not per-modality

4M does not have one encoder/decoder per modality. The architecture is: - One shared encoder (2 or 3 transformer layers of self-attention) - One shared decoder (2 or 3 layers of cross-attention + self-attention) - Per-modality lookup tables: each modality has its own encoder embedding (vocab × dim) and decoder input/output embedding - Per-modality output heads: a small linear projection per modality from dim → vocab

num_heads is the number of attention heads in the single shared transformer. The constraint head_dim = dim / num_heads = 64 (so dim=256 → 4 heads, dim=512 → 8 heads) is just to keep each head's representational capacity fixed as you scale up. None of this is per-modality.

The modality-specific pieces — the embedding tables and output heads — are what make different modalities representable. The transformer itself is completely shared, which is why find_unused_params: true matters: when a batch has no EEG samples, the EEG embedding and head get zero gradient, but DDP still needs to know to sync them.

On increasing num_input_tokens / num_target_tokens

Yes, these are configurable. At patch_size: 16, input_size: 224: - RGB or depth: 14×14 = 196 tokens each - MEG rvq0–3: 32 tokens each (128 total for all 4 layers) - EEG: likely 32 tokens

So your full multimodal budget is 196+196+128 = 520 tokens for MEG, or 196+196+32 = 424 for EEG. With 256 input + 256 target = 512 slots, you're already slightly undersized for the full MEG case — some tokens get dropped or the masking alpha is doing heavy selection.

The practical cost of increasing: - Attention is O(sequence²), so 384→512 input tokens raises attention memory ~78% - For your A100-40GB budget at dim=512, you're already running bs=256 to fit memory — increasing tokens would require reducing batch size further

For your current experiments it's probably not worth it: EEG at 32 tokens fits comfortably in 256, and the Dirichlet masking already handles the case where not all modality tokens appear. If you were to add higher-resolution modalities (448×448 → 784 tokens per image) it would become necessary.

Capacity and attention heads in the larger model

Yes, dim=512 has 8 heads vs 4, and each head has the same 64-dim capacity, so in principle there's more room to learn independent subspaces for visual and neural features. The bottleneck isn't representation capacity — it's that the model hasn't received enough gradient signal to use that capacity well. At 5B tokens and 44M params, the model is compute-suboptimal: it has initialized 44M parameters but the training signal has only partially shaped them. The 17M model at the same token count is much more "full" relative to its capacity.

Data diversity vs. more tokens — what's actually different

These are genuinely distinct:

  • More tokens at fixed dataset size = more epochs over the same images. For your ~300K CC12M + ~26K THINGS, 5B tokens is already ~27 passes through CC12M and ~63 passes through THINGS (due to the 5:1 weighting). Additional passes over identical images give diminishing returns because the model has already learned what those images have to teach. Gradient updates become small and redundant.

  • More diverse data = new images the model hasn't seen, providing genuinely novel gradient signal. Each new image contributes information the model can't have memorized.

For the 512 dim model specifically: if you trained it to ~20B tokens, it would reach roughly the same tokens/param ratio as 256 dim at 5B tokens, and would likely surpass it on CC12M fixed eval. But — with only 300K CC12M images and 26K THINGS images, a 50M parameter model will start memorizing specific image patterns well before 20B tokens. The data diversity ceiling means the practical compute-optimal point is lower than you'd predict from a pure Chinchilla calculation on an unlimited dataset. So it's both: the model needs more tokens and it would benefit from more diverse data, and at large N the diversity wall hits harder than at small N.
