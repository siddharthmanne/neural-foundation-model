For a 44M parameter model on a T4:

  • Model load + checkpoint: ~5-10s
  • Tokenizer loads (both DiVAE, first run downloading from HF): ~60-90s first time, ~10s from cache
  • 4 samples × RGB→Depth generation: ROAR with 1 decoding step is essentially a single forward pass per sample → ~1-2s total for all 4 samples
  • Decoding tokens back to images: ~2-4s per tokenizer call × 3 calls (RGB GT, depth GT, depth pred) × 4 samples → ~5-10s

Total per model: ~30-60s (after first HF download). With 2 models in one run, expect ~90s after tokenizers are loaded.

On CPU: the forward pass for a 44M transformer with batch size 1 is ~30-60s per sample, so 4 samples → ~3-5 minutes per model. The smaller 7M models would be ~30-60s per model on CPU. CPU is fine for debugging with 1 sample (--n_samples 1) but slow for a full comparison grid.

The dominant cost at small batch sizes is actually the DiVAE decode calls (CNN upsampling to 224×224), not the FM transformer — so model size matters less than you'd expect.

Assistant2026-06-03T23:03:46.997Z

No diffusion — it's fully discrete token prediction throughout. Here's the full pipeline:

Stage 0 — Pre-tokenization (already done, stored on disk)

Both RGB images and depth maps were passed through VQ-VAE encoders offline. Each 224×224 image becomes a flat sequence of 196 integer codes (14×14 grid, codebook size 16k for RGB, 8k for depth). These are stored as .npy int16 arrays in the THINGS shard tars. The FM never sees raw pixels — only token indices.

Stage 1 — The FM generates depth token predictions

The 4M foundation model is a masked transformer (encoder-decoder). At inference:

  1. RGB tokens (196 integers) → encoder embeddings → transformer encoder → context
  2. Depth token positions initialized as masked/empty → decoder attends to encoder context
  3. ROAR (Random Order AutoRegression) with decoding_steps=1 predicts all 196 depth tokens in one parallel shot — this is one forward pass (plus a second for CFG, explained below)

Each output token is a logit distribution over the depth codebook (8k entries). Temperature 0.01 makes this near-deterministic (essentially argmax). CFG scale 2.0 runs the forward pass twice — once with RGB conditioning, once without (empty RGB) — and sharpens the conditioning: logits = logits_uncond + 2.0 * (logits_cond - logits_uncond).

Output: 196 depth token indices (integers from 0–8191).

Stage 2 — DiVAE decodes tokens back to an image

The predicted 196 integers are passed to DiVAE.decode_tokens(tokens_2d):

  1. Codebook lookup: each integer → its learned embedding vector
  2. Reshape to (14, 14, D) spatial feature map
  3. CNN decoder (convolutional upsampling) → 224×224 float map

This is a learned CNN decoder, not diffusion. The VQ-VAE codebook was trained to compress depth maps into discrete codes and reconstruct them faithfully — it's the same decoder that was used to tokenize the training data, so the round-trip (image → codes → image) is lossless within the VQ-VAE's capacity.

What you're measuring

The quality gap between GT depth (re-decoded from ground-truth tokens) and predicted depth (decoded from FM-generated tokens) is purely the FM's prediction quality. The DiVAE decoder is the same in both cases, so it cancels out.

The .cpu() call forces a CUDA synchronization point — if the GPU decode is async-queued, .cpu() blocks until it completes before transferring. For 3 decode calls × 4 samples = 12 round-trips, that synchronization overhead adds up.
