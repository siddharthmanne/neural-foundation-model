When you draw 80% of the 22,449 images at random, the trial yield jitters because some images repeat 1× and others 10×+. This run happened to draw a few extra high-repetition images into train.

2. Only ~19% trainable is intentional — mode=adapt freezes the backbone

You ran with the default finetune_mode="adapt" (modal_meg_finetune_brainomni.py:63). That mode does this (trainer.py:47-51):

if mode in ("adapt", "rvq_only"):
    for p in model.encoder.seanet_encoder.parameters():
        p.requires_grad = False
    for p in model.decoder.seanet_decoder.parameters():
        p.requires_grad = False

So: - Frozen (~4.1M params): SEANet conv encoder + decoder — BrainOmni's pretrained backbone (656 h of resting MEG) - Trainable (~960k): sensor layout embedding, cross-attention latent mapping, RVQ codebooks, cross-attn decoder path

The rationale is in the FinetuneConfig docstring: you have ~42 h of THINGS-MEG event-related data vs. 656 h of resting pretraining, so the conservative default protects against catastrophic forgetting of the conv kernels. RVQ + sensor layout absorb the THINGS-specific domain shift.

Mode Trainable When
adapt (default) sensor + cross-attn + RVQ (~960k) conservative — small-data finetune
full everything (~5.05M) only if adapt underfits
rvq_only nothing gradient-updated; RVQ EMA only diagnostic

Run adapt first, log val loss, then run full and compare. That comparison is exactly the kind of experiment the §5 harness leaderboard is designed to settle — don't pick by intuition.

Rough estimate: ~3–5 hours wall clock for the default 10-epoch adapt run on an A10G, with most of that being step-by-step GPU work, not I/O. Math below.

Step count

With batch_size=16 (default) and the splits you saw:

Steps
Train steps / epoch 88,340 / 16 ≈ 5,521
Val steps / epoch 9,816 / 16 ≈ 614
Total steps × 10 ep ~61,350

Per-step time on A10G

The model is small (~5M total params) but each step does a lot per trial: - Resample 281 → 360 timepoints (200 → 256 Hz), pad to 512 → input shape (16, 271, 512) (preprocess.py) - SEANet conv stack over 271 channels × 512 timepoints (frozen but forward + backward through downstream) - Cross-attention + RVQ + SEANet decoder - Loss = time MSE + PCC + amp + phase + commitment (trainer.py:127-133) — the STFT-based amp/phase terms aren't free

On A10G (≈31 TFLOPS FP32, 24 GB VRAM), a model of this shape + the per-channel attention typically lands at 150–300 ms/step in FP32 fwd+bwd at batch 16. That gives:

Per-step 10-epoch wall time
150 ms ~2.5 h
200 ms ~3.4 h
300 ms ~5.1 h

The first epoch will run ~10–20% slower from kernel autotune + cuDNN benchmark warm-up. Don't extrapolate from epoch 1.

The 15 GB / 32 GB memory thing is the bigger risk

Your "15 GB" is probably the on-disk .fif size (which I read as ~33 GB compressed across all 4 subjects in the CLAUDE.md). The thing to worry about is in-memory float32 materialization:

  • load_trials_pooled at modal_meg_finetune_brainomni.py:115-116 loads X_train and X_val as dense tensors.
  • X_train: 88,340 × 271 × 281 × 4 bytes ≈ 27 GB
  • X_val: ~3 GB
  • Modal function has memory=32 * 1024 = 32 GB cap (line 52)

That's basically full. If you OOM, the issue isn't your 15 GB on disk — it's the float32 explosion in CPU RAM. Two cheap fixes: load as float16 (then cast per-batch), or stream from disk with a small worker.

Levers to actually speed it up

In rough order of bang-for-buck:

  1. Mixed precision (bf16/fp16) — wrap _step in torch.autocast + GradScaler. ~1.5–2× speedup. A10G has good bf16 throughput. Easiest single change.
  2. Bigger batch — at 5M params, you almost certainly have VRAM headroom for batch 32 or 64. Doubles throughput per epoch if compute-bound.
  3. --max-train-trials 5000 for a sanity run first. The flag exists at line 119-122. 5k trials × 10 epochs ≈ 10–15 min on A10G — long enough to see whether loss is moving at all without burning 4 hours on a bad LR.
  4. Fewer epochs. Cho 2026's tokenizers used lr=1e-5 and saw the loss flatten within a handful of epochs — 10 is a defensive default, not a must.

What I'd actually do

Before launching the full 10-epoch run, do this in order:

  1. Run with --max-train-trials 5000 --epochs 2 (~5 min). Confirms the pipeline works and loss decreases on a small slice.
  2. If yes, run --lr-sweep --max-train-trials 20000 --epochs 3 to pick LR cheaply (~30–45 min total across 3 LRs).
  3. Then commit to the full 10-epoch run with the winning LR.

The μ-transform codebook is 256 by default (meg_config.py:77, vocab_size: int = 256), matching the Cho 2026 paper (n_tokens: 256, μ=255).

One important caveat about comparing the two numbers directly — they aren't equivalent units:

Tokenizer Nominal codebook Effective capacity Why
μ-transform 256 256 per timestep Single uniform bin lookup
BrainOmni RVQ 512 512⁴ ≈ 6.9 × 10¹⁰ per timestep Residual VQ with num_quantizers=4 (meg_config.py:188)

So BrainOmni's "512" is per-quantizer-level in a 4-level residual stack, not a flat 512-way choice. Per-timestep it's actually picking one of ~7 × 10¹⁰ joint codes — orders of magnitude more expressive than the μ-transform's 256.

That's also why §5.2 codebook utilization is harder to compare across the two: μ-transform "dead code fraction out of 256" means something different from "dead code fraction out of 512 × 4 codebooks." Worth treating them as separate columns in the leaderboard table, not the same metric.

The μ-transform's V is sweepable — meg/CLAUDE.md §9 explicitly calls out {64, 128, 256, 512} as a sweep once the harness is stable.

Yes, this is the right concern — but the good news is the normalization side is already handled by your adapter. The harder distribution mismatch is what fine-tuning is actually for. Let me break it down because there's a subtle bug worth flagging too.

Layer 1: Amplitude normalization — already done, on both sides

Two z-scores happen back-to-back:

Your adapter (preprocess.py:44-48):

def zscore_per_trial(x):
    mean = x.mean(dim=-1, keepdim=True)
    std = x.std(dim=-1, keepdim=True).clamp_min(eps)
    return (x - mean) / std, mean, std

BrainTokenizer internally (external/BrainOmni/braintokenizer/model.py:118-125):

def norm_target(self, x):    # x is (B, C, N, L)
    x = x - x.mean(dim=-1, keepdim=True)
    x = x / (x.std(dim=-1, keepdim=True) + 1e-6)
    return x

BrainOmni was pretrained with per-channel-per-window z-score, applied to the target side of the loss. Your adapter pre-normalizes the input the same way. Since each THINGS trial fits in one window (281 samples → resampled to ~360 → padded to 512, so N=1), per-trial = per-window. Match.

This is why mean/std scale of THINGS vs. BrainOmni's pretraining sets doesn't matter — both get whitened to mean=0, var=1 before any conv ever sees them.

Layer 2: What z-score does NOT fix

These survive normalization and are the real "distribution mismatch":

Channel THINGS-MEG BrainOmni pretrain
Recording type event-related, stimulus-locked 1.4s epochs mostly continuous / resting-state
Spectral content evoked transients, sharp visual ERPs ~100–200ms post-stim dominated by slow oscillations (α/β rhythms)
Sensor layout CTF 271 magnetometers @ specific positions mixed datasets, different MEG systems

Spatial layout is handled explicitly via the pos and sensor_type inputs (sensor_metadata.py) — that's the whole point of the sensor_embed module. So channel geometry is not your problem.

What's left — temporal/spectral structure — is exactly what fine-tuning the cross-attention and RVQ codebooks is supposed to absorb. The frozen SEANet convs encode generic local autocorrelation patterns that transfer; the trainable parts re-learn the THINGS-specific spectral mix.

This is the symmetric MEG version of the LABRAM/EEG concern in neural_tokenizers/CLAUDE.md §4, and it's called out explicitly for Cho 2026 in meg/CLAUDE.md §5 — "their pretraining data is resting-state MEG; THINGS-MEG is event-related. Reconstruction MSE may look fine while the §5.3 linear probe quietly tanks."

Layer 3: Subtle bug — padding interacts badly with the internal z-score

This one I noticed reading through the code. Watch the data flow:

  1. Adapter pads to 512 with zeros (preprocess.py:60-68)
  2. Model unfolds → (B, C, 1, 512) with ~232 zero samples on the right
  3. norm_target(x_unfold) computes mean/std over all 512 samples, including the zeros (model.py:122-124)
  4. The zero pad drags the per-window mean down and inflates std — the target's normalization is biased by padding

The loss masks padded positions out (time_loss = (diff * mask_u).sum() / mask_u.sum()), but the target values themselves on the valid samples are shifted because they were normalized against statistics that included the zeros. So during finetune, the model is being asked to reconstruct a slightly-distorted target.

How bad is this in practice? Probably modest — ~232/512 ≈ 45% of samples are zero, so the mean shift is ~−0.45 × (true mean over valid samples). Since the input was already z-scored to mean≈0 by your adapter, this is actually small. But it's not zero, and it gets worse if you ever reduce window_length or extend trial duration.

Cheap fix: replace the right-pad with masked normalization, or pad with the per-channel mean (which is ~0 after the adapter z-score, so equivalent to current behavior but more robust). Or: pad with the last valid sample (edge mode) — keeps the std estimate clean.

So the answer is

Concern Status
Match amplitude scale ✅ done (double z-score)
Match channel layout ✅ explicit via pos/sensor_type
Match spectral content (resting → evoked) ⚠️ that's why you're finetuning
Padding artifact in target normalization ❌ mild bug, fix when you have time

The thing to actually watch during finetuning: per meg/CLAUDE.md §5, the §5.3 linear probe is the metric most sensitive to distribution mismatch. Reconstruction MSE can look fine while evoked-response information is being thrown away. Track that across adapt vs full vs scratch.
