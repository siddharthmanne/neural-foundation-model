1	"""Reusable train/(val)/test split policy for THINGS-MEG.
2	
3	Why this module exists separately from data.py:
4	  Splits are a *policy* (which trials go where), data.py is a *loader* (how
5	  to read .fif files). The same policy applies whether you're feeding a
6	  μ-transform calibration step or a Cho2026 finetune loop.
7	
8	Why per-image (not per-trial):
9	  THINGS-MEG repeats each of the ~22,449 unique images across multiple
10	  trials per subject. A random per-trial split puts the same image in
11	  train and test → the linear probe (§5.3) gets a free boost from
12	  memorizing image-specific noise. See meg/CLAUDE.md §9.
13	
14	The critical invariant maintained here:
15	  **The TEST set is determined by (seed, test_frac) alone.**
16	  Whether you ask for val_frac=0 (Phase 1) or val_frac=0.1 (Phase 2+),
17	  the test images are the same. This makes the §5 leaderboard meaningful
18	  across phases — every tokenizer is graded on the same held-out trials.
19	
20	When the team-wide things_split.json lands later, the *only* change is the
21	implementation of `split_by_image`. Every downstream caller still gets a
22	`Splits(train, val, test)` object back.
23	"""
24	
25	from __future__ import annotations
26	
27	from dataclasses import dataclass
28	
29	import numpy as np
30	
31	from .meg_config import SplitDefaults
32	
33	
34	@dataclass(frozen=True)
35	class Splits:
36	    """Trial-index arrays for one subject, partitioned by image identity.
37	
38	    Each array holds *trial* indices into a subject's epoch list, not image
39	    IDs. `val` may be a length-0 array if val_frac=0 was requested — callers
40	    should treat it as "no val set" rather than special-casing the field.
41	
42	    Attributes:
43	        train, val, test: int64 arrays of trial indices (sorted ascending).
44	        image_ids_test:   the image IDs that ended up in test, kept around
45	                          for debugging / cross-phase consistency checks.
46	    """
47	
48	    train: np.ndarray
49	    val: np.ndarray
50	    test: np.ndarray
51	    image_ids_test: np.ndarray
52	
53	    def sizes(self) -> dict[str, int]:
54	        return {"train": len(self.train), "val": len(self.val), "test": len(self.test)}
55	
56	
57	def split_by_image(
58	    image_ids_per_trial: np.ndarray,
59	    defaults: SplitDefaults,
60	) -> Splits:
61	    """Partition trial indices into train/val/test grouped by image identity.
62	
63	    Args:
64	        image_ids_per_trial: (n_trials,) int array of THINGS image codes,
65	            one entry per trial in the subject's epoch list. Equivalent to
66	            `epochs.events[:, 2]` from MNE.
67	        defaults: SplitDefaults with (train_frac, val_frac, test_frac, seed).
68	
69	    Returns:
70	        Splits with .train, .val, .test arrays of trial indices. .val is a
71	        length-0 array if defaults.val_frac == 0.
72	
73	    Invariant:
74	        Calling this twice with the same `image_ids_per_trial` and the same
75	        (test_frac, seed) — but DIFFERENT val_frac values — yields the same
76	        .test array. This is what guarantees Phase 1 and Phase 2 evaluate on
77	        the same held-out trials.
78	    """
79	    if image_ids_per_trial.ndim != 1:
80	        raise ValueError(
81	            f"image_ids_per_trial must be 1-D, got shape {image_ids_per_trial.shape}"
82	        )
83	
84	    unique_images = np.unique(image_ids_per_trial)
85	    n_images = len(unique_images)
86	
87	    # Deterministic shuffle of image IDs by seed only — independent of
88	    # val_frac. This is what enforces the cross-phase test-set invariant.
89	    rng = np.random.default_rng(defaults.seed)
90	    shuffled = unique_images.copy()
91	    rng.shuffle(shuffled)
92	
93	    n_test = _round_count(n_images, defaults.test_frac)
94	    n_val = _round_count(n_images, defaults.val_frac)
95	    n_train = n_images - n_test - n_val
96	    if n_train <= 0:
97	        raise ValueError(
98	            f"After taking test ({n_test}) and val ({n_val}) out of {n_images} "
99	            f"unique images, no train images left."
100	        )
101	
102	    # Order: [train | val | test]. Test is the LAST `n_test` images so its
103	    # identity depends only on (seed, test_frac), not on val_frac.
104	    train_images = shuffled[:n_train]
105	    val_images = shuffled[n_train : n_train + n_val]
106	    test_images = shuffled[n_train + n_val :]
107	
108	    train_idx = _trials_for_images(image_ids_per_trial, train_images)
109	    val_idx = _trials_for_images(image_ids_per_trial, val_images)
110	    test_idx = _trials_for_images(image_ids_per_trial, test_images)
111	    return Splits(
112	        train=train_idx,
113	        val=val_idx,
114	        test=test_idx,
115	        image_ids_test=np.sort(test_images),
116	    )
117	
118	
119	def _round_count(n_total: int, frac: float) -> int:
120	    """Round to nearest int, clamped to [0, n_total]."""
121	    return int(np.clip(round(frac * n_total), 0, n_total))
122	
123	
124	def _trials_for_images(
125	    image_ids_per_trial: np.ndarray,
126	    selected_images: np.ndarray,
127	) -> np.ndarray:
128	    """Return sorted trial indices whose image_id is in `selected_images`."""
129	    if len(selected_images) == 0:
130	        return np.empty(0, dtype=np.int64)
131	    mask = np.isin(image_ids_per_trial, selected_images)
132	    return np.nonzero(mask)[0].astype(np.int64)

Rough estimate: ~3–5 hours wall clock for the default 10-epoch adapt run on an A10G, with most of that being step-by-step GPU work, not I/O. Math below.

Step count

With batch_size=16 (default) and the splits you saw:

Steps
Train steps / epoch 88,340 / 16 ≈ 5,521
Val steps / epoch 9,816 / 16 ≈ 614
Total steps × 10 ep ~61,350

Per-step time on A10G

The model is small (~5M total params) but each step does a lot per trial: - Resample 281 → 360 timepoints (200 → 256 Hz), pad to 512 → input shape (16, 271, 512) (preprocess.py) - SEANet conv stack over 271 channels × 512 timepoints (frozen but forward + backward through downstream) - Cross-attention + RVQ + SEANet decoder - Loss = time MSE + PCC + amp + phase + commitment (trainer.py:127-133) — the STFT-based amp/phase terms aren't free

On A10G (≈31 TFLOPS FP32, 24 GB VRAM), a model of this shape + the per-channel attention typically lands at 150–300 ms/step in FP32 fwd+bwd at batch 16. That gives:

Per-step 10-epoch wall time
150 ms ~2.5 h
200 ms ~3.4 h
300 ms ~5.1 h

The first epoch will run ~10–20% slower from kernel autotune + cuDNN benchmark warm-up. Don't extrapolate from epoch 1.

The 15 GB / 32 GB memory thing is the bigger risk

Your "15 GB" is probably the on-disk .fif size (which I read as ~33 GB compressed across all 4 subjects in the CLAUDE.md). The thing to worry about is in-memory float32 materialization:

  • load_trials_pooled at modal_meg_finetune_brainomni.py:115-116 loads X_train and X_val as dense tensors.
  • X_train: 88,340 × 271 × 281 × 4 bytes ≈ 27 GB
  • X_val: ~3 GB
  • Modal function has memory=32 * 1024 = 32 GB cap (line 52)

That's basically full. If you OOM, the issue isn't your 15 GB on disk — it's the float32 explosion in CPU RAM. Two cheap fixes: load as float16 (then cast per-batch), or stream from disk with a small worker.

Levers to actually speed it up

In rough order of bang-for-buck:

  1. Mixed precision (bf16/fp16) — wrap _step in torch.autocast + GradScaler. ~1.5–2× speedup. A10G has good bf16 throughput. Easiest single change.

Layer 3: Subtle bug — padding interacts badly with the internal z-score

This one I noticed reading through the code. Watch the data flow:

  1. Adapter pads to 512 with zeros (preprocess.py:60-68)
  2. Model unfolds → (B, C, 1, 512) with ~232 zero samples on the right
  3. norm_target(x_unfold) computes mean/std over all 512 samples, including the zeros (model.py:122-124)
  4. The zero pad drags the per-window mean down and inflates std — the target's normalization is biased by padding

The loss masks padded positions out (time_loss = (diff * mask_u).sum() / mask_u.sum()), but the target values themselves on the valid samples are shifted because they were normalized against statistics that included the zeros. So during finetune, the model is being asked to reconstruct a slightly-distorted target.

How bad is this in practice? Probably modest — ~232/512 ≈ 45% of samples are zero, so the mean shift is ~−0.45 × (true mean over valid samples). Since the input was already z-scored to mean≈0 by your adapter, this is actually small. But it's not zero, and it gets worse if you ever reduce window_length or extend trial duration.

Cheap fix: replace the right-pad with masked normalization, or pad with the per-channel mean (which is ~0 after the adapter z-score, so equivalent to current behavior but more robust). Or: pad with the last valid sample (edge mode) — keeps the std estimate clean.
