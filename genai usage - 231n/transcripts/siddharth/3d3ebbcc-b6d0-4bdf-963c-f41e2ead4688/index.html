Short answer: yes — validation accepts WebDataset (WDS) format, and "WDS" = WebDataset, the same .tar-shard format your training data uses.

What "WDS format" means

WebDataset stores your dataset as a sequence of POSIX .tar archives ("shards" — your shard_{000..026}.tar). Inside a tar, each sample is a group of files that share a filename stem (the __key__), one file per modality:

shard_000.tar
 ├── image_0001.tok_rgb.npy
 ├── image_0001.tok_depth.npy
 ├── image_0001.tok_meg.npy
 ├── image_0001.meg_mask.npy
 ├── image_0002.tok_rgb.npy
 └── ...

The loader streams through the tar sequentially and regroups files by key into one multimodal sample. The alternative 4M supports is folder format — every modality as loose {image_id}.npy files in parallel directories on disk. Tars win for throughput (sequential reads, far fewer file-open syscalls), which is why training uses them.

Where this happens in your code

The relevant gap: stock 4M's training loader speaks WDS, but its validation loader is folder-only. Your repo patches that. From fourm_dataloader.py:8:

get_val_dataloader → WDS branch when use_wds: true (stock val is folder-only)

So validation accepts WDS only when you opt in with use_wds: true in the val config. fourm_dataloader.py:172-174:

use_wds = cfgs_get("use_wds", dataset_config, dataset_name, train_configs, False)
if not (dataset_type == "multimodal" and use_wds):
    return _ORIG_GET_VAL(...)   # fall back to stock folder loader

When enabled, it routes to _wds_eval_loader, and validate_4m.py:51 sets "use_wds": True for every val task.

Why the val WDS loader is separate from the train one

It reuses the same transform stack (UnifiedDataTransform + your PresenceAwareUnifiedMasking), but the pipeline source differs (fourm_dataloader.py:99-104):

Train WDS Val WDS (_wds_eval_loader)
Shard source shuffled, repeated stream wds.SimpleShardList — in order, no repeat
Determinism random deterministic — every eval sees identical batches
partial batches dropped/repeated partial=False, one clean pass

That determinism is the whole point of a val loader: validation numbers across checkpoints are only comparable if each run scores the exact same samples in the same order. Shuffling/repeating (great for training) would make two eval runs incomparable.

So the design is the modular ideal your standards call for: don't reimplement WebDataset, delegate to upstream 4M and add only the ~50 lines covering the one thing stock 4M lacks (deterministic val-on-tars). The file's docstring states exactly this contract.

Now I can ground this concretely. Here's what each piece means in your 4M/THINGS system.

First: what is a "token"?

4M doesn't operate on raw pixels or raw brain signals — every modality is pre-tokenized into a sequence of discrete integer codes (like word-pieces in an LLM, but produced by a VQ tokenizer). In your setup, one sample (one THINGS stimulus, identified by an image_id) carries several modality token-arrays:

Modality Tokens per sample Where it comes from
tok_rgb 196 224px image ÷ 16 patch = 14×14 grid (neural_constants.py:59)
tok_depth 196 same grid, depth map
tok_meg 512 per trial MEG sources × time × RVQ depth (neural_constants.py:21)
tok_eeg 17 per trial EEG channels (neural_constants.py:27)

So a single sample holds hundreds of tokens across modalities. That total is not what 4M feeds the model in one step — which is the whole point of the next part.

Input tokens vs. target tokens = the masking split

4M trains by masked any-to-any prediction. For each sample, it randomly splits the available tokens into two budgeted groups:

  • Input tokens (num_input_tokens, your config = 256): the tokens the encoder gets to see — the context. Think "the visible half."
  • Target tokens (num_target_tokens, 256): the tokens the decoder must reconstruct — and the loss is computed only on these. Think "the hidden half the model is quizzed on."

Two things to internalize:

  1. These are fixed budgets, not the full token count. A sample has ~600+ tokens; the model sees 256 as input and is asked to predict 256 as target. The rest are simply not used that step. The budget is constant regardless of how many tokens the sample actually contains (extra dropped, short ones padded).
  2. Input and target are disjoint subsets chosen by random masking. The same sample seen in a later epoch gets a different random split — different tokens visible, different tokens predicted. That's why re-seeing a sample is still informative.

Connecting to your model call — in validate_4m.py:137:

loss, mod_loss = model(mod_dict, num_encoder_tokens=n_in, num_decoder_tokens=n_out, ...)

num_encoder_tokens = the input budget, num_decoder_tokens = the target budget. The encoder ingests n_in tokens, the decoder predicts n_out, loss lands on the target.

"Samples seen"

A sample = one dataset item = one image_id with all its modality arrays bundled (this is exactly the WDS regrouping-by-key we discussed earlier). "Samples seen" = how many sample-presentations the model has processed during training:

samples_seen = epoch_size × epochs_trained        # e.g. 10000 × 51

Note "presentations," not "unique samples": epoch_size (10000) is how many samples the trainer iterates per epoch, and across epochs the same samples recur (with fresh random masking each time). So it's a count of exposures, not distinct stimuli.

Putting it together

Each exposure contributes exactly one input-budget and one target-budget worth of tokens, so:

input_seen  = samples_seen × num_input_tokens   = (epoch_size × epochs) × 256
target_seen = samples_seen × num_target_tokens  = (epoch_size × epochs) × 256
total_seen  = input_seen + target_seen

Which is identical to your "tokens-per-epoch × epochs" definition, just expanded.

The one subtlety that ties to your validation tests

The "tokens seen" metric describes the training history baked into the weights — it uses the training budgets (256/256). Your validation tasks use different budgets: 4m_things_val_tasks.yaml sets fixed_eval_input_tokens: 128 / fixed_eval_target_tokens: 128. Those 128/128 govern how the model is quizzed at eval time (so eval numbers are comparable across runs) — they do not enter the tokens-seen count. In other words:

  • Tokens seen = a property of the checkpoint — "how much training did these weights get," computed from training config (256, epoch_size, epochs).
  • Eval token budgets (128) = a property of the validation run — "how hard is the quiz," unrelated to the training count.

So when you validate checkpoint-50 on the rgb2depth task, the reported "tokens seen" answers "these weights were trained on 261.12M tokens," while the task itself shows the model 128 RGB input tokens and scores it on 128 predicted depth tokens. Two independent token counts, easy to conflate — keeping them separate is the key insight.

So, to your questions directly

  • No seeding — is that a problem for val? It's a non-issue, because eval never touches the RNG (idx = 0 regardless of seed). The seed param only matters in training, where you want variation.
  • Is it a fair comparison baseline? Yes. So loss differences across epochs/checkpoints reflect the weights changing, not which trial got sampled.

The one tradeoff worth naming

"Trial 0" is a fixed but arbitrary choice — validation only ever sees the first repetition per image and ignores the rest. That's the deliberate price for reproducibility. Investigated. Here's the honest evaluation: stock 4M does NOT seed validation masking — the token input/target split is random and unseeded. And critically, our in-loop suite inherits this.

How stock 4M validation masking works

The masking is the same UnifiedMasking for train and val. Its randomness, none of it seeded per-eval:

What's chosen How Random?
Which tokens are input vs target noise = torch.rand(num_tokens); argsort(noise) → first input_budget are input (masking.py:248-249) random (global torch RNG, no generator=)
Budget split across modalities Dirichlet .sample() (masking.py:192,221) random
Which Dirichlet (mixture) torch.multinomial / random.randint (masking.py:530,533) random
Number of input/target tokens input_tokens_range fixed iff fixed_eval

The key misconception fixed_eval invites: it only pins the count — pretrain_utils.py:226-228 sets input_tokens_range = (fixed, fixed). It does not fix which tokens are masked. So with fixed_eval you always mask, say, 128 tokens — but a different random 128 each time.

And evaluate() does no seeding (run_training_4m.py:799+) — just model.eval() + iterate. The only torch.manual_seed in the whole run is once at main() start (run_training_4m.py:395-397), never reset before eval. By the time eval runs, training steps have advanced the global RNG, so each eval epoch draws a different mask.

That's why stock prints, verbatim: "Eval stats may vary slightly as the masking applied on images is random." The 4M authors know it's not reproducible; they rely on it being slight because it averages over many tokens/samples.

What this means for your setup

Stacking it with the trial finding from the last question:

Dimension In our in-loop val Reproducible?
Shard order SimpleShardList, no shuffle ✅ fixed
MEG/EEG trial training=False → trial 0 ✅ fixed
Vision token input/target split stock UnifiedMasking torch.rand ❌ random, unseeded

So your concern is well-founded, just located one layer over from where you first looked: the neural trial is a perfect baseline, but the masking split is not. Same-weights eval will wobble epoch-to-epoch.

The fix, if you want a perfect baseline

Because our ValidationSuite.run() owns the eval loop (and uses num_workers=0, so masking runs in the main process), we can make it fully deterministic with one cheap change stock can't easily do: reseed torch/numpy at the top of each run(). Then every epoch masks the identical tokens → exactly comparable.

The tradeoff is a genuine design choice: - Seeded (fixed mask): perfectly reproducible, but you're scoring one arbitrary mask realization — could be a lucky/unlucky draw. - Random + more n_batches: estimates the expected loss; variance shrinks as you average more samples. (This is the stock philosophy.)

For tracking a checkpoint's progress over training, I'd lean seeded — you want curve movement to mean "weights improved," not "different mask."
