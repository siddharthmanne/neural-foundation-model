Short answer: yes — validation accepts WebDataset (WDS) format, and "WDS" = WebDataset, the same .tar-shard format your training data uses.

What "WDS format" means

WebDataset stores your dataset as a sequence of POSIX .tar archives ("shards" — your shard_{000..026}.tar). Inside a tar, each sample is a group of files that share a filename stem (the __key__), one file per modality:

shard_000.tar
 ├── image_0001.tok_rgb.npy
 ├── image_0001.tok_depth.npy
 ├── image_0001.tok_meg.npy
 ├── image_0001.meg_mask.npy
 ├── image_0002.tok_rgb.npy
 └── ...

The loader streams through the tar sequentially and regroups files by key into one multimodal sample. The alternative 4M supports is folder format — every modality as loose {image_id}.npy files in parallel directories on disk. Tars win for throughput (sequential reads, far fewer file-open syscalls), which is why training uses them.

Where this happens in your code

The relevant gap: stock 4M's training loader speaks WDS, but its validation loader is folder-only. Your repo patches that. From fourm_dataloader.py:8:

get_val_dataloader → WDS branch when use_wds: true (stock val is folder-only)

The important edge case: omitting --checkpoint

The flag is optional. If you leave it off: - modal_train.py default is "", validate_4m.py default is None → both falsy - validate_4m.py:169 if checkpoint: is skipped → load_checkpoint is never called - The model runs validation with randomly initialized weights

That's intentional — the help text at line 200 calls it "omit for a pipeline smoke." It exercises the loader → masking → forward → loss plumbing without needing a trained model. The loss numbers are meaningless, but it proves the val path runs. So if you ran validation and got garbage losses, the first thing to check is whether --checkpoint was actually passed.

Why the checkpoint is a CLI arg and not config (the system-level reason)

This is a clean separation of what varies on what timescale:

Concern Lives in Changes
Model architecture main YAML (model, input_size) rarely
What to evaluate (tasks, masking) val-tasks YAML rarely
Which weights to score --checkpoint CLI every single run

You validate checkpoint-last, then checkpoint-best, then epoch-50, then a re-run — all against the same architecture and same tasks. If the checkpoint lived in YAML you'd be editing config (or keeping N near-duplicate YAMLs) on every run. Putting the most-frequently-varying input on the command line is the right boundary. Note also load_checkpoint uses strict=False, so a checkpoint whose architecture is a superset of the eval model (e.g. trained with more domains than a given task uses) still loads — it just reports missing/unexpected keys instead of crashing.

One caveat worth flagging: because the checkpoint is decoupled from the main YAML, nothing checks that --checkpoint's architecture matches main_cfg["model"]. If you point it at a checkpoint from a differently-sized model, strict=False will silently load only the overlapping keys and you'll validate a partially-random model. Part 2 — "tokens seen" per checkpoint (input / target / total)

I understand the goal: for any checkpoint, report how many input tokens, target (output) tokens, and total the model had seen in training by the time those weights were saved — usable both in standalone validation and later in eval-during-training.

I investigated before proposing, and one finding shapes everything: every 4M checkpoint already stores epoch and args (checkpoint.py:102-103), and args carries num_input_tokens, num_target_tokens, and epoch_size. So tokens-seen is derivable from data already on disk:

samples_seen   = (epoch + 1) × epoch_size
input_tokens   = samples_seen × num_input_tokens
target_tokens  = samples_seen × num_target_tokens
total_tokens   = input_tokens + target_tokens

And because your THINGS config sets num_input_tokens without a min_* (so the trainer's token budget is fixed, min == max), this analytic formula is exact for your current runs — it only becomes approximate if you later train with a variable token-count range. That fact is what separates the three plans:

Plan A — Read-time derivation (pure, zero training-side changes). A small token_accounting.py with a pure tokens_seen(epoch, args) -> {input, target, total}; validate_4m prints it in the summary. Nothing touches the trainer; works retroactively on every checkpoint you already have. Downside: recomputed each validation, not stored in the file; approximate if you ever use variable token ranges.

Plan B — Embed at save-time (monkeypatch save_model). Same formula, but a monkeypatch (the pattern you already use in fourm_dataloader.py) injects tokens_seen into the checkpoint dict when it's written. validate_4m reads ckpt['tokens_seen'], falling back to Plan A's derivation for older checkpoints. The number becomes intrinsic to the checkpoint — ideal for eval-during-training and provenance. Downside: one more monkeypatch to maintain; still analytic.

Plan C — True runtime counter (forward hook) + embed. A hook counts the actual token tensor sizes each step and accumulates an exact total, persisted into the checkpoint. Exact even with variable ranges, partial epochs, and odd resume boundaries. Downside: most invasive (hooks the stock loop), per-step overhead, hardest to test — and overkill while your budgets are fixed, since A/B are already exact there.

Testing strategy (TDD, all three): the core is a pure function, so I'd write failing unit tests first with hand-computed numbers (e.g. epoch=0, epoch_size=10000, 256/256 → input=2.56M, target=2.56M, total=5.12M), plus edge cases: 0-based epoch indexing, missing args keys, and the variable-range caveat. Plans B/C add an integration test that the field lands in a written checkpoint.
