Good news — the design already handles exactly this concern. During validation, trial selection is deterministic, not random.

The selection logic is one line, neural_trial_transform.py:91:

idx = int(self.rng.integers(0, n_trials)) if self.training else 0

So the behavior splits on a training flag:

Mode MEG/EEG trial picked Reproducible?
Training uniform random trial each time (rng.integers) no — and intentionally so (augmentation: the model sees different repetitions across epochs)
Validation always trial 0 (else 0) yes — same trial every epoch, every run

And the validation path is wired to training=False. Your in-loop suite builds loaders via _wds_eval_loader, which registers the neural transforms in eval mode — fourm_dataloader.py:147-148:

fourm_neural_modalities.register(training=False)
_set_neural_target_splitter(training=False)  # eval: deterministic trial 0

(Training takes the other branch, line 329-330, training=True → random.)

So, to your questions directly

  • Is it random trial selection? In training, yes (and unseeded — np.random.default_rng(None), fresh random trial per sample per epoch). In validation, no — it's hard-coded to trial 0.
  • No seeding — is that a problem for val? It's a non-issue, because eval never touches the RNG (idx = 0 regardless of seed). The seed param only matters in training, where you want variation.
  • Is it a fair comparison baseline? Yes. Combined with the deterministic WDS val loader (SimpleShardList, no shuffle, no repeat — established earlier), the entire val batch sequence is fixed: image 000 always contributes trial 0 of its MEG and EEG. So loss differences across epochs/checkpoints reflect the weights changing, not which trial got sampled.

The one tradeoff worth naming

"Trial 0" is a fixed but arbitrary choice — validation only ever sees the first repetition per image and ignores the rest. That's the deliberate price for reproducibility. The alternative (averaging the loss over all trials per image) would be a more representative estimate of how the model does on that stimulus, but it's slower and a different design point. If you ever want that, it'd be a change in NeuralTargetSplitter/the trial sampler to loop over trials rather than pick one — not a seeding change. Investigated. Here's the honest evaluation: stock 4M does NOT seed validation masking — the token input/target split is random and unseeded. And critically, our in-loop suite inherits this.

How stock 4M validation masking works

The masking is the same UnifiedMasking for train and val. Its randomness, none of it seeded per-eval:

What's chosen How Random?
Which tokens are input vs target noise = torch.rand(num_tokens); argsort(noise) → first input_budget are input (masking.py:248-249) random (global torch RNG, no generator=)
Budget split across modalities Dirichlet .sample() (masking.py:192,221) random
Which Dirichlet (mixture) torch.multinomial / random.randint (masking.py:530,533) random
Number of input/target tokens input_tokens_range fixed iff fixed_eval

The key misconception fixed_eval invites: it only pins the count — pretrain_utils.py:226-228 sets input_tokens_range = (fixed, fixed). It does not fix which tokens are masked. So with fixed_eval you always mask, say, 128 tokens — but a different random 128 each time.

And evaluate() does no seeding (run_training_4m.py:799+) — just model.eval() + iterate. The only torch.manual_seed in the whole run is once at main() start (run_training_4m.py:395-397), never reset before eval. By the time eval runs, training steps have advanced the global RNG, so each eval epoch draws a different mask.

That's why stock prints, verbatim: "Eval stats may vary slightly as the masking applied on images is random." The 4M authors know it's not reproducible; they rely on it being slight because it averages over many tokens/samples.

What this means for your setup

Stacking it with the trial finding from the last question:

Dimension In our in-loop val Reproducible?
Shard order SimpleShardList, no shuffle ✅ fixed
MEG/EEG trial training=False → trial 0 ✅ fixed
Vision token input/target split stock UnifiedMasking torch.rand ❌ random, unseeded

So your concern is well-founded, just located one layer over from where you first looked: the neural trial is a perfect baseline, but the masking split is not. Same-weights eval will wobble epoch-to-epoch.

The fix, if you want a perfect baseline

Because our ValidationSuite.run() owns the eval loop (and uses num_workers=0, so masking runs in the main process), we can make it fully deterministic with one cheap change stock can't easily do: reseed torch/numpy at the top of each run(). Then every epoch masks the identical tokens → exactly comparable.

The tradeoff is a genuine design choice: - Seeded (fixed mask): perfectly reproducible, but you're scoring one arbitrary mask realization — could be a lucky/unlucky draw. - Random + more n_batches: estimates the expected loss; variance shrinks as you average more samples. (This is the stock philosophy.)

For tracking a checkpoint's progress over training, I'd lean seeded — you want curve movement to mean "weights improved," not "different mask." Want me to add a val_seed to ValidationSuite (reseed at the start of run, default on)? It's ~3 lines and makes in-loop val strictly more comparable than stock.
