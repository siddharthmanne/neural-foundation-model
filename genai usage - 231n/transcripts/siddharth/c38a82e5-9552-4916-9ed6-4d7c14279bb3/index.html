Old SVM rule of thumb: 10 examples/param. Deep-net rule of thumb: ~50–100. Chinchilla for transformers: ~20 tokens/param. We are roughly 5–30× above any reasonable data-sufficiency threshold for this model size.

What this means for pretraining

The reason you pretrain is when you don't have enough task data to fit the model. We have abundant task data for a model this small. So the data-efficiency case for 2a/2b is essentially zero.

Read LaBraM's tokenizer (modeling_vqnsp.py) and the building blocks it depends on (modeling_finetune.py::NeuralTransformer, norm_ema_quantizer.py). Here's the system-level analysis.

What LaBraM-VQNSP actually is

It is not a swap-in upgrade of the Cho2026 AE. It's a fundamentally different design:

  • Per-patch, not per-sample tokens. Trial reshaped to (B, n_chan, n_patches, patch_size=200). One token per (channel, time-patch), not per (channel, timestep) like Cho2026.
  • FFT-domain reconstruction loss. Trains the decoder to predict the amplitude + phase spectrum of each patch, not the raw time-domain signal. Loss = mse(amp) + mse(angle) + emb_loss. This is the "VQ-NSP" — Vector Quantized Neural Spectrum Prediction.
  • Channel embeddings via shared pos_embed[input_chans]. Channel identity is a learned embedding indexed by channel ID. This is the design's main novelty — one model works across channel sets, EEG montages, MEG sensors, etc. Free win for us.
  • L2-normalized EMA codebook (NormEMAVectorQuantizer), k-means initialized, with cluster-size tracking. Strictly better than vanilla VQ-VAE codebooks at avoiding dead codes.
  • Transformer encoder + shallow transformer decoder. Encoder is the heavy lift (12 Blocks); decoder is 3 Blocks. Then decode_task_layer projects to amp/phase bins.

Parameter count

Base config (vqnsp_encoder_base_decoder_3x200x12, embed_dim=200, depth=12+3, num_heads=10, n_embed=8192):

Block Params
Encoder transformer (12 Blocks @ embed_dim=200) ~5.8M
Decoder transformer (3 Blocks @ embed_dim=200) ~1.5M
Patch/temporal embeds, pos+time embeds, cls ~30K
encode_task_layer (200→200→32) ~47K
decode_task_layer ×2 (amp + angle, 200→200→200) ~160K
Quantizer codebook (8192 × 32, EMA-updated, not gradient) ~262K
Total trainable ~7.5M

Compare to Cho2026: ~100K params. LaBraM-base is ~75× larger.

Param/data ratio

Critical: LaBraM is not per-channel like Cho2026. Each trial (271, 281) becomes one training example, not 271. So our effective dataset is 108k trials, not 29M sequences.

Model Params Trials Trials per param
Cho2026 (per-channel) 100K ~29M sequences 290
LaBraM-base (per-trial) 7.5M 108K 0.014

LaBraM-base is ~700× under-fed on THINGS-MEG. Trying to train it from scratch on 108k trials would overfit hard — there are simply not enough trials to constrain that many params from random init. That's why their paper used 2,500+ hours of EEG pretraining.

Scaled down

To get to a sensible regime (say 1-10 trials/param), we want 10k–100k params for an AE on 108k trials. But the patch-token nature of the transformer means it gets many gradient signals per trial (271 channels × ~7 patches = ~1900 patch positions per trial), so it's more sample-efficient than the per-example ratio suggests. A reasonable target is 0.5–2M params:

patch_size:  40 (drops 1 timepoint: 280/40 = 7 patches per channel)

depth_enc:   4   (was 12)

This "MEG-tiny" config: ~1.4M trainable params. Effective examples per param ≈ 108k × 1900 patches / 1.4M ≈ 150. Now we're in a healthy regime — comparable to the Cho2026 ratio.
