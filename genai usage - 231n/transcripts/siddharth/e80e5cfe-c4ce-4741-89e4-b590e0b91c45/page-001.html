15	## 1. Goal
16	
17	Build per-modality tokenizers for a mini-4M model. Each tokenizer maps raw
18	neural signals into discrete token IDs that the 4M transformer can consume.
19	
20	Current target modalities, in priority order:
21	- **EEG** (THINGS-EEG2)
22	- **MEG** (THINGS-MEG)
23	- **Intracortical** (TVSD) — future, same interface
24	
25	The output of this subdir is (a) trained tokenizer checkpoints and (b) a
26	modality-agnostic evaluation harness (`test_tokenizer.py`) that any tokenizer
27	must pass before it is wired into mini-4M.
28	
29	## 2. System-level framing — why VQ-VAE per modality
30	
31	4M's transformer only sees integer token sequences. To add a modality you must
32	jvide three things:
33	
34	- **Encoder** `x -> z` : continuous signal → continuous latents `(L, D)`
35	- **Quantizer** `z -> t` : continuous latents → discrete token IDs in `[0, V)`
36	- **Decoder** `t -> x_hat` : tokens → reconstructed signal (used for eval and
37	  detokenization at inference; not used during 4M training itself)
38	
39	Why this is the right abstraction:
40	- A single masked-token objective works across every modality if every modality
41	  speaks the same `(seq_len, vocab_size)` interface.
42	- Tokens are pre-computed once and cached → 4M training does not pay the
43	  tokenizer's forward cost per step.
44	- The tokenizer can be swapped without touching 4M, so we can A/B different
45	  EEG tokenizers (scratch vs LABRAM, etc.) cleanly.
46	
47	The 4M repo lives at [`../external/ml-4m/`](../external/ml-4m/). The relevant
48	entry points are:
49	
50	- [`external/ml-4m/fourm/vq/vqvae.py`](../external/ml-4m/fourm/vq/vqvae.py) — `VQVAE` class
51	- [`external/ml-4m/fourm/vq/models/`](../external/ml-4m/fourm/vq/models/) — encoder/decoder architectures
52	- [`external/ml-4m/fourm/vq/quantizers/`](../external/ml-4m/fourm/vq/quantizers/) — quantizer implementations (vanilla, `memcodes`, FSQ, …)
53	- [`external/ml-4m/cfgs/default/tokenization/vqvae/human_poses/BMLP1024-BMLP1024_1k_224.yaml`](../external/ml-4m/cfgs/default/tokenization/vqvae/human_poses/BMLP1024-BMLP1024_1k_224.yaml) —
54	  closest template to ours: structured non-image signal, MLP backbone,
55	  `memcodes` quantizer, `codebook_size=1024`, `num_codebooks=8`,
56	  `smooth_l1` loss.
57	- [`external/ml-4m/run_training_vqvae.py`](../external/ml-4m/run_training_vqvae.py) — training entry point
58	
59	When in doubt, mirror the human_poses config rather than the image configs.

79	## 4. Architecture — baseline vs target
80	
81	This subdir trains tokenizers in **two explicit stages**. Stage 1 is a
82	deliberately simple baseline that we expect to be beaten; Stage 2 is the
83	actual production tokenizer that ships into 4M. Do not skip Stage 1 — the §5
84	harness only produces interpretable numbers if there is a baseline to
85	compare against.
86	
87	### Stage 1 — Baseline (start here, all modalities)
88	
89	- Backbone: `BottleneckMLP/B_6-Wi_1024` (encoder and decoder), inputs flattened
90	  to `(channels * time,)`. Simplest faithful backbone for a structured signal,
91	  mirrors 4M's existing non-image tokenizer (human poses).
92	- Loss: `smooth_l1` reconstruction in the time domain only. Single-term, easy
93	  to debug.
94	- Quantizer: `memcodes`, `codebook_size=1024`, `num_codebooks=8`,
95	  `latent_dim=1024`.
96	
97	This baseline is **expected to fail the spectral-MSE metric** in §5.1 — that
98	failure is the motivation for Stage 2.  Get clean numbers across all four §5
99	axes for this baseline before touching anything.
100	
101	### Stage 2 — Target (what actually ships into 4M)
102	
103	Whatever the final tokenizer is, two upgrades over the baseline are
104	non-negotiable:
105	
106	- **A stronger temporal-structure backbone than a flat MLP.** Neural responses
107	  are locally autocorrelated; an MLP on flattened `(C·T,)` ignores that prior.
108	  Either a 1-D CNN over the time axis or a temporal-patch transformer is the
109	  right shape, and either generalizes naturally to MEG / TVSD where `T` varies.
110	- **A spectral / band-power loss term**, added to `smooth_l1`. EEG/MEG task
111	  signal lives in frequency bands (θ/α/β/γ). Pure time-domain L1 can drive
112	  MSE down while still scrambling band power. Likely forms:
113	  - PSD MSE: `||PSD(x) - PSD(x_hat)||²` (Welch estimator)
114	  - Band-power MSE: integrate PSD over named bands and match per band
115	  - Multi-resolution STFT loss (borrowed from audio neural codecs)
116	  - Composite: `L = L1_time + λ_spec · L_spec + λ_band · L_band`, with `λ`
117	    chosen so each term contributes comparable gradient magnitude at init.
118	
119	These two requirements are about *what the tokenizer must learn*, not about
120	which codebase produces it. Two independent decisions then determine the
121	actual Stage 2 we ship — architecture (whose design) and weights (whose data):
122	
123	|                            | **Our own scratch design**       | **LABRAM's architecture**            |
124	|----------------------------|----------------------------------|--------------------------------------|
125	| **Trained from scratch**   | (2a) Generic, modality-agnostic  | (2b) Borrow design, not data         |
126	| **Pretrained + finetuned** | —                                | (2c) Borrow design AND data          |
127	
128	- **(2a) Scratch design, scratch weights.** We design the upgraded backbone
129	  (e.g. 1-D ResNet/U-Net over time + spectral loss) ourselves. Same recipe
130	  works for EEG, MEG, and TVSD. Full control; no external dependency. Risk:
131	  we may rediscover what LABRAM already figured out about EEG.
132	- **(2b) LABRAM architecture, scratch weights.** Adopt LABRAM's
133	  temporal-patch transformer + channel embeddings + multi-objective loss
134	  *as a design template*, but train weights from scratch on THINGS data.
135	  We get the structural priors (patch-wise tokenization, channel-aware
136	  attention, frequency-domain loss already in the recipe) without inheriting
137	  any potential distribution mismatch from LABRAM's pretraining corpus
138	  (mostly resting-state EEG; THINGS is event-related). Cost: more complex
139	  than 2a, and we still have only ~160k trials — the data-efficiency argument
140	  for LABRAM disappears here.
141	- **(2c) LABRAM architecture, LABRAM weights, finetuned.** Full transfer:
142	  load LABRAM's pretrained tokenizer and finetune on THINGS-EEG. This is
143	  the only path that exploits the 2,500+ hrs of EEG LABRAM saw in
144	  pretraining. EEG-only — MEG and TVSD still need 2a or 2b.
145	
146	Stage 2 selection is the open decision in §6. The §5 harness numbers — not
147	authorial preference — decide which one wins.
148	
149	- Per modality: train one tokenizer at a time. Do *not* try to share encoders
150	  across EEG and MEG until both single-modality baselines are working.
151	
152	## 5. Evaluation contract — what `test_tokenizer.py` must enforce
153	
154	A tokenizer is a lossy compressor. `test_tokenizer.py` is the gate that any
155	tokenizer (scratch VQ-VAE, finetuned LABRAM, future TVSD tokenizer) must pass.
156	
157	The harness consumes any object implementing this interface:
158	
159	```python
160	class Tokenizer(Protocol):
161	    def encode(self, x: Tensor) -> LongTensor: ...   # (B, C, T) -> (B, L)
162	    def decode(self, t: LongTensor) -> Tensor:  ...  # (B, L)    -> (B, C, T)
163	    @property
164	    def codebook_size(self) -> int: ...
165	```
166	
167	Four required evaluation axes — implement one module per axis so they can be
168	run independently:
169	
170	1. **Reconstruction fidelity** — encode → decode round trip.
171	   - Per-sample MSE and per-channel Pearson r between `x` and `x_hat`.
172	   - **Spectral MSE**: PSD of `x` vs `x_hat` (Welch). Neural data lives in
173	     bands; if alpha/beta power is wrong the tokenizer is broken even when
174	     time-domain MSE looks fine. This is the most diagnostic single metric.
175	2. **Codebook utilization**.
176	   - Perplexity `exp(H(p))` where `p` is the empirical token distribution.
177	     For `codebook_size=V`, perplexity ≪ V means dead codes.
178	   - Dead-code fraction (codes never selected on the eval set).
179	3. **Downstream linear probe** — does the token sequence still carry the
180	   information we actually care about?
181	   - *The failure mode this catches:* a tokenizer can have low reconstruction
182	     MSE (passes §5.1) and good codebook utilization (passes §5.2) while
183	     having silently thrown away the task-relevant signal. Example: the
184	     tokenizer faithfully reconstructs slow baseline drift (high-variance,
185	     drives MSE down) but loses the small evoked response that actually
186	     distinguishes one stimulus from another. Reconstruction looks great;
187	     the tokens are useless to 4M.
188	   - *The test:* freeze the trained tokenizer. Encode every THINGS trial to
189	     tokens. Train a **logistic regression** (linear probe) on those tokens
190	     to predict the trial's stimulus concept (THINGS has ~1,854 labels — use
191	     a coarser grouping or top-K if 1,854-way is too noisy). Report
192	     top-1 / top-5 accuracy on a held-out split.
193	   - *Why a linear probe and not a strong classifier:* if a linear model can
194	     read the stimulus off the tokens, the information is **linearly
195	     decodable** — which is what 4M's transformer needs from its input
196	     embeddings. A strong nonlinear classifier could rescue tokens that are
197	     entangled in ways 4M can't undo cheaply, so it would overstate
198	     tokenizer quality.
199	   - *Token representation for the probe:* either one-hot pool-then-flatten
200	     (count of each code per trial; bag-of-tokens), or embed each token via
201	     the VQ-VAE's codebook lookup and mean-pool over the sequence. Try
202	     both — they measure subtly different things (presence vs. average
203	     content).
204	   - *Reference points to bracket the score:* run the same probe on
205	     **(upper)** the raw flattened signal `x` itself (ceiling — all info is
206	     there) and on **(lower)** uniformly random tokens of the same shape
207	     (floor — no info). A good tokenizer should land closer to the upper
208	     bound than to the lower; if it sits near the floor, the tokens are not
209	     informative no matter how pretty the reconstruction looks.
210	
211	4. **Token-sequence statistics** — is the token *sequence* learnable by a
212	   transformer, or did the tokenizer collapse to a degenerate code?
213	   - *The failure mode this catches:* a tokenizer can pass §5.1, §5.2, and
214	     §5.3 while emitting sequences so predictable that 4M's masked-token
215	     objective learns nothing. Example: suppose the tokenizer outputs an
216	     8-token sequence per trial, but for any given trial it emits *the same
217	     code 8 times* (e.g. `[42, 42, 42, ..., 42]`), with the specific code
218	     varying by stimulus class. The decoder can still reconstruct (mapping
219	     code → trial), the codebook is fully used (§5.2 sees 1,024 distinct
220	     trial-level codes), and the probe works perfectly (one code = one
221	     stimulus). But 4M is trained to predict masked tokens given context —
222	     if `t₂` is always equal to `t₁`, the prediction task is trivial and the
223	     model learns nothing transferable.
224	   - *What to compute:*
225	     - **Unigram entropy** `H(t) = -Σ p(t) log p(t)` — the entropy of the
226	       marginal token distribution. Already implied by perplexity in §5.2,
227	       but report it explicitly per-position too (some positions may collapse
228	       while others don't).
229	     - **Bigram conditional entropy** `H(tᵢ | tᵢ₋₁)`. This is the key number.
230	       If a transformer's next-token prediction has nothing left to learn
231	       after seeing one neighbor, this collapses toward 0. **Healthy
232	       tokenizers have `H(tᵢ | tᵢ₋₁) ≈ H(t)` — i.e. transitions are nearly
233	       independent of the previous token.** A large gap `H(t) - H(tᵢ | tᵢ₋₁)`
234	       means the sequence is highly redundant.
235	     - **Run-length distribution** — how long are runs of the same code?
236	       Long runs are the most visually obvious form of collapse. Report mean
237	       run length and the fraction of trials with run length ≥ 2.
238	   - *Pass criterion:* bigram-vs-unigram entropy gap small (say <20%), no
239	     position with degenerate marginal, mean run length near 1. If any of
240	     these fail, the tokenizer will technically train 4M to convergence but
241	     the resulting model will not generalize — fix the tokenizer first.
242	
243	The harness should report all four for each tokenizer in a single table so we
244	can compare tokenizers side by side. Treat this as the source of truth for
245	"is tokenizer X better than tokenizer Y" — not training-loss curves.

19	## 2. What 4M expects from a new modality — the binding constraint
20	
21	Before designing anything, the codebook size we ship has to fit 4M's modality
22	registration model. Read [`../../external/ml-4m/fourm/data/modality_info.py`](../../external/ml-4m/fourm/data/modality_info.py)
23	— that file is the source of truth.
24	
25	Concrete facts that drive the design:
26	
27	- **No fixed vocab across modalities.** Each modality declares its own
28	  `vocab_size` in `MODALITY_INFO`. Image-token modalities range from 4096
29	  (semseg) to 16384 (RGB); text/sequence is 30_000; `human_poses` "borrows"
30	  the 30k text vocab via `shared_vocab: ['caption']`. We are free to pick.
31	- **But vocab IS fixed at registration time.** 4M allocates a learnable
32	  embedding `[vocab_size, d_model]` for the modality. Adding tokens
33	  mid-training breaks that table. **Pick V before computing the token cache
34	  that 4M trains on; commit to it.**
35	- **`type: 'seq'` is the right modality type for MEG**, not image-grid.
36	  Mirror `human_poses` (`modality_info.py:172`) more than `tok_rgb`. The
37	  per-channel-per-timestep token stream is a sequence, not a 2D grid.
38	  Pose-tokenizer registration template:
39	  `vocab_size`, `encoder_embedding = SequenceEncoderEmbedding(...)`,
40	  `decoder_embedding = SequenceDecoderEmbedding(...)`,
41	  `max_length`, `num_channels`, `type: 'seq'`,
42	  `id: generate_uint15_hash('meg_tokens')`.
43	- **Multi-codebook (`num_codebooks > 1`) is supported by the `memcodes`
44	  quantizer** (used by `human_poses`, 1024 × 8). Cho 2026's tokenizers use
45	  a single codebook per channel. Stick with **`num_codebooks = 1`** for MEG
46	  — there is no reason to fight the upstream tokenizer's design.
47	- **Whether to share vocab with another modality is a 4M-side decision.**
48	  Human_poses reuses the text vocab because pose tokens are mapped into the
49	  WordPiece range; this lets a 4M decoder generate poses with the same head
50	  as captions. We do **not** want that for MEG — it couples MEG to text
51	  training. **Register a dedicated `meg_tokens` modality with its own
52	  vocab.**
53	
54	So the headline constraint: pick `V` and `max_length` (= per-trial token
55	sequence length) when you write the tokenizer, because the modality
56	registration that goes into 4M has to declare both.
57	
58	Sensible starting points, derived from Cho 2026 + the verified THINGS-MEG
59	data spec in §3 + 4M conventions:
60	
61	| Tokenizer            | V                              | num_codebooks | per-channel max_length |
62	|----------------------|--------------------------------|---------------|------------------------|
63	| Phase 1 (μ-transform)| 256 (paper default)            | 1             | 281 (= trial timepoints) |
64	| Phase 2 (Cho2026 learned) | 128 (paper) → V* after refactor | 1        | 281                    |
65	| Phase 3 (BrainOmni)  | TBD                            | TBD           | TBD                    |
66	
67	Registration shape (per-channel sequence, **not** flattened):
68	`num_channels = 271`, `max_length = 281`. The alternative — flatten to one
69	giant sequence of 271 × 281 = 76,151 tokens per trial — blows past every
70	sensible context length and discards the channel/time factorization 4M can
71	exploit. Mirror `human_poses`, which keeps `num_channels = 207` separate

141	## 4. Why three phases, in this order
142	
143	The phases trade off cost vs. expected fidelity. We deliberately start with
144	the cheapest possible MEG tokenizer to (a) prove the §5 harness is correctly
145	wired for MEG-shaped data, (b) put a floor on the leaderboard, and (c) avoid
146	prematurely committing to a deep architecture before we know what "good
147	enough" looks like on this specific THINGS-MEG distribution.
148	
149	| Phase | Tokenizer                                | Learns? | Cost     | Role                         |
150	|-------|------------------------------------------|---------|----------|------------------------------|
151	| 1     | μ-transform (Cho 2026 baseline)          | No¹     | seconds  | Floor + harness sanity check |
152	| 2     | Cho 2026 learnable AE (causal / non-causal) | Yes  | hours-GPU | Likely production tokenizer |
153	| 3     | BrainOmni tokenizer, finetuned           | Yes     | hours-GPU | Stretch / comparison         |
154	
155	¹ "Non-learnable" means no gradient-descent weights — but the μ-transform
156	still has dataset-fit parameters (clip thresholds and max-abs scaler) that
157	**must** be re-estimated on THINGS-MEG. See §5 for what "recalibration"
158	actually means.

160	## 5. What "recalibration" / "finetuning" actually mean here
161	
162	The Cho 2026 paper warns that both tokenizer families need dataset-specific
163	adaptation. The mechanism is different for each.
164	
165	### Phase 1 — μ-transform recalibration (no gradient descent, but still a `fit` step)
166	
167	The μ-transform pipeline (paper §3.3.1):
168	
169	1. **Clip** the input to a percentile range `[q_lo, q_hi]` to suppress
170	   outliers.
171	2. **Max-abs scale** to `[−1, 1]`: `x' = x / s` with `s = max|x|`.
172	3. **μ-law compand**: `F(x') = sgn(x') · ln(1 + μ|x'|) / ln(1 + μ)`.
173	4. **Uniform bin** the compressed values into `V` bins → integer token IDs.
174	
175	The reference config Cho 2026 ships
176	([`../../external/Cho2026_Tokenizer/models/tokenizer/mu_transform/config.yml`](../../external/Cho2026_Tokenizer/models/tokenizer/mu_transform/config.yml))
177	is:
178	
179	```yaml
180	mu: 255
181	n_tokens: 256
182	normalization: max_abs
183	```
184	
185	There are **no learned weights**, but two parameters are dataset-dependent
186	and must be fit on a THINGS-MEG training split, frozen, and reused at
187	inference:
188	
189	- `s = max|x|`: the per-channel (recommended) or global max-abs scaler.
190	- `q_lo, q_hi`: clip thresholds (defaults around the 0.5 / 99.5 percentiles).
191	
192	`μ` and `V` are hyperparameters, not fit from data — start with μ=255,
193	V=256.
194	
195	Concretely: there is no `.train()` loop. There is a `fit(train_dataset) →
196	calibration.json` step that produces a small JSON sidecar with
197	`{s_per_channel, q_lo, q_hi, mu, V, channel_mode}`. At inference we load
198	that JSON and apply the stateless transform. Treat the calibration JSON as
199	a versioned tokenizer checkpoint.
200	
201	> Verified §3 fact that matters here: all 271 channels are `mag`, so the
202	> grad-vs-mag amplitude disparity problem does not apply. Per-channel `s`
203	> is still the safer default (inter-sensor amplitude variance is real even
204	> within one channel type), but the bin-saturation risk is much lower than
205	> it would be on a mixed-sensor dataset.
206	
207	### Phase 2 — Cho 2026 learnable tokenizer (actual gradient training)
208	
209	The Cho 2026 learnable tokenizer (paper §3.2) is an **autoencoder**, not a
210	VQ-VAE: encoder is GRU → Dense → LayerNorm producing per-time-step logits
211	over a vocab of size `V`; decoder is a bank of `V` 1-D conv kernels (causal
212	or non-causal) summed with learned per-token gains. Trained end-to-end with
213	MSE + an annealed soft-to-hard argmax relaxation, then unused tokens are
214	pruned ("token refactorization").
215	
216	Reference configs they ship
217	(`../../external/Cho2026_Tokenizer/models/tokenizer/{causal,noncausal}/config.yml`):
218	
219	```yaml
220	sequence_length: 200
221	n_channels: 52         # NB: 52 source-parcellated channels, NOT sensor
222	n_tokens: 128
223	token_dim: 10
224	rnn_n_units: 128
225	token_kernel_padding: causal  # or `same` for non-causal
226	training: { lr: 1e-5, batch: 32, epochs: 40, anneal_stages: 40 }
227	```
228	
229	Three things the configs tell us, in light of the verified data spec:
230	
231	1. **They trained on source-reconstructed data (52 parcels), not sensor
232	   space.** Our THINGS-MEG is **271-sensor**. The tokenizer is applied
233	   per-channel and the GRU has no channel-conditioning, so weights *can*
234	   be reused on sensor data, but the *amplitude distribution* of source
235	   parcels (after `osl_dynamics.standardize()`) is not identical to sensor
236	   amplitudes. This is the symmetric MEG version of the LABRAM/THINGS
237	   distribution mismatch in the parent doc.
238	2. **Their sequence_length = 200 samples ≈ 0.8 s.** Our trials are 281
239	   samples (1.4 s). Either reduce our windows to 200 samples (matches their
240	   training distribution → reuse pretrained weights cleanly) or retrain
241	   with `sequence_length = 281`. Strong prior: **window to 200** for 2a/2b
242	   to keep the pretrained encoder in-distribution; only change
243	   `sequence_length` for 2c.
244	3. **Their LR (1e-5) is very small.** Tells us finetuning is intended to be
245	   conservative — they expect the loss landscape to be flat around the
246	   pretrained optimum. Start with that LR and only increase if the loss is
247	   not moving after 1 epoch.
248	
249	"Finetuning on our dataset" then has three concrete recipes:
250	
251	- **2a.** Load pretrained weights, apply on THINGS-MEG **as-is** (zero
252	  finetune). Establishes the floor for what their pretrained tokenizer
253	  buys us on distribution-shifted data.
254	- **2b.** Load pretrained weights, continue MSE training on THINGS-MEG
255	  (lr=1e-5, anneal κ already at 0 → hard tokens — or restart anneal).
256	  This is the "actual finetune."
257	- **2c.** Take their architecture, train from scratch on THINGS-MEG. This
258	  is needed if (a) pretrained weights fail the §5 harness even after 2b,
259	  (b) we want to change `sequence_length` or `n_tokens`, or (c) the
260	  source-vs-sensor mismatch turns out to dominate. With ~108k trials
261	  pooled across P1–P4 (see §3 — much more than the parent doc estimated),
262	  from-scratch is more viable than first assumed.
263	
264	Run **2a → 2b → 2c** in order; stop as soon as the harness numbers are
265	acceptable. Do not skip 2a — the gap between 2a and 2b is the answer to
266	"how much did pretraining buy us."
267	
268	> System-level note: their pretraining data is **resting-state** MEG;
269	> THINGS-MEG is **event-related** (stimulus-locked). Expect the pretrained
270	> kernels to capture continuous oscillatory structure well and to
271	> underweight evoked transients. Reconstruction MSE may look fine while
272	> the §5.3 linear probe quietly tanks — watch that metric closely after
273	> each of 2a/2b/2c.
274	
275	### Phase 3 — BrainOmni (deferred)
276	
277	Multi-modal tokenizer. Not the path forward unless phase 2 has clear
278	distribution-mismatch symptoms AND BrainOmni's pretraining corpus includes
279	event-related MEG. Revisit after phase 2 numbers land.

349	## 8. Evaluation contract
350	
351	Identical to the parent §5. The same `test_tokenizer.py` harness must pass
352	for each of the three MEG tokenizers before it is wired into 4M:
353	
354	1. Reconstruction fidelity (time MSE + per-channel Pearson + **spectral
355	   MSE**). Spectral MSE is the diagnostic metric for MEG — band power
356	   (especially α/β) is what carries task signal. PSD range: 1–100 Hz
357	   (Nyquist for 200 Hz data is 100 Hz).
358	2. Codebook utilization (perplexity, dead-code fraction). The μ-transform's
359	   dead-code fraction is the metric to watch; Cho 2026 prunes dead codes
360	   ("token refactorization") so its effective `V*` can be reported
361	   alongside nominal `V`.
362	3. Linear probe on THINGS labels. With 22,449 image-level codes / 1854
363	   concepts / ~27 superordinate categories, use the THINGS
364	   **superordinate** categories as the primary probe target, 1854-way
365	   concept ID as a secondary, image ID never (too noisy at 27k trials per
366	   subject). The `image_id → concept_id → superordinate` mapping comes
367	   from THINGS metadata, not the .fif files — add it under
368	   `meg/data.py`.
369	4. Token-sequence statistics (unigram entropy, bigram conditional entropy,
370	   run-length). The μ-transform is at particular risk of degenerate run
371	   lengths on slow drifts — watch this.
372	
373	MEG-specific harness parameters (PSD frequency range, named bands) go in
374	`meg_config.py` and are passed into the harness as `EvalConfig`. Do not
375	hardcode them.

        ratios,

        last_kernel_size,
        n_dim,

        self.n_dim = n_dim

            quantize_optimize_method=quantize_optimize_method,

            last_kernel_size=last_kernel_size,

            torch.nn.init.trunc_normal_(m.weight, std=0.02)
            if m.bias is not None:
                nn.init.constant_(m.bias, 0)
        elif isinstance(m, RMSNorm):
            if isinstance(m.weight, nn.Parameter):
                nn.init.constant_(m.weight, 1.0)
        elif isinstance(m, nn.Embedding):
            nn.init.trunc_normal_(m.weight, std=0.02)
        elif isinstance(m, nn.Parameter):
            nn.init.trunc_normal_(m, std=0.02)

    def get_parameters_groups(self, lr: float, codebook_lr: float, weight_decay: float):
        normal_params = []
        no_decay_params = []
        codebook_params = []
        for n, p in self.named_parameters():

                if "norm" in n or n in [
                    "sensor_embed.sensor_embedding_layer.weight",

            {"params": no_decay_params, "lr": lr, "weight_decay": 0.0},
            {"params": codebook_params, "lr": codebook_lr, "weight_decay": 0.0},

    def unfold(self, x: torch.Tensor, overlap_ratio: float = 0.0):
        if x.shape[-1] < self.window_length:
            x = F.pad(x, pad=(0, self.window_length - x.shape[-1]))
        if overlap_ratio > 0.0:
            stride = int(self.window_length * (1 - overlap_ratio))
            right_remain = (x.shape[-1] - self.window_length) % stride
            if right_remain > 0:
                x = F.pad(x, pad=(0, stride - right_remain))
        return x.unfold(
            dimension=-1,
            size=self.window_length,
            step=int(self.window_length * (1 - overlap_ratio)),
        )

    def norm_target(self, x: torch.Tensor):
        """
        x: B C N L
        """
        x = x.float()
        x = x - x.mean(dim=-1, keepdim=True)
        x = x / (x.std(dim=-1,keepdim=True)+1e-6)
        return x

    def add_noise(self, x: torch.Tensor):
        return x + torch.randn_like(x) * 0.1

    def forward(
        self, x: torch.Tensor, pos: torch.Tensor, sensor_type: torch.Tensor, **kwargs

        """
        x: B C (N L)
        pos: B C 6
        sensor_type: B C
        """
        x = self.unfold(x)

        sensor_embedding = self.sensor_embed(pos, sensor_type)
        random_index = torch.randperm(x.shape[1], device=x.device)
        x = x.index_select(dim=1, index=random_index)
        sensor_embedding = sensor_embedding.index_select(dim=1, index=random_index)
        n_mask_channel = max(int(x.shape[1] * self.mask_ratio), 1)
        feature = self.encoder(
            self.add_noise(x[:, n_mask_channel:]),
            sensor_embedding[:, n_mask_channel:],
        )

        feature, indices, commitment_loss = self.quantizer(feature)

        x_rec = self.decoder(feature, sensor_embedding)

        x_rec = x_rec.float()
        x = self.norm_target(x)

        time_loss = get_time_loss(x_rec, x)
        pcc = get_pcc(x_rec, x)
        amp_loss, phase_loss = get_frequency_domain_loss(x_rec, x)
        return {
            "loss": time_loss
            + torch.exp(-pcc)
            + commitment_loss
            + amp_loss
            + 0.5 * phase_loss,
            "time_loss": time_loss.detach(),
            "pcc": pcc.detach(),
            "amp_loss": amp_loss.detach(),
            "phase_loss": phase_loss.detach(),
            "commitment_loss": commitment_loss.detach(),
            "judge_loss": (
                time_loss
                + torch.exp(-pcc)
                + commitment_loss
                + amp_loss
                + 0.5 * phase_loss
            ).detach(),
        }, indices

    def visualize(
        self, x: torch.Tensor, pos: torch.Tensor, sensor_type: torch.Tensor, **kwargs
    ):
        """
        x: B C (W T)
        pos: B C 6
        sensor_type: B C
        """

        sensor_embedding = self.sensor_embed(pos, sensor_type)
        feature = self.encoder(x, sensor_embedding)
        feature, indices, commitment_loss = self.quantizer(feature)
        x_rec = self.decoder(feature, sensor_embedding)
        return {
            "x": self.norm_target(x),
            "x_rec": x_rec.float(),
            "sensor_type": sensor_type,
        }

    @torch.no_grad()
    def tokenize(
        self,
        x: torch.Tensor,
        pos: torch.Tensor,
        sensor_type: torch.Tensor,
        overlap_ratio: float,
        **kwargs,
    ):
        """
        x: B C T
        pos: B C 6
        sensor_type: B C
        """
        self.eval()
        x = self.unfold(x, overlap_ratio=overlap_ratio)
        sensor_embedding = self.sensor_embed(pos, sensor_type)
        feature = self.encoder(x, sensor_embedding)
        feature, indices, commitment_loss = self.quantizer(feature)
        feature = rearrange(feature, "B C N T D->B C (N T) D")
        indices = rearrange(indices, "B C N T Q -> B C (N T) Q")
        return feature, indices

        del self.quantizer

                    "weight_decay": this_weight_decay,
                    "params": [],
                    "lr_scale": layer_decay,
                }

            parameter_groups[group_name]["params"].append(p)

        return list(parameter_groups.values())
