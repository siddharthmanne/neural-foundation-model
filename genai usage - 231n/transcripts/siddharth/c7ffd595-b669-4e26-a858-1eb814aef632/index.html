The trap in the Exp 1 vs Exp 1.5 comparison (the one I claimed as "3.6× cleaner" in the notes — that phrasing is misleading and I should fix it):

Both runs z-score per trial per channel inside the BrainOmni preprocess, so the input has unit variance in both cases. The MSE drop is explained mechanically by the noise floor of the target, not by the tokenizer doing a categorically better job:

  • For a single z-scored trial: ~90% of unit variance is per-trial noise. Tokenizer's bottleneck preserves the slow/structured component and drops noise → ‖x − x̂‖² ≈ ‖noise‖² ≈ 0.9–1.0.
  • For an averaged z-scored trial (N≈4–48 reps): noise variance reduces by 1/N before z-scoring, so post-z-score the noise fraction of unit variance is much smaller. The tokenizer drops the same kind of stuff, but there's less of it to drop → MSE ≈ 0.3.

The cleanest test that the tokenizer isn't actually reconstructing better: channel Pearson r barely moves, 0.870 (Exp 1) → 0.859 (Exp 1.5). If reconstruction were genuinely 3.6× better, correlation should rise. It doesn't — it slightly drops. The MSE delta is dominated by "less noise in the target," not "more signal in the output."

This strengthens (not weakens) the Exp 1.5 verdict, by the way: if the tokenizer were actually reconstructing the averaged signal better and the probe still couldn't decode category, that would be even more damning for the reconstruction objective. The current reading — "averaging reaches raw features but tokens don't benefit" — is robust to this nuance.

Good question. Direct answer: yes, every cell in the existing tables is a top-1 metric — either unweighted top-1 (the "Linear, unweighted" row) or class-weighted balanced accuracy (the rest), which is also top-1 in spirit (it just reweights by class). Each JSON also computes top-5 for free; we just don't render it.

Where top-5 makes sense:

Label space n_classes Top-5 chance Top-5 meaningful?
category27 27 5/27 ≈ 18.5% (uniform) ✅ Yes
animacy 2 100% trivially ❌ No — verified in JSON: top5_*_mean = 1.0 everywhere
subject 4 100% trivially ❌ No — k=5 > n_classes

So expanding to top-5 only buys you one of three tables. For animacy and subject the column is mechanically 100%.

What top-5 actually measures and when it helps:

It loosens the bar from "name the right class" to "right class in the top-5 short list." Diagnostically it distinguishes two failure modes that look identical under top-1:

  1. Tokens have zero category info → at random under both top-1 and top-5
  2. Tokens have weak/diffuse category info → at random under top-1 but lift above random floor under top-5 (right neighborhood, wrong specific class)

So it's most useful as a tiebreaker when top-1 is ambiguous (i.e. tokens hovering just above the random column).

What our top-5 numbers actually say (Exp 1.5, category27, linear class-weighted column):

Feature top-1 (bal_acc) top-5
tokens_all 3.35% ± 0.90 18.30% ± 3.75
tokens_rvq0 3.88% ± 2.74 18.21% ± 4.85
raw 6.01% ± 2.37 35.88% ± 1.77
random 4.02% ± 2.24 21.94% ± 3.94

Same story at both granularities: tokens ≤ random; raw lifts well above. The top-5 column doesn't change the verdict — it just confirms the floor is solid at multiple thresholds (failure mode #1 above).

Practical recommendation: I'd skip adding a top-5 column by default. Reasons:

  • For 2/3 label spaces it's trivially 100%
  • For the one where it's meaningful (category27), top-5 says the same thing top-1 says — tokens at random — so it doesn't carry independent information here
  • More columns make the table harder to read

The case for adding it: if a future checkpoint (e.g. 3b_avg or Cho2026) lands tokens slightly above random at top-1 on category27 and we want to test "real but coarse signal vs noise," top-5 is the right diagnostic to add at that point. Until then it's just confirmatory.

Each row uses a different label space — they're three independent retrieval computations on the same underlying tokens but with different ground-truth label arrays.

How the pipeline works:

  1. modal_meg_eval.py builds labels from --probe-label-space (one of category27, animacy, subject)
  2. The same labels tensor is handed to both compute_probe_metrics() and compute_retrieval_metrics()
  3. Inside retrieval.py:68-69: y = labels.to(torch.long).cpu(); n_classes = int(y.max().item()) + 1 — the metric just inherits whatever label space was passed

So in the table at meg_tokenization.md:99-103:

Table row Label space n_classes Chance
Category27 THINGS 27-way superordinate (SuperordinateMapping) 27 3.70%
Animacy living vs. non-living from same superordinate file (AnimacyMapping) 2 50%
Subject which of P1–P4 recorded the trial 4 25%

The tokens (and their cosine-similarity matrix) are identical across the three rows — that's a B×B matrix from one tokenizer forward pass. The only thing that changes per row is which labels we use to ask "is the retrieved neighbor a hit?" So:

  • Category27 = 3.71% ≈ chance → tokens do not cluster trials by 27-way superordinate
  • Animacy = 50.5% ≈ chance → tokens do not cluster by living/non-living
  • Subject = 50.3% vs 25% chance → tokens do cluster by subject

That's why I was careful to call animacy and category27 separate questions even though their queries (the tokens) are the same — the targets (labels) differ entirely.

One related artifact worth knowing: the sample counts differ across rows because the cat27 and animacy mappings only cover ~50% of THINGS concepts (only concepts with a clean single Bottom-up superordinate label). That's why Category27 (~5032) and Animacy (~5032) but Subject (~10k) in the same table — subject labels apply to every trial.
