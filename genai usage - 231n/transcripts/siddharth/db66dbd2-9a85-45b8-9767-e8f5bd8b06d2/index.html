MEG Tokenization with BrainOmni — Deep Dive

Let me start at the highest level and zoom progressively in, then explain the freeze/finetune decision once you understand what the pieces are.

1. What "tokenization" even means in this setting

4M (the multimodal transformer downstream) only knows how to read integer ID sequences. RGB images come in as integer codes, text as WordPiece IDs. To let 4M see MEG, we need a function:

raw MEG trial  (271 channels × 281 timepoints of floats)  →  some short sequence of integers

And, importantly, we need an inverse — integers → reconstructed MEG — because the only way we can train the tokenizer without labeled "ground truth tokens" is by demanding that the round-trip x → tokens → x_hat stays close to x. This is exactly the autoencoder + bottleneck idea, plus the bottleneck happens to be discrete (a finite codebook).

So a "neural tokenizer" is just a VQ-style autoencoder: - Encoder squashes the raw signal into a small latent representation - Quantizer snaps each latent vector to its nearest entry in a learned codebook of V vectors → the index becomes the token - Decoder takes those codebook vectors and reconstructs the original signal

The quality criterion is "tokens that 4M can learn from." That is what §5 of the parent doc tries to measure.

2. BrainOmni's BrainTokenizer at a high level

BrainOmni (OpenTSLab/BrainOmni) was pretrained on 656 hours of resting-state MEG/EEG across many sensor geometries. Its tokenizer is called BrainTokenizer. It has four sub-modules; see external/BrainOmni/braintokenizer/model.py:15-66 for the constructor.

BrainTokenizer
├── sensor_embed     : BrainSensorModule       — "where is each sensor and what kind is it?"
├── encoder          : BrainTokenizerEncoder   — sensor space → latent neuro space
│   ├── seanet_encoder  : SEANetEncoder            (1-D conv stack, per channel)
│   ├── neuros          : nn.Parameter(N_neuro, D) (learnable latent "source" queries)
│   ├── backwardsolution: cross-attention          (sensors → neuros)
│   └── k_proj          : Linear
├── quantizer        : BrainQuantizer          — RVQ over latent neuro features
│   └── rvq          : 4 residual VQ layers, V=512 codes each
└── decoder          : BrainTokenizerDecoder   — latent neuro space → sensor space
    ├── forwardsolution: cross-attention            (neuros → sensors)
    └── seanet_decoder : SEANetDecoder              (1-D transposed conv stack, per channel)

The conceptual move that makes BrainOmni interesting: instead of trying to model 271 raw MEG channels directly, it introduces an intermediate "latent neural source" space. The encoder asks 16 imagined source-like queries to attend over the 271 sensors; the decoder asks the 271 sensors to attend over the 16 latent sources. This is structurally similar to how MEG source-reconstruction works (a small number of "neural sources" projected to many sensors via a mixing matrix) — but the mixing is learned via attention.

That n_neuro = 16 is the secret reason BrainOmni produces such a small token sequence: 271 sensors get compressed into 16 latent channels before tokenization.

3. The pieces, one by one

3a. BrainSensorModule (the "sensor embed")

external/BrainOmni/model_utils/module.py:11-32

Input: for each sensor c, a 6-D vector pos = [x, y, z, ox, oy, oz] (sensor 3-D position + 3-D orientation) and an integer sensor_type (1 for mag, etc.).

Output: a per-sensor embedding vector of dim n_dim.

Why it matters: BrainOmni was pretrained on many MEG/EEG headsets. The sensor-embed lets the model interpret different sensor layouts at runtime. THINGS-MEG uses the CTF 271-mag-sensor system, which has its own xyz/orientation — these positions are read from the MNE .fif raw file in our neural_tokenizers/meg/brainomni/sensor_metadata.py and passed in at every forward.

3b. seanet_encoder (the time-domain compressor)

SEANet is a 1-D conv stack borrowed from the SoundStream / EnCodec neural-audio-codec line of work. With ratios [8, 4, 2] it downsamples the time axis by 8·4·2 = 64×. Applied per-channel (sensors processed independently). On a 512-sample window: 512 / 64 = 8 temporal latent positions. This is where the T_win = 8 in your token shape comes from.

Crucial property: this is a dataset-agnostic time-series compressor. The same convolutional kernels work on resting-state MEG, sleep MEG, task MEG, EEG — they just learn local temporal structure. BrainOmni's 656-hour pretraining poured most of its compute into these kernels.

3c. neuros + backwardsolution (the sensor → latent attention)

After SEANet, you have (B, C=271, T_win=8, D=n_dim). Now the encoder needs to fold 271 sensors into a small latent set.

It uses cross-attention: - Queries: 16 learnable embeddings (self.neuros, shape (16, D)) — these are the "imagined latent sources." They are the same across all trials and all subjects, and the model gets to learn what each one specializes in. - Keys/Values: the SEANet features at each timestep, plus the sensor embedding (so the attention layer knows which sensor each key comes from).

Output: (B, C'=16, T_win=8, D). The 271-sensor axis is gone; 16 latent channels remain.

Mental model: each latent neuro is like a learned source-reconstruction filter — it learns to read off something coherent across the 271 sensors. The sensor embedding tells the attention where each measurement was taken, so the same neuros[i] can extract "frontal alpha" from a CTF headset and from a different MEG/EEG layout it has never seen.

3d. quantizer.rvq (Residual Vector Quantization)

external/BrainOmni/model_utils/module.py:138-171

At this point the per-trial latent is (B, C'=16, T_win=8, D). RVQ now snaps each latent vector to a sequence of codebook indices.

Plain VQ: find nearest codebook entry c[i], token = i, residual r = z - c[i] is thrown away.

Residual VQ: chain Q=4 quantizers. Each quantizer encodes the residual left over by the previous one. So a single latent vector becomes a tuple of 4 integers (i₁, i₂, i₃, i₄) and is reconstructed by summing c₁[i₁] + c₂[i₂] + c₃[i₃] + c₄[i₄].

Why RVQ instead of a giant single codebook? With V=512 per layer and Q=4 layers you have effectively 512⁴ ≈ 6.9 × 10¹⁰ representable vectors, but you only train 4 × 512 = 2048 codebook entries — vastly cheaper than maintaining a ~10¹⁰-entry table. Same trick neural audio codecs (SoundStream, EnCodec) use to compress speech.

The codebooks are updated by EMA (exponential moving average) of the inputs that get assigned to each code — not by gradient descent. That's a non-obvious thing to know, and it affects our finetune decisions (see §5).

Final token tensor: (B, 16, 8, 4) int — that's the 512 integer indices per MEG trial that the docs refer to.

3e. Decoder (forwardsolution + seanet_decoder)

The decoder mirrors the encoder: - forwardsolution: cross-attention with sensor embeddings as queries, latent neuros as keys/values. Maps (B, 16 latent channels, 8 timesteps) → (B, 271 sensor channels, 8 timesteps). - seanet_decoder: transposed conv stack with ratios [8, 4, 2] reversed; upsamples 8 timesteps → 512 timepoints.

So the round trip is:

(B, 271, 512)  →  SEANet enc  →  (B, 271, 8, D)
                ↓ backward cross-attn (271 → 16 via sensor embed)
              (B, 16, 8, D)
                ↓ RVQ (4 codebooks × 512 codes)
              (B, 16, 8, 4) integer indices  ← these are the "tokens"
                ↓ codebook lookup + sum across 4 layers
              (B, 16, 8, D)
                ↓ forward cross-attn (16 → 271 via sensor embed)
              (B, 271, 8, D)
                ↓ SEANet dec
              (B, 271, 512)

4. What we actually do on THINGS-MEG, end to end

4a. Preprocessing per trial — neural_tokenizers/meg/brainomni/preprocess.py

THINGS-MEG trials are (271, 281) at 200 Hz. BrainOmni expects 256 Hz and a multiple-of-64 window length. So:

  1. Resample 200 → 256 Hz (preprocess.py:29-41). 281 samples at 200 Hz → 460 samples at 256 Hz.
  2. Per-trial per-channel z-score (preprocess.py:44-48). Mean and std computed before padding (so the pad zeros do not poison the stats). The stats are stored so we can invert at decode time.
  3. Zero-pad to 512 samples (preprocess.py:51-68). The pad goes on the post-stimulus tail (the part with the least task-relevant information).
  4. Build a validity mask (preprocess.py:71-82) marking real samples (1) vs zero-pad (0). This mask gates the reconstruction loss so we don't penalize the model for "failing to reconstruct" the padded zeros.

4b. Forward + loss — neural_tokenizers/meg/brainomni/trainer.py:97-142

x_unfold = model.unfold(x_pad)                                          # window into (B, C, N=1, L=512)
sensor_embedding = model.sensor_embed(pos, sensor_type)                 # (B, C, D)
feature = model.encoder(x_unfold, sensor_embedding)                     # (B, 16, N, T, D)
feature, indices, commitment_loss = model.quantizer(feature)            # tokens fall out here
x_rec = model.decoder(feature, sensor_embedding)                        # (B, C, N, L)
x_tgt = model.norm_target(x_unfold)                                     # what the recon should match

The composite loss (matches BrainOmni's pretraining recipe):

L = time_loss(masked MSE)
  + exp(-pcc)                  # encourages high Pearson correlation
  + commitment_loss            # standard VQ "encoder, commit to your codebook" term
  + amp_loss                   # FFT amplitude MSE
  + 0.5 * phase_loss           # FFT phase MSE

The two frequency-domain terms are critical for MEG — band power is what carries task signal (cf. parent §5.1). Time MSE alone would let the model rebuild the slow baseline and ignore the high-frequency evoked transients.

5. The freeze / finetune decision — what's trainable and why

This is in neural_tokenizers/meg/brainomni/trainer.py:42-59. Three modes are defined; we ran adapt.

What adapt does

for p in model.parameters(): p.requires_grad = True          # start: all trainable
for p in model.encoder.seanet_encoder.parameters():          # FREEZE the time-domain conv stack
    p.requires_grad = False
for p in model.decoder.seanet_decoder.parameters():          # FREEZE its transpose
    p.requires_grad = False

So we freeze: - encoder.seanet_encoder — the 1-D conv stack - decoder.seanet_decoder — its transpose

And we train: - sensor_embed — the per-sensor position/type embedding - encoder.neuros — the 16 learnable latent-source queries - encoder.backwardsolution — the sensors → neuros cross-attention - encoder.k_proj - decoder.forwardsolution — the neuros → sensors cross-attention - quantizer.rvq — the codebooks (gradient + EMA)

This works out to roughly 959k trainable / 5.05M total ≈ 19% of weights (meg/CLAUDE.md:303-308).

Why this particular split — the system-level reasoning

The decision falls out of asking what is genuinely new about THINGS-MEG, and what is shared with the pretraining distribution?

What is shared (so reuse the pretrained weights):

The SEANet conv stack learns local temporal structure of MEG/EEG voltages. The local temporal statistics of a 256-Hz MEG trace are roughly the same whether the subject is resting or seeing a picture — same Nyquist, same dominant oscillations (delta/theta/alpha/beta), same noise characteristics. BrainOmni saw 656 hours of MEG/EEG to learn these convolutions; we have ~42 hours of THINGS-MEG. Letting our small finetune dataset rewrite these kernels would (a) waste pretraining compute and (b) risk catastrophic forgetting — the model overfits THINGS and loses the generic temporal structure that was the whole point of using BrainOmni. So we freeze them.

What is NOT shared (so we have to train these):

  1. Sensor layout. THINGS-MEG is CTF 271-mag. BrainOmni pretrained on a different mix of headsets. The 6-D sensor positions/orientations are physically different. The sensor-embed must adapt to read out the new geometry. Train it.

  2. Latent neuros + cross-attention. The cross-attention modules are the interface between the (frozen) per-channel SEANet features and the (frozen) per-channel SEANet decoder. They are the only modules that "see" the relationship between the 271 sensors. The pretraining cross-attention learned to combine signals from a different head layout — it has to relearn how to combine THINGS-MEG's 271 sensors into 16 latents. Train them.

  3. RVQ codebooks. The pretrained codebooks were optimized for resting-state MEG features. THINGS-MEG is event-related — stimulus-locked evoked responses look statistically different from spontaneous oscillations. The codebooks must redistribute their entries over the new feature distribution. Train them (and EMA-update them).

In other words: we freeze what is physics-level / signal-level invariant and we finetune what is dataset-, layout-, and task-distribution-specific. That is the right cut for transfer learning with a small target dataset and a strong pretrained backbone.

Why not full or rvq_only

  • full: trains all 5.05M params. With only ~88k trials, this is fast catastrophic-forgetting territory — the pretrained SEANet kernels would drift to overfit THINGS evoked responses, and zero-shot transfer to other MEG datasets later would be gone. Higher overfit risk per trainer.py:25.
  • rvq_only: trains nothing via gradient — only EMA-updates the codebooks. This is a diagnostic mode: it tells you "if I let only the codebook adapt to THINGS, do reconstruction numbers improve?" It is a sanity check, not the production recipe.

Empirically adapt won: best val loss ~1.401 vs zero-shot ~4.35 MSE (meg/CLAUDE.md:521-530). Channel Pearson went from 0.345 (zero-shot) to 0.870 (after adapt).

Optimizer groups — a smaller but important detail

trainer.py:62-84 splits the trainable params into three AdamW groups:

Group Examples LR Weight decay
normal cross-attn linear layers lr (3e-5) 1e-2
no_decay RMSNorm weights, sensor-type embedding lr (3e-5) 0.0
codebook quantizer.* codebook_lr (3e-5) 0.0

Why split: norms and embeddings should not be regularized toward zero (would push the model toward trivial outputs); codebooks have their own EMA dynamics so weight decay on them fights the EMA update. This is standard for VQ-style models and is copied from BrainOmni's own pretraining recipe (model.py:84-102).

6. The "why this architecture for 4M" bit

Worth saying once: the reason BrainOmni won over μ-transform for the 4M production cache despite the linear probe being chance is token-count economics. μ-transform produces ~76,151 tokens per trial (one per (channel, sample)). BrainOmni produces 512. 4M's transformer cost is quadratic in sequence length, so 76k tokens per MEG trial is infeasible as a regular sequence; 512 is. The architecture's 16 latent × 8 time × 4 RVQ = 512 is by design tuned to the right order of magnitude for a multimodal transformer to consume.

End-of-turn

That's the full picture: BrainOmni is a SEANet-encoded, attention-bottlenecked, RVQ-quantized autoencoder; we freeze the SEANet kernels because they encode signal-level priors that transfer cleanly across MEG datasets, and we train the sensor embedding, the cross-attention bridge, and the codebooks because those are where the dataset-, layout-, and task-distribution differences live.

Weight-sharing structure in BrainTokenizer

Short answer: no, nothing meaningful is tied. The SEANet encoder and decoder are two distinct modules with separately allocated weights. The two cross-attention layers are two different classes with different parameter shapes — they couldn't share even if you wanted them to. The one thing that IS shared between the encoder and decoder paths is the sensor embedding module, but that's because it's instantiated once at the top level and called from both sides — it's not weight-tying, it's "same module used twice."

Let me lay out exactly which nn.Parameter tensors live where.

The full weight inventory

Every module is constructed once in BrainTokenizer.__init__ (model.py:34-66). PyTorch allocates fresh parameter tensors per construction. The three sub-modules below are independent.

BrainTokenizer
├── self.sensor_embed     ← single BrainSensorModule instance
├── self.encoder          ← BrainTokenizerEncoder instance
├── self.quantizer        ← BrainQuantizer instance  (just wraps an RVQ)
└── self.decoder          ← BrainTokenizerDecoder instance

self.sensor_embed — module.py:11-31

Parameter tensor Shape What it does
sensor_embedding_layer.weight (3, D) Embedding for sensor type (mag / grad / eeg)
pos_embedding_layer.0.weight/bias (D/2, 6), (D/2) MLP layer 1 on 6-D position+orientation
pos_embedding_layer.2.weight/bias (D, D/2), (D) MLP layer 2
aggregate_mlp.* (FeedForward) — Residual MLP
norm.weight (D,) RMSNorm gain

This module is called twice per forward — once from inside the encoder's forward, once from inside the decoder's forward — but those two calls hit the same parameters. So in that sense the encoder and decoder do share the sensor-embed weights. It is the only such sharing in the whole model.

self.encoder — BrainTokenizerEncoder — module.py:89-134

Parameter Shape Notes
seanet_encoder.* many conv weights 1-D conv stack with downsample ratios [8,4,2]. Distinct from the decoder's SEANet.
neuros (N_neuro=16, D) The learnable latent-source queries. Encoder-only — the decoder never references this tensor directly; it sees its post-quantization version flowing through the RVQ.
backwardsolution.v.weight/bias (D, D), (D,) Value projection in cross-attention
backwardsolution.proj.weight/bias (D, D), (D,) Output projection
k_proj.weight/bias (D, D), (D,) Key projection (applied to sensor_feat + sensor_embedding)

self.quantizer.rvq — module.py:138-171

Parameter Shape Notes
rvq.layers[q].codebook (V=512, codebook_dim) × Q=4 layers Four separate codebooks, one per RVQ residual layer. They do not share entries.
projection in / out of codebook_dim small Maps n_dim ↔ codebook_dim

self.decoder — BrainTokenizerDecoder — module.py:175-205

Parameter Shape Notes
forwardsolution.kv.weight/bias (2D, D), (2D,) Combined K+V projection — note this is one matrix that outputs both K and V via split(... , dim=-1).
forwardsolution.proj.weight/bias (D, D), (D,) Output projection
seanet_decoder.* many conv weights 1-D transposed conv stack with reversed ratios. Distinct from the encoder's SEANet.

So: is anything actually shared between encoder and decoder?

Pair Shared? Why
seanet_encoder ↔ seanet_decoder No Different classes (SEANetEncoder vs SEANetDecoder). Encoder downsamples (strided conv), decoder upsamples (transposed conv). The kernel shapes are mirrored but the weight tensors are independent. Even the kernel counts at each stage may differ.
backwardsolution ↔ forwardsolution No — and structurally cannot be They are different classes with different parameter layouts: BackWard has v: Linear(D, D) (K is computed elsewhere via k_proj), Forward has kv: Linear(D, 2D) (K and V from one combined matmul). Different shapes → can't tie.
encoder.k_proj ↔ anything in decoder No The decoder has no K projection. The role is asymmetric: in the encoder, K comes from sensor features; in the decoder, K and V both come from the latent-neuro stream via one kv matmul.
encoder.neuros ↔ decoder No The neuros tensor lives only inside the encoder. The decoder sees the quantized output that resulted from a cross-attention against neuros, not the parameter itself.
sensor_embed (encoder side) ↔ sensor_embed (decoder side) Yes Because self.sensor_embed is one module instance at the top of BrainTokenizer, and both forward paths call it. Same parameter tensors get used both times.
RVQ codebooks across the 4 layers No Each residual layer has its own (V, codebook_dim) matrix.

Why the asymmetry matters for our finetune

When we freeze SEANet encoder + SEANet decoder, we freeze two completely separate sets of conv weights. There's no risk of "I froze one and accidentally also froze the other via tying" — they're independent tensors and you'd have to call requires_grad = False on both, which is exactly what trainer.py:47-51 does:

for p in model.encoder.seanet_encoder.parameters(): p.requires_grad = False
for p in model.decoder.seanet_decoder.parameters(): p.requires_grad = False

Similarly, the cross-attention modules (backwardsolution and forwardsolution) are independently trainable — we are training two distinct sets of attention weights, one on each side of the bottleneck. That's the right setup, because the two attention layers have different jobs (271 sensors → 16 neuros vs 16 neuros → 271 sensors), and the data distribution of their queries/keys/values differs.

And because sensor_embed is the single shared module, when we train it, one gradient update improves both the encoder's and the decoder's sensor reasoning at once. That's why the sensor-embed parameter count counts once in the 959k trainable, not twice.

End-of-turn

Confirmed: SEANet encoder ↔ decoder do not share weights, and the two cross-attention modules can't share weights (different shapes). The only genuinely shared parameter set is sensor_embed, which is instantiated once at the top of BrainTokenizer and called from both the encoder and decoder forward passes.

Why frozen SEANet decoder still works — it never sees the 16 channels

Your intuition is reasonable but rests on a misreading of where the SEANet decoder sits in the data flow. Let me walk through the decoder forward with actual shapes; the answer falls out.

The decoder forward, with shapes

From external/BrainOmni/model_utils/module.py:191-205:

def forward(self, x, sensor_embedding):
    # x:                (B, 16, N=1, T=8, D)     ← 16 latent neuros, 8 time tokens
    # sensor_embedding: (B, 271, D)              ← per-sensor embedding

    x = self.forwardsolution(sensor_embedding, x)
    # ^^^ CROSS-ATTENTION HAPPENS HERE
    #     queries: sensor embeddings  (271 sensors)
    #     K/V:     the 16 latents
    #     output:  (B*N*T, 271, D)    ← already projected to 271-sensor space

    x = rearrange(x, "(B N T) C D -> (B C N) D T", ...)
    # ^^^ Reshape so each (sample, sensor) becomes its own (D, T=8) feature map

    x = self.seanet_decoder(x)
    # ^^^ Operates on (B*271, D, T=8) → (B*271, 1, L=512)
    #     Each of the 271 sensors' 8-timestep latent gets upsampled independently
    #     using THE SAME conv kernel

    return rearrange(x, "(B C N) 1 L -> B C N L", ...)
    # Output: (B, 271, 1, 512)

The critical fact: by the time data reaches seanet_decoder, the cross-attention has already converted the 16 latent channels into 271 sensor-space channels. The SEANet decoder never sees the 16-channel latent space at all.

What SEANet decoder actually does

It's a 1-D transposed convolution applied per-output-sensor, with the same kernel for every sensor. Its job is exactly:

Given an 8-timestep, D-dimensional feature map for one sensor channel, upsample it to a 512-sample waveform that "looks like MEG."

That's it. It's not learning anything about the 16 latents, and it's not learning anything sensor-specific. The sensor specialization is entirely the cross-attention's job (which uses sensor_embedding as queries to mix the right latents for each output sensor).

So the function the SEANet decoder represents is:

single-sensor latent (D, T=8)  →  single-sensor waveform (1, L=512)

This function is: - Sensor-agnostic — same kernel for every output sensor - Dataset-shift robust — the local temporal structure of an MEG voltage trace (its frequency content, smoothness, transient shapes) is similar across resting-state and event-related MEG. That's what the 656 h of pretraining captured.

Now the subtler version of your question

You might rephrase: "OK, the SEANet decoder doesn't see the 16 latents directly. But by finetuning the cross-attention + RVQ, I'm changing the feature distribution that gets fed into the frozen decoder. Doesn't that decoder need to adapt to the new feature distribution?"

That's a real concern. The answer is:

The frozen decoder acts as a fixed "language" that the trainable upstream layers have to speak. During finetune, the cross-attention and RVQ learn to emit features that the (frozen) SEANet decoder kernels can interpret as MEG waveforms. This is a feature, not a bug:

  1. It constrains the upstream layers to produce "MEG-shaped" features, because anything else would round-trip badly. The decoder is acting as a learned regularizer.
  2. It prevents catastrophic forgetting. If we unfroze the decoder, both ends could co-drift to memorize THINGS reconstruction while losing the generic-MEG prior — bad if you ever want to use this tokenizer on another MEG dataset, and bad for generalization.
  3. It is empirically sufficient. Channel Pearson 0.87 after adapt says the trainable layers successfully learned to feed the frozen decoder usable features. If the frozen decoder were the bottleneck, we would have plateaued well below 0.87.

Think of it like this: the SEANet decoder defines a manifold of "valid MEG-shaped reconstructions" in feature space. Finetuning the upstream layers is just teaching them to project THINGS data onto that manifold. We don't need to reshape the manifold; we just need to learn to land on it.

When you WOULD unfreeze the SEANet decoder

If you suspected the frozen kernels were systematically distorting THINGS waveforms — e.g., zero-shot 3a had reasonable cross-attention but high-frequency MEG transients were being smeared into smooth shapes — then the decoder's local temporal prior would be wrong and unfreezing helps. The diagnostic would be:

  • Time-domain MSE plateaus while spectral MSE in the gamma band (30–80 Hz) stays high
  • Per-channel residuals look like a fixed kernel-shape error rather than random misfit

Neither showed up here. PSD MSE dropped 67%, channel correlation went to 0.87 → the kernels were fine.

The asymmetric setup, finally

To make this concrete, here's the actual freeze structure relative to where the 16-channel bottleneck is:

Input (B, 271, 512)
   │
   ├── seanet_encoder ←──── FROZEN
   │     per-sensor 1-D conv stack, 512 → 8 timesteps
   │
   ├── backwardsolution ←─── TRAINABLE   (271 sensors → 16 latents)
   │     cross-attention; sensor embeddings as keys
   │
   ├── RVQ quantizer ←────── TRAINABLE   (the 16 latents get tokenized)
   │
   │    ╔═════════════ This is where the 16 latents live ═════════════╗
   │
   ├── forwardsolution ←──── TRAINABLE   (16 latents → 271 sensors)
   │     cross-attention; sensor embeddings as queries
   │
   ├── seanet_decoder ←───── FROZEN
   │     per-sensor 1-D conv stack, 8 → 512 timesteps
   │
Output (B, 271, 512)

Notice the symmetry: the trainable layers form a contiguous block around the 16-channel bottleneck, and the frozen SEANet stacks live entirely in 271-sensor space. The "fine-grain details of the 16 channels" you mentioned are exactly what backwardsolution, neuros, the RVQ codebooks, and forwardsolution are responsible for — and all four of those are trainable. The SEANet kernels never touch the 16-channel space.

End-of-turn

The SEANet decoder doesn't see the 16 latent channels — by the time data hits it, the cross-attention has already projected back to 271 sensor channels, and the decoder applies the same per-sensor upsampling kernel to all of them. So freezing it freezes a sensor-agnostic, dataset-shift-robust function; the dataset-specific work of decoding the 16-channel bottleneck is done by the trainable cross-attention right before it.

Perplexity and entropy in the codebook context

These two numbers are answering the same question — "how evenly is the tokenizer using its codebook?" — but in different units. Perplexity is the one that translates directly into intuition; entropy is the mathematically cleaner version that perplexity is built from.

The setup

You tokenize a big batch of MEG trials and count how often each of the 512 codebook entries gets selected. That gives you an empirical distribution p over the 512 codes:

p(code_0)   = fraction of all tokens that were code 0
p(code_1)   = fraction of all tokens that were code 1
...
p(code_511) = fraction of all tokens that were code 511

These 512 numbers sum to 1. Entropy and perplexity are summary statistics of this distribution.

Entropy: "how spread out is the distribution?"

Shannon entropy (in nats, since we use natural log):

H(p) = − Σᵢ p(i) · ln p(i)

Two extremes:

  • Uniform distribution (p(i) = 1/512 for every code): every code gets equal usage. H = ln(512) ≈ 6.238 nats. This is the maximum possible entropy.
  • Collapsed distribution (only one code ever gets used, p(0) = 1): H = 0 nats.

So H ranges from 0 (collapsed) to ln(V) ≈ 6.238 (perfectly uniform). The closer to ln(V), the better.

Our numbers (eval JSONs):

Entropy (nats) Fraction of max
3a (frozen) 6.094 97.7%
3b (finetuned) 6.202 99.4%
Max (uniform) 6.238 100%

Both runs are very close to maximum-entropy. Finetuning closed about 70% of the remaining gap.

Perplexity: "effective number of codes in use"

Perplexity is just entropy run through exp:

perplexity = exp(H(p))

That looks like a random transform, but it gives you a much more interpretable number: perplexity is the size of a uniform distribution that would have the same entropy as p.

Concretely:

  • If H = ln(512), perplexity = 512 → the distribution is "as spread out as a uniform over 512 codes." You're effectively using all 512.
  • If H = ln(100), perplexity = 100 → your distribution is "as spread out as a uniform over 100 codes." Even if all 512 are technically used, the rare ones contribute so little that the effective vocabulary is only 100.
  • If H = 0, perplexity = 1 → "as spread out as a uniform over 1 code" = degenerate, one code dominates.

Our numbers:

Perplexity Fraction of V=512 Wasted codebook capacity
3a (frozen) 443.3 86.6% ~69 codes worth
3b (finetuned) 493.5 96.4% ~18 codes worth
Max (uniform) 512 100% 0

Reading the 3a row: you have 512 codes available, but the way they're being used is only as informative as a uniform draw from 443 of them. About 69 codes are "wasted" in the sense that they're rarely selected and barely contribute. After finetuning, that wasted capacity shrinks to ~18 codes worth.

Why "codes_used = 512" but perplexity < 512

This is the subtle bit. Both 3a and 3b report codes_used = 512 and dead_code_fraction = 0. That means every single code got picked at least once on the eval set. So how can perplexity be less than 512?

Because "used at least once" is a much weaker statement than "used uniformly." Consider a toy example with V = 3:

  • Distribution A: p = (1/3, 1/3, 1/3). All 3 codes used. Entropy = ln(3) ≈ 1.099. Perplexity = 3.
  • Distribution B: p = (0.998, 0.001, 0.001). All 3 codes used (rare codes aren't dead). Entropy ≈ 0.018. Perplexity ≈ 1.02.

In B, the codebook reports "3/3 codes used, 0 dead," but the model is effectively a one-code tokenizer. Dead-code fraction is a binary test (was code i ever selected?), while perplexity is a continuous measure that penalizes uneven distributions.

This is exactly why we report both:

  • codes_used / dead_code_fraction catches gross collapse (codes that literally never fire).
  • perplexity catches soft collapse (codes that fire too rarely to carry information).

Our tokenizer passes the gross check on both runs but the soft check tells us finetune made the distribution noticeably flatter.

A worked sanity check on our numbers

3a entropy is 6.094 nats. Convert to perplexity:

exp(6.094) ≈ 442.7    ← matches the reported 443.3 (modulo rounding)

3b entropy is 6.202 nats:

exp(6.202) ≈ 493.5    ← matches exactly

Max possible:

ln(512) = 6.2383
exp(6.2383) = 512

So they really are the same number expressed two ways.

Why we care

A VQ tokenizer with low perplexity is a tokenizer that's leaving capacity on the table. The whole point of V = 512 is to give 4M a 512-way vocabulary; if the actual usage is concentrated on 100 codes, you might as well have built a V = 100 tokenizer and saved memory. Conversely, a tokenizer where perplexity is close to V is using its codebook the way the codebook was designed to be used: every entry is doing real work.

The trajectory 443 → 494 (out of 512) tells us finetune mostly equalized a codebook that was already broadly used but lopsided. It didn't unlock dead codes (there were none); it just redistributed mass off of overused codes onto underused ones.

End-of-turn

Entropy = how spread out the token distribution is, max = ln(V) ≈ 6.238 for V=512. Perplexity = exp(entropy) = effective vocabulary size, max = V = 512. Our 3a → 3b move was 443 → 494, meaning finetune turned a "mostly-but-unevenly-used" codebook into a "nearly-uniformly-used" one — using 96% of capacity instead of 87%, with no dead codes either way.

Sequence structure — what each stat is asking

The four codebook stats from last time told you "are all the codes being used?" The four sequence stats now ask a different question:

Given a sequence of tokens for one trial — say [42, 117, 9, 311, 88, ...] — does the sequence have anything left to predict, or is it so redundant that a transformer has nothing to learn?

This matters because 4M's training objective is "given some context tokens, predict the masked ones." If the tokenizer emits sequences where the next token is trivially determined by the previous one (e.g., always the same as the previous, or always alternating), the masked-prediction task is solved without learning anything generalizable. The model converges; nothing transfers. The sequence stats are designed to catch this silent failure.

Here are our numbers (3a eval JSON and 3b) again as the reference:

Stat 3a frozen 3b finetune Max possible
unigram_entropy 6.094 6.202 6.238
bigram_conditional_entropy 5.503 5.851 6.238
entropy_gap_pct 9.69% 5.65% 0%
mean_run_length 1.0017 1.0019 1.0 (ideal)
frac_runs_ge_2 0.582 0.628 —

1. unigram_entropy — how spread out is the marginal token distribution?

This is exactly the codebook entropy from last time, just computed on the full flattened token stream over all trials in the test set, not on a single trial.

H(t) = − Σᵢ p(i) · ln p(i)         where p(i) = fraction of all tokens equal to code i

Max possible: ln(512) ≈ 6.238 (every code used equally).

Why report it again in the sequence section: the codebook section's entropy is sometimes computed per RVQ layer or across a specific subset; the sequence section computes it on whatever exact token stream the harness is treating as "the sequence." Reporting both is a consistency check — they should be close, and they are (6.094 nats matches the codebook section).

Reading our numbers: healthy. Both runs are within a hair of the maximum.

2. bigram_conditional_entropy — given the previous token, how much uncertainty is left?

This is the key one. Bigram conditional entropy is:

H(tᵢ | tᵢ₋₁) = − Σⱼ p(tᵢ₋₁ = j) · Σₖ p(tᵢ = k | tᵢ₋₁ = j) · ln p(tᵢ = k | tᵢ₋₁ = j)

In plain English: for each possible "previous token" value j, look at the distribution of "next tokens" k that follow it. Compute the entropy of that conditional distribution. Then average those entropies, weighted by how often token j actually occurs.

Two intuition-building extreme cases:

  • Perfectly i.i.d. tokens (next token completely independent of previous): H(tᵢ | tᵢ₋₁) = H(tᵢ). Knowing the previous token tells you nothing about the next.
  • Perfectly deterministic transitions (every token uniquely determines the next): H(tᵢ | tᵢ₋₁) = 0. Knowing one token tells you everything.

For a transformer to have something to learn, you want H(tᵢ | tᵢ₋₁) close to H(tᵢ). That's the "transitions are nearly independent of the previous token, so the model has to use longer context to predict" regime.

If H(tᵢ | tᵢ₋₁) = 0, the transformer's next-token prediction is trivial — tᵢ is fully determined by tᵢ₋₁. It learns the identity function and nothing more.

Reading our numbers: - 3a: 5.503 nats — knowing the previous token reduced uncertainty from 6.094 → 5.503 nats. The previous token contained 6.094 − 5.503 = 0.591 nats of information about the next. - 3b: 5.851 nats — knowing the previous token reduced uncertainty from 6.202 → 5.851 nats. Only 0.351 nats of information leaked.

So finetuning made consecutive tokens more independent of each other — a transformer trained on 3b tokens has more left to learn from the previous-token cue than one trained on 3a tokens. Good.

3. entropy_gap_pct — the previous two, expressed as a fraction

entropy_gap_pct = (unigram_entropy − bigram_conditional_entropy) / unigram_entropy × 100

This normalizes the "how much does the previous token tell you about the next" quantity by the total marginal entropy, so you can compare tokenizers with different vocab sizes apples-to-apples.

Bounds: - 0% = consecutive tokens are perfectly independent. Ideal. - 100% = consecutive tokens are perfectly deterministic. Maximum redundancy.

The parent CLAUDE.md §5 sets <20% as a "healthy" pass threshold.

Reading our numbers: - 3a: 9.69% — already healthy. - 3b: 5.65% — even closer to i.i.d. Roughly halved.

In dollar terms: 3b sequences look more like fresh dice rolls (good for transformer learning), 3a sequences are slightly more "the next token kind of looks like the previous one" (still passing, but less ideal).

4. mean_run_length — the most direct redundancy test

A "run" is a stretch of the sequence where the same token repeats. For example, in [7, 7, 7, 12, 3, 3, 88] you have runs of lengths 3, 1, 2, 1. Mean run length is (3+1+2+1)/4 = 1.75.

Two extreme cases: - All tokens unique ([7, 12, 3, 88, 41]): mean run length = 1.0. Best. - All tokens identical ([7, 7, 7, 7, 7]): mean run length = sequence length. Worst — the tokenizer has effectively emitted one code per trial, padded out.

This is the most visually obvious sign of collapse: if your tokenizer keeps spitting out the same code 8 times in a row, masked-token prediction is trivially solvable and 4M learns nothing.

Reading our numbers: both runs are ~1.002, basically the floor. Consecutive identical tokens are essentially nonexistent. Finetune didn't move this — there was nothing to fix.

5. frac_runs_ge_2 — how many trials have any repeat?

The fraction of trials that contain at least one run of length ≥ 2 (i.e., at least one place where tᵢ = tᵢ₋₁).

This is a coarser, complementary measure to mean run length. Mean run length tells you the average; frac_runs_ge_2 tells you the prevalence. You could have a low mean run length (good) but still have some pathological trials where one token repeats a lot — that case would show up here.

Reading our numbers: - 3a: 58.2% of trials have at least one repeat somewhere - 3b: 62.8% of trials have at least one repeat somewhere

This one got slightly worse after finetune, which initially looks bad. But notice that mean run length stayed at ~1.002 in both cases, so the repeats are isolated (tᵢ, tᵢ₊₁) = (k, k) events, not long runs. Combined with the entropy-gap improvement (9.69% → 5.65%), the picture is: finetune produced sequences where occasional adjacent repeats happen by chance (consistent with a more uniform / more i.i.d. emission), but those repeats are statistically incidental rather than evidence of collapse. If anything, an i.i.d. process should have some adjacent repeats: a uniform draw from 512 codes over a length-512 sequence will have expected number of adjacent equal pairs ≈ 511/512 ≈ 1, so the per-trial probability of at least one repeat is high.

This is a good example of why you need to read sequence stats together: any one number is ambiguous in isolation.

6. max_possible_entropy — the ceiling, included for reference

max_possible_entropy = ln(V) = ln(512) ≈ 6.238

This is constant per tokenizer. It's reported so the reader can quickly check unigram_entropy / max_possible_entropy to see how close to maximum-spread the marginal is — and so the harness's "is this number good?" thresholds can be relative to vocab size rather than absolute. A tokenizer with V = 128 and unigram_entropy = 4.8 looks worse than one with V = 512 and unigram_entropy = 6.0 in absolute terms, but actually has higher fractional codebook usage (4.8 / ln(128) = 99% vs 6.0 / ln(512) = 96%).

Putting the four diagnostic stats together

The four real measurements (ignoring max_possible_entropy, which is a constant) form a layered diagnostic:

Measures What it catches
unigram_entropy Is the marginal distribution spread out? (single-token redundancy)
bigram_conditional_entropy + entropy_gap_pct Are consecutive tokens redundant with each other? (transformer has nothing to learn)
mean_run_length Severity of the most visible collapse pattern (long repeats)
frac_runs_ge_2 Prevalence of even short repeats

A healthy tokenizer hits roughly: unigram entropy near max, entropy gap < 20%, mean run length ≈ 1.0, frac_runs_ge_2 consistent with an i.i.d. process at that vocab size.

Our 3b run satisfies all four: 6.20 / 6.24 marginal, 5.6% gap, 1.002 mean run, 62.8% repeat-prevalence (in the ballpark of what pure chance predicts at V=512, L=512). The sequence is transformer-ready in the structural sense — whether it carries semantic content is a separate question (the linear probe), and that one still fails.

End-of-turn

Marginal entropy = "are the codes evenly used overall." Bigram conditional entropy = "given the previous token, how much uncertainty is left about the next." Entropy gap % = those two as a normalized redundancy fraction. Mean run length = "how long do same-code stretches get on average." Frac runs ≥ 2 = "how many trials have any adjacent repeat at all." Together they catch every flavor of "transformer-unlearnable sequence collapse." Our 3b run passes all four.

μ-transform — what it is and why it's the baseline

The μ-transform isn't a deep learning model at all. It's a 50-year-old idea from telephone audio compression, repurposed by Cho et al. 2026 as a "what's the dumbest sensible tokenizer for neural data?" baseline. Walking through it tells you a lot about what we mean by "tokenizer" in the first place.

1. The big picture: a stateless per-sample codec

μ-transform takes one MEG sample (one float — say the voltage at one sensor at one timepoint) and maps it to one integer in [0, V). That's literally it. Apply it independently to every (channel, timepoint) cell of a trial and you get a (271, 281) grid of integers.

Compare to BrainOmni: BrainOmni reads 271 sensors × 512 timepoints and emits 512 integers total for the whole trial. μ-transform reads the same input and emits 271 × 281 = 76,151 integers — one per input sample. So μ-transform is barely "compression" — it's requantization. It replaces 32-bit floats with 8-bit integers, like converting WAV to telephone-quality audio.

This is the production-pipeline reason BrainOmni wins for 4M, even though μ-transform "wins" reconstruction: BrainOmni's 512 tokens fit in a transformer context; μ-transform's 76k tokens don't.

2. The four-step pipeline

mu_transform/encoder.py:20-45 and mu_transform/quantizer.py:18-30:

Step 1 — Clip

x_clipped = clamp(x, q_lo[c], q_hi[c])

For each channel c, clip the voltage to a per-channel range — defaults [0.5th, 99.5th] percentiles of that channel's distribution on the training data. This kills extreme outliers (movement artifacts, electrode glitches) that would otherwise eat up the codec's dynamic range.

Why per-channel: different MEG sensors record different absolute amplitudes depending on where they sit on the head. A global clip would over-clip quiet sensors and under-clip loud ones.

Step 2 — Max-abs scale to [−1, 1]

x_scaled = x_clipped / s[c]      where s[c] = max(|x|) on channel c

Per-channel rescale so each channel's signal lives in [−1, 1]. Now every channel speaks the same numerical units regardless of physical amplitude. The bin grid we'll apply next is universal across channels.

Step 3 — μ-law compand

F(x) = sign(x) · log(1 + μ·|x|) / log(1 + μ)         with μ = 255

This is the telephone trick. A plain uniform quantizer of 256 bins on [−1, 1] would waste resolution on the rarely-occupied tails and starve the densely-occupied middle. μ-law warps the input so small values get expanded (more dynamic range allocated to the dense center) and large values get compressed (less resolution where data is rare).

Visually, the input-output curve looks like an S, steep near 0 and flat near ±1. After companding, the distribution is roughly uniform on [−1, 1] — exactly what a uniform binner wants to see.

The constant μ = 255 is a historical choice from G.711 telephony, kept by Cho 2026 for continuity.

Step 4 — Uniform binning

token = floor((F(x) + 1) / 2 · V)        with V = 256

Slice [−1, 1] into 256 equal-width bins, emit the bin index. That's the integer token.

The inverse is bin-center lookup: token i → −1 + (i + 0.5) · 2/V → inverse μ-law → multiply by s[c] → reconstructed voltage. No learning, no decoder weights.

3. "Calibration": what "fit" means without gradients

μ-transform has zero gradient-descent weights, but it still has two dataset-dependent parameters per channel that must be set before inference:

  • s[c] — the per-channel max-abs scaler
  • (q_lo[c], q_hi[c]) — per-channel clip thresholds

These are estimated on the training split, written to a small JSON sidecar (calibration.json), and reused frozen at inference. The CLAUDE.md calls this "calibration" specifically to distinguish it from "training." Think of it as fitting a statistic, not a model.

The hyperparameters μ = 255, V = 256, percentile choice [0.5, 99.5] are not fit — they're paper defaults.

The shipped artifact is the JSON; total state is 271 × 3 + 2 ≈ 815 numbers. Compare to BrainOmni's 5M parameters.

4. Why it's the baseline — four concrete reasons

(a) Floor on the leaderboard

If you ship a deep tokenizer that fails to beat a stateless 50-year-old telecom codec on some axis, you've learned something important — either the deep model is broken or that axis is not what differentiates them. Without a floor, you have no idea whether your fancy tokenizer's MSE of 1.2 is "good" or "barely above hand-crafted." With the floor present, you can see immediately that BrainOmni 3a's MSE of 4.35 is worse than the trivial baseline — that's how we knew zero-shot wasn't enough.

(b) Harness sanity check

The §5 evaluation harness has to compute reconstruction MSE, channel Pearson, PSD MSE, codebook perplexity, bigram entropies, linear probes on a multi-axis dataset. Any one of those metrics could be implemented wrong. The μ-transform tokenizer is so trivially analyzable that you know what every metric should return — near-perfect MSE, full codebook utilization, no degenerate runs. If the harness produces a surprising number on μ-transform, the harness is the bug. If it agrees with hand-derived expectations on μ-transform, you trust it on BrainOmni.

This is the same principle as testing a sort function on a 3-element input before trusting it on a million-element input.

(c) Defers the deep architecture commitment

The parent CLAUDE.md is explicit about this: "avoid prematurely committing to a deep architecture before we know what 'good enough' looks like on this specific THINGS-MEG distribution." Numbers from the baseline tell you what the easy part of the problem looks like — once you see μ-transform achieves 0.998 channel Pearson, you know reconstruction per se is not hard on this data; the hard part must be elsewhere (semantic content, downstream task accuracy, sequence compressibility). That reframes what Phase 2/3 actually has to deliver.

(d) Reveals the trivial-vs-meaningful distinction

This is the most interesting one. μ-transform shows that you can ace reconstruction (MSE = 1.0, Pearson = 0.998) and still fail the linear probe (top-1 ≈ chance). That's structural: if the hand-crafted codec fails the probe, the probe failure cannot be the deep model's fault — it's an inherent property of "reconstruction-quality tokens." This is what motivated the BrainOmni §5 verdict ("probe failure is expected under reconstruction-only training").

Without the baseline, when BrainOmni 3b also failed the probe you might have concluded "BrainOmni is broken, try harder." With the baseline, you can see that even a near-perfect codec doesn't carry concept-decodability, so the issue is the training objective, not the tokenizer architecture.

5. What μ-transform's strengths and weaknesses tell you

The §5 leaderboard numbers from CLAUDE.md §9, read with this framing:

Axis μ-transform What it teaches
MSE = 0.999, channel Pearson = 0.998 Near-invertible per-sample codec Reconstruction is "easy" — almost any sensible codec gets there. Reconstruction is a necessary but not sufficient property.
PSD MSE = 1.01 × 10⁹ (much worse than BrainOmni 3b's 6.13 × 10⁷) Time-domain perfection doesn't imply spectral perfection Per-sample quantization noise is white-ish, which dumps energy across the whole spectrum and corrupts band power. MEG cares about bands.
256/256 codes used, perplexity 194 (out of 256 = 76%) A simple codec already uses its codebook well Codebook utilization is a low bar. Failing it means something is really broken; passing doesn't prove anything interesting.
Entropy gap 22.9%, frac_runs_ge_2 = 1.0 Strong adjacent-sample redundancy A per-sample codec on continuous data is guaranteed to produce highly autocorrelated token streams — neighboring MEG samples are very similar, so they map to neighboring or identical bins. This is the "transformer would have nothing to learn" failure mode, made obvious in the baseline.
Linear probe top-1 ≈ chance Reconstruction-quality tokens carry no decodable concept signal The failure is in the training objective, not the architecture. (Confirmed once BrainOmni 3b also fails it.)
76,151 tokens per trial No real compression Disqualifies it from 4M production. This single fact is why μ-transform is a baseline and not a candidate.

End-of-turn

μ-transform is a stateless per-sample codec (clip → scale → μ-law → bin) borrowed from telephone audio. It's the baseline because (a) it's the simplest thing that fits the Tokenizer interface, (b) it lets us sanity-check the evaluation harness against numbers we can derive by hand, (c) it puts a floor on what counts as "real compression," and (d) its passes-reconstruction-but-fails-probe pattern is what showed us that the linear-probe failure is intrinsic to reconstruction-only training, not a BrainOmni bug. It will never ship into 4M — 76k tokens per trial doesn't fit in a transformer context — but it makes every later number interpretable.
