19	## 2. What 4M expects from a new modality — the binding constraint
20	
21	Before designing anything, the codebook size we ship has to fit 4M's modality
22	registration model. Read [`../../external/ml-4m/fourm/data/modality_info.py`](../../external/ml-4m/fourm/data/modality_info.py)
23	— that file is the source of truth.
24	
25	Concrete facts that drive the design:
26	
27	- **No fixed vocab across modalities.** Each modality declares its own
28	  `vocab_size` in `MODALITY_INFO`. Image-token modalities range from 4096
29	  (semseg) to 16384 (RGB); text/sequence is 30_000; `human_poses` "borrows"
30	  the 30k text vocab via `shared_vocab: ['caption']`. We are free to pick.
31	- **But vocab IS fixed at registration time.** 4M allocates a learnable
32	  embedding `[vocab_size, d_model]` for the modality. Adding tokens
33	  mid-training breaks that table. **Pick V before computing the token cache
34	  that 4M trains on; commit to it.**
35	- **`type: 'seq'` is the right modality type for MEG**, not image-grid.
36	  Mirror `human_poses` (`modality_info.py:172`) more than `tok_rgb`. The
37	  per-channel-per-timestep token stream is a sequence, not a 2D grid.
38	  Pose-tokenizer registration template:
39	  `vocab_size`, `encoder_embedding = SequenceEncoderEmbedding(...)`,
40	  `decoder_embedding = SequenceDecoderEmbedding(...)`,
41	  `max_length`, `num_channels`, `type: 'seq'`,
42	  `id: generate_uint15_hash('meg_tokens')`.
43	- **Multi-codebook (`num_codebooks > 1`) is supported by the `memcodes`
44	  quantizer** (used by `human_poses`, 1024 × 8). Cho 2026's tokenizers use
45	  a single codebook per channel. Stick with **`num_codebooks = 1`** for MEG
46	  — there is no reason to fight the upstream tokenizer's design.
47	- **Whether to share vocab with another modality is a 4M-side decision.**
48	  Human_poses reuses the text vocab because pose tokens are mapped into the
49	  WordPiece range; this lets a 4M decoder generate poses with the same head
50	  as captions. We do **not** want that for MEG — it couples MEG to text
51	  training. **Register a dedicated `meg_tokens` modality with its own
52	  vocab.**
53	
54	So the headline constraint: pick `V` and `max_length` (= per-trial token
55	sequence length) when you write the tokenizer, because the modality
56	registration that goes into 4M has to declare both.

162	## 5. What "recalibration" / "finetuning" actually mean here
163	
164	The Cho 2026 paper warns that both tokenizer families need dataset-specific
165	adaptation. The mechanism is different for each.
166	
167	### Phase 1 — μ-transform recalibration (no gradient descent, but still a `fit` step)
168	
169	The μ-transform pipeline (paper §3.3.1):
170	
171	1. **Clip** the input to a percentile range `[q_lo, q_hi]` to suppress
172	   outliers.
173	2. **Max-abs scale** to `[−1, 1]`: `x' = x / s` with `s = max|x|`.
174	3. **μ-law compand**: `F(x') = sgn(x') · ln(1 + μ|x'|) / ln(1 + μ)`.
175	4. **Uniform bin** the compressed values into `V` bins → integer token IDs.
176	
177	The reference config Cho 2026 ships
178	([`../../external/Cho2026_Tokenizer/models/tokenizer/mu_transform/config.yml`](../../external/Cho2026_Tokenizer/models/tokenizer/mu_transform/config.yml))
179	is:
180	
181	```yaml
182	mu: 255
183	n_tokens: 256
184	normalization: max_abs
185	```
186	
187	There are **no learned weights**, but two parameters are dataset-dependent
188	and must be fit on a THINGS-MEG training split, frozen, and reused at
189	inference:
190	
191	- `s = max|x|`: the per-channel (recommended) or global max-abs scaler.
192	- `q_lo, q_hi`: clip thresholds (defaults around the 0.5 / 99.5 percentiles).
193	
194	`μ` and `V` are hyperparameters, not fit from data — start with μ=255,
195	V=256.
196	
197	Concretely: there is no `.train()` loop. There is a `fit(train_dataset) →
198	calibration.json` step that produces a small JSON sidecar with
199	`{s_per_channel, q_lo, q_hi, mu, V, channel_mode}`. At inference we load
200	that JSON and apply the stateless transform. Treat the calibration JSON as
201	a versioned tokenizer checkpoint.
202	
203	**Phase 1 status (shipped):** calibrated on THINGS-MEG train split with
204	μ=255, V=256, per-channel clip `[0.5, 99.5]` percentiles. Run slug:
205	`V256_mu255_clip0.5-99.5_per_channel_s0`. Eval via
206	`modal_meg_eval.py --tokenizer mu_transform` (see §9).
207	
208	> Verified §3 fact that matters here: all 271 channels are `mag`, so the
209	> grad-vs-mag amplitude disparity problem does not apply. Per-channel `s`
210	> is still the safer default (inter-sensor amplitude variance is real even
211	> within one channel type), but the bin-saturation risk is much lower than
212	> it would be on a mixed-sensor dataset.

214	### Phase 2 — Cho 2026 learnable tokenizer (actual gradient training)
215	
216	The Cho 2026 learnable tokenizer (paper §3.2) is an **autoencoder**, not a
217	VQ-VAE: encoder is GRU → Dense → LayerNorm producing per-time-step logits
218	over a vocab of size `V`; decoder is a bank of `V` 1-D conv kernels (causal
219	or non-causal) summed with learned per-token gains. Trained end-to-end with
220	MSE + an annealed soft-to-hard argmax relaxation, then unused tokens are
221	pruned ("token refactorization").
222	
223	Reference configs they ship
224	(`../../external/Cho2026_Tokenizer/models/tokenizer/{causal,noncausal}/config.yml`):
225	
226	```yaml
227	sequence_length: 200
228	n_channels: 52         # NB: 52 source-parcellated channels, NOT sensor
229	n_tokens: 128
230	token_dim: 10
231	rnn_n_units: 128
232	token_kernel_padding: causal  # or `same` for non-causal
233	training: { lr: 1e-5, batch: 32, epochs: 40, anneal_stages: 40 }
234	```
235	
236	Three things the configs tell us, in light of the verified data spec:
237	
238	1. **They trained on source-reconstructed data (52 parcels), not sensor
239	   space.** Our THINGS-MEG is **271-sensor**. The tokenizer is applied
240	   per-channel and the GRU has no channel-conditioning, so weights *can*
241	   be reused on sensor data, but the *amplitude distribution* of source
242	   parcels (after `osl_dynamics.standardize()`) is not identical to sensor
243	   amplitudes. This is the symmetric MEG version of the LABRAM/THINGS
244	   distribution mismatch in the parent doc.
245	2. **Their sequence_length = 200 samples ≈ 0.8 s.** Our trials are 281
246	   samples (1.4 s). Either reduce our windows to 200 samples (matches their
247	   training distribution → reuse pretrained weights cleanly) or retrain
248	   with `sequence_length = 281`. Strong prior: **window to 200** for 2a/2b
249	   to keep the pretrained encoder in-distribution; only change
250	   `sequence_length` for 2c.
251	3. **Their LR (1e-5) is very small.** Tells us finetuning is intended to be
252	   conservative — they expect the loss landscape to be flat around the
253	   pretrained optimum. Start with that LR and only increase if the loss is
254	   not moving after 1 epoch.
255	
256	"Finetuning on our dataset" then has three concrete recipes:
257	
258	- **2a.** Load pretrained weights, apply on THINGS-MEG **as-is** (zero
259	  finetune). Establishes the floor for what their pretrained tokenizer
260	  buys us on distribution-shifted data.
261	- **2b.** Load pretrained weights, continue MSE training on THINGS-MEG
262	  (lr=1e-5, anneal κ already at 0 → hard tokens — or restart anneal).
263	  This is the "actual finetune."
264	- **2c.** Take their architecture, train from scratch on THINGS-MEG. This
265	  is needed if (a) pretrained weights fail the §5 harness even after 2b,
266	  (b) we want to change `sequence_length` or `n_tokens`, or (c) the
267	  source-vs-sensor mismatch turns out to dominate. With ~108k trials
268	  pooled across P1–P4 (see §3 — much more than the parent doc estimated),
269	  from-scratch is more viable than first assumed.
270	
271	Run **2a → 2b → 2c** in order; stop as soon as the harness numbers are
272	acceptable. Do not skip 2a — the gap between 2a and 2b is the answer to
273	"how much did pretraining buy us."
274	
275	> System-level note: their pretraining data is **resting-state** MEG;
276	> THINGS-MEG is **event-related** (stimulus-locked). Expect the pretrained
277	> kernels to capture continuous oscillatory structure well and to
278	> underweight evoked transients. Reconstruction MSE may look fine while
279	> the §5.3 linear probe quietly tanks — watch that metric closely after
280	> each of 2a/2b/2c.

467	## 8. Evaluation contract
468	
469	Identical to the parent §5. The same `test_tokenizer.py` harness must pass
470	for each of the three MEG tokenizers before it is wired into 4M:
471	
472	1. Reconstruction fidelity (time MSE + per-channel Pearson + **spectral
473	   MSE**). Spectral MSE is the diagnostic metric for MEG — band power
474	   (especially α/β) is what carries task signal. PSD range: 1–100 Hz
475	   (Nyquist for 200 Hz data is 100 Hz).
476	2. Codebook utilization (perplexity, dead-code fraction). The μ-transform's
477	   dead-code fraction is the metric to watch; Cho 2026 prunes dead codes
478	   ("token refactorization") so its effective `V*` can be reported
479	   alongside nominal `V`.
480	3. Linear probe on THINGS labels. With 22,449 image-level codes / 1854
481	   concepts / ~27 superordinate categories, use the THINGS
482	   **superordinate** categories as the primary probe target, 1854-way
483	   concept ID as a secondary, image ID never (too noisy at 27k trials per
484	   subject). The `image_id → concept_id → superordinate` mapping comes
485	   from THINGS metadata, not the .fif files — add it under
486	   `meg/data.py`.
487	4. Token-sequence statistics (unigram entropy, bigram conditional entropy,
488	   run-length). The μ-transform is at particular risk of degenerate run
489	   lengths on slow drifts — watch this.
490	
491	MEG-specific harness parameters (PSD frequency range, named bands) go in
492	`meg_config.py` and are passed into the harness as `EvalConfig`. Do not
493	hardcode them.
494

513	## 9. §5 leaderboard — μ-transform vs BrainOmni (2026-05-22)
514	
515	All runs: **n=3000 test trials**, **seed=0**, learnable split 80/10/10
516	(BrainOmni) / μ-transform uses same test indices via `splits.py`. Eval JSONs
517	under each run's `evals/` folder.
518	
519	### Reconstruction (§5.1)
520	
521	| Metric | μ-transform | BrainOmni 3a (zero-shot) | BrainOmni 3b (finetuned) |
522	|--------|-------------|--------------------------|--------------------------|
523	| MSE ↓ | **0.999** | 4.35 | 1.20 |
524	| Channel Pearson ↑ | **0.998** | 0.345 | 0.870 |
525	| PSD MSE ↓ | 1.01×10⁹ | 1.88×10⁸ | **6.13×10⁷** |
526	
527	μ-transform's near-perfect channel correlation is **misleading** — it is a
528	near-invertible per-sample codec (clip → μ-law → 256 bins). BrainOmni is a
529	real compressor (271×281 → 512 codes). **3b finetune** closes most of the
530	reconstruction gap vs 3a; μ-transform still wins trivial round-trip fidelity.
531	
532	### Codebook (§5.2)
533	
534	| Metric | μ-transform (V=256) | BrainOmni 3a/3b (V=512) |
535	|--------|----------------------|-------------------------|
536	| Codes used | 256/256 | 512/512 |
537	| Utilization | 100% | 100% |
538	| Perplexity | 194 | 443 → **494** (3b) |
539	| Entropy (nats) | 5.27 | 6.09 → **6.20** (3b) |
540	
541	Both use full vocab. BrainOmni spreads mass over a larger codebook.
542	
543	### Sequence structure (§5.4)
544	
545	| Metric | μ-transform | BrainOmni 3a | BrainOmni 3b |
546	|--------|-------------|--------------|--------------|
547	| Entropy gap % ↑ | **22.9%** | 9.7% | 5.6% |
548	| Mean run length | 1.06 | ~1.00 | ~1.00 |
549	| frac runs ≥ 2 | **1.0** | 0.58 | 0.63 |
550	
551	Different token geometries (76k sample tokens vs 512 latent codes) — entropy
552	gap is not directly comparable. μ-transform shows strong local temporal
553	redundancy; BrainOmni tokens look more i.i.d.
554	
555	### Linear probe (§5.3) — **gate metric**
556	
557	| Metric | μ-transform | BrainOmni 3a | BrainOmni 3b | Random |
558	|--------|-------------|--------------|--------------|--------|
559	| top-1 tokens | 0.17% | 0.17% | **0.00%** | 0.17% |
560	| top-5 tokens | 2.0% | **2.67%** | 2.17% | 2.17% |
561	| top-1 raw | 0.33% | 0.33% | 0.33% | — |
562	
563	**Neither tokenizer passes the probe gate** (~1152 concepts in this 3000-trial
564	subset). Finetune improved reconstruction, not linear object decodability.
565	Raw waveform probe is identical across runs (~0.33% top-1) — barely above
566	chance. Full test split (~10k trials): μ-transform top-1 tokens 0.25% vs
567	0.40% random — still essentially chance.
568	
569	### Verdict for 4M
570	
571	| Criterion | μ-transform | BrainOmni 3b |
572	|-----------|-------------|--------------|
573	| Tokens / trial | ~76,151 | **512** |
574	| 4M context cost | Very high | **Practical** |
575	| Reconstruction | Excellent (near-identity) | Good (post-finetune) |
576	| Linear object info in tokens | None detected | None detected |
577	| Trained on THINGS | Calibration only | **Yes (adapt finetune)** |
578	
579	**Recommendation:** ship **BrainOmni 3b** into 4M for token caching.
580	Probe failure is expected under reconstruction-only training; semantic readout
581	may require 4M's nonlinear/multimodal objective (cf. BrainOmni Stage 2 in
582	the paper — not implemented here).

MEG Tokenization with BrainOmni — Deep Dive

Let me start at the highest level and zoom progressively in, then explain the freeze/finetune decision once you understand what the pieces are.

1. What "tokenization" even means in this setting

4M (the multimodal transformer downstream) only knows how to read integer ID sequences. RGB images come in as integer codes, text as WordPiece IDs. To let 4M see MEG, we need a function:

raw MEG trial  (271 channels × 281 timepoints of floats)  →  some short sequence of integers

And, importantly, we need an inverse — integers → reconstructed MEG — because the only way we can train the tokenizer without labeled "ground truth tokens" is by demanding that the round-trip x → tokens → x_hat stays close to x. This is exactly the autoencoder + bottleneck idea, plus the bottleneck happens to be discrete (a finite codebook).

So a "neural tokenizer" is just a VQ-style autoencoder: - Encoder squashes the raw signal into a small latent representation - Quantizer snaps each latent vector to its nearest entry in a learned codebook of V vectors → the index becomes the token - Decoder takes those codebook vectors and reconstructs the original signal

The quality criterion is "tokens that 4M can learn from." That is what §5 of the parent doc tries to measure.

2. BrainOmni's BrainTokenizer at a high level

BrainOmni (OpenTSLab/BrainOmni) was pretrained on 656 hours of resting-state MEG/EEG across many sensor geometries. Its tokenizer is called BrainTokenizer. It has four sub-modules; see external/BrainOmni/braintokenizer/model.py:15-66 for the constructor.

BrainTokenizer
├── sensor_embed     : BrainSensorModule       — "where is each sensor and what kind is it?"
├── encoder          : BrainTokenizerEncoder   — sensor space → latent neuro space
│   ├── seanet_encoder  : SEANetEncoder            (1-D conv stack, per channel)
│   ├── neuros          : nn.Parameter(N_neuro, D) (learnable latent "source" queries)
│   ├── backwardsolution: cross-attention          (sensors → neuros)
│   └── k_proj          : Linear
├── quantizer        : BrainQuantizer          — RVQ over latent neuro features
│   └── rvq          : 4 residual VQ layers, V=512 codes each
└── decoder          : BrainTokenizerDecoder   — latent neuro space → sensor space
    ├── forwardsolution: cross-attention            (neuros → sensors)
    └── seanet_decoder : SEANetDecoder              (1-D transposed conv stack, per channel)

The conceptual move that makes BrainOmni interesting: instead of trying to model 271 raw MEG channels directly, it introduces an intermediate "latent neural source" space. The encoder asks 16 imagined source-like queries to attend over the 271 sensors; the decoder asks the 271 sensors to attend over the 16 latent sources. This is structurally similar to how MEG source-reconstruction works (a small number of "neural sources" projected to many sensors via a mixing matrix) — but the mixing is learned via attention.

That n_neuro = 16 is the secret reason BrainOmni produces such a small token sequence: 271 sensors get compressed into 16 latent channels before tokenization.

3. The pieces, one by one

3a. BrainSensorModule (the "sensor embed")

external/BrainOmni/model_utils/module.py:11-32

Input: for each sensor c, a 6-D vector pos = [x, y, z, ox, oy, oz] (sensor 3-D position + 3-D orientation) and an integer sensor_type (1 for mag, etc.).

Output: a per-sensor embedding vector of dim n_dim.

Why it matters: BrainOmni was pretrained on many MEG/EEG headsets. The sensor-embed lets the model interpret different sensor layouts at runtime. THINGS-MEG uses the CTF 271-mag-sensor system, which has its own xyz/orientation — these positions are read from the MNE .fif raw file in our neural_tokenizers/meg/brainomni/sensor_metadata.py and passed in at every forward.

3b. seanet_encoder (the time-domain compressor)

SEANet is a 1-D conv stack borrowed from the SoundStream / EnCodec neural-audio-codec line of work. With ratios [8, 4, 2] it downsamples the time axis by 8·4·2 = 64×. Applied per-channel (sensors processed independently). On a 512-sample window: 512 / 64 = 8 temporal latent positions. This is where the T_win = 8 in your token shape comes from.

Crucial property: this is a dataset-agnostic time-series compressor. The same convolutional kernels work on resting-state MEG, sleep MEG, task MEG, EEG — they just learn local temporal structure. BrainOmni's 656-hour pretraining poured most of its compute into these kernels.

3c. neuros + backwardsolution (the sensor → latent attention)

After SEANet, you have (B, C=271, T_win=8, D=n_dim). Now the encoder needs to fold 271 sensors into a small latent set.

It uses cross-attention: - Queries: 16 learnable embeddings (self.neuros, shape (16, D)) — these are the "imagined latent sources." They are the same across all trials and all subjects, and the model gets to learn what each one specializes in. - Keys/Values: the SEANet features at each timestep, plus the sensor embedding (so the attention layer knows which sensor each key comes from).

Output: (B, C'=16, T_win=8, D). The 271-sensor axis is gone; 16 latent channels remain.

Mental model: each latent neuro is like a learned source-reconstruction filter — it learns to read off something coherent across the 271 sensors. The sensor embedding tells the attention where each measurement was taken, so the same neuros[i] can extract "frontal alpha" from a CTF headset and from a different MEG/EEG layout it has never seen.

3d. quantizer.rvq (Residual Vector Quantization)

external/BrainOmni/model_utils/module.py:138-171

At this point the per-trial latent is (B, C'=16, T_win=8, D). RVQ now snaps each latent vector to a sequence of codebook indices.

Plain VQ: find nearest codebook entry c[i], token = i, residual r = z - c[i] is thrown away.

Residual VQ: chain Q=4 quantizers. Each quantizer encodes the residual left over by the previous one. So a single latent vector becomes a tuple of 4 integers (i₁, i₂, i₃, i₄) and is reconstructed by summing c₁[i₁] + c₂[i₂] + c₃[i₃] + c₄[i₄].

Why RVQ instead of a giant single codebook? With V=512 per layer and Q=4 layers you have effectively 512⁴ ≈ 6.9 × 10¹⁰ representable vectors, but you only train 4 × 512 = 2048 codebook entries — vastly cheaper than maintaining a ~10¹⁰-entry table. Same trick neural audio codecs (SoundStream, EnCodec) use to compress speech.

The codebooks are updated by EMA (exponential moving average) of the inputs that get assigned to each code — not by gradient descent. That's a non-obvious thing to know, and it affects our finetune decisions (see §5).

Final token tensor: (B, 16, 8, 4) int — that's the 512 integer indices per MEG trial that the docs refer to.

3e. Decoder (forwardsolution + seanet_decoder)

The decoder mirrors the encoder: - forwardsolution: cross-attention with sensor embeddings as queries, latent neuros as keys/values. Maps (B, 16 latent channels, 8 timesteps) → (B, 271 sensor channels, 8 timesteps). - seanet_decoder: transposed conv stack with ratios [8, 4, 2] reversed; upsamples 8 timesteps → 512 timepoints.

So the round trip is:

(B, 271, 512)  →  SEANet enc  →  (B, 271, 8, D)
                ↓ backward cross-attn (271 → 16 via sensor embed)
              (B, 16, 8, D)
                ↓ RVQ (4 codebooks × 512 codes)
              (B, 16, 8, 4) integer indices  ← these are the "tokens"
                ↓ codebook lookup + sum across 4 layers
              (B, 16, 8, D)
                ↓ forward cross-attn (16 → 271 via sensor embed)
              (B, 271, 8, D)
                ↓ SEANet dec
              (B, 271, 512)

4. What we actually do on THINGS-MEG, end to end

4a. Preprocessing per trial — neural_tokenizers/meg/brainomni/preprocess.py

THINGS-MEG trials are (271, 281) at 200 Hz. BrainOmni expects 256 Hz and a multiple-of-64 window length. So:

  1. Resample 200 → 256 Hz (preprocess.py:29-41). 281 samples at 200 Hz → 460 samples at 256 Hz.
  2. Per-trial per-channel z-score (preprocess.py:44-48). Mean and std computed before padding (so the pad zeros do not poison the stats). The stats are stored so we can invert at decode time.
  3. Zero-pad to 512 samples (preprocess.py:51-68). The pad goes on the post-stimulus tail (the part with the least task-relevant information).
  4. Build a validity mask (preprocess.py:71-82) marking real samples (1) vs zero-pad (0). This mask gates the reconstruction loss so we don't penalize the model for "failing to reconstruct" the padded zeros.

The composite loss (matches BrainOmni's pretraining recipe):

L = time_loss(masked MSE)
  + exp(-pcc)                  # encourages high Pearson correlation
  + commitment_loss            # standard VQ "encoder, commit to your codebook" term
  + amp_loss                   # FFT amplitude MSE
  + 0.5 * phase_loss           # FFT phase MSE

The two frequency-domain terms are critical for MEG — band power is what carries task signal (cf. parent §5.1). Time MSE alone would let the model rebuild the slow baseline and ignore the high-frequency evoked transients.

5. The freeze / finetune decision — what's trainable and why

This is in neural_tokenizers/meg/brainomni/trainer.py:42-59. Three modes are defined; we ran adapt.

What adapt does

for p in model.parameters(): p.requires_grad = True          # start: all trainable
for p in model.encoder.seanet_encoder.parameters():          # FREEZE the time-domain conv stack
    p.requires_grad = False
for p in model.decoder.seanet_decoder.parameters():          # FREEZE its transpose
    p.requires_grad = False

So we freeze: - encoder.seanet_encoder — the 1-D conv stack - decoder.seanet_decoder — its transpose

And we train: - sensor_embed — the per-sensor position/type embedding - encoder.neuros — the 16 learnable latent-source queries - encoder.backwardsolution — the sensors → neuros cross-attention - encoder.k_proj - decoder.forwardsolution — the neuros → sensors cross-attention - quantizer.rvq — the codebooks (gradient + EMA)

This works out to roughly 959k trainable / 5.05M total ≈ 19% of weights (meg/CLAUDE.md:303-308).

Why this particular split — the system-level reasoning

The decision falls out of asking what is genuinely new about THINGS-MEG, and what is shared with the pretraining distribution?

What is shared (so reuse the pretrained weights):

The SEANet conv stack learns local temporal structure of MEG/EEG voltages. The local temporal statistics of a 256-Hz MEG trace are roughly the same whether the subject is resting or seeing a picture — same Nyquist, same dominant oscillations (delta/theta/alpha/beta), same noise characteristics. BrainOmni saw 656 hours of MEG/EEG to learn these convolutions; we have ~42 hours of THINGS-MEG. Letting our small finetune dataset rewrite these kernels would (a) waste pretraining compute and (b) risk catastrophic forgetting — the model overfits THINGS and loses the generic temporal structure that was the whole point of using BrainOmni. So we freeze them.

What is NOT shared (so we have to train these):

  1. Sensor layout. THINGS-MEG is CTF 271-mag. BrainOmni pretrained on a different mix of headsets. The 6-D sensor positions/orientations are physically different. The sensor-embed must adapt to read out the new geometry. Train it.

  2. Latent neuros + cross-attention. The cross-attention modules are the interface between the (frozen) per-channel SEANet features and the (frozen) per-channel SEANet decoder. They are the only modules that "see" the relationship between the 271 sensors. The pretraining cross-attention learned to combine signals from a different head layout — it has to relearn how to combine THINGS-MEG's 271 sensors into 16 latents. Train them.

  3. RVQ codebooks. The pretrained codebooks were optimized for resting-state MEG features. THINGS-MEG is event-related — stimulus-locked evoked responses look statistically different from spontaneous oscillations. The codebooks must redistribute their entries over the new feature distribution. Train them (and EMA-update them).

In other words: we freeze what is physics-level / signal-level invariant and we finetune what is dataset-, layout-, and task-distribution-specific. That is the right cut for transfer learning with a small target dataset and a strong pretrained backbone.

Why not full or rvq_only

  • full: trains all 5.05M params. With only ~88k trials, this is fast catastrophic-forgetting territory — the pretrained SEANet kernels would drift to overfit THINGS evoked responses, and zero-shot transfer to other MEG datasets later would be gone. Higher overfit risk per trainer.py:25.
  • rvq_only: trains nothing via gradient — only EMA-updates the codebooks. This is a diagnostic mode: it tells you "if I let only the codebook adapt to THINGS, do reconstruction numbers improve?" It is a sanity check, not the production recipe.

Empirically adapt won: best val loss ~1.401 vs zero-shot ~4.35 MSE (meg/CLAUDE.md:521-530). Channel Pearson went from 0.345 (zero-shot) to 0.870 (after adapt).

Optimizer groups — a smaller but important detail

trainer.py:62-84 splits the trainable params into three AdamW groups:

Group Examples LR Weight decay
normal cross-attn linear layers lr (3e-5) 1e-2
no_decay RMSNorm weights, sensor-type embedding lr (3e-5) 0.0
codebook quantizer.* codebook_lr (3e-5) 0.0

Why split: norms and embeddings should not be regularized toward zero (would push the model toward trivial outputs); codebooks have their own EMA dynamics so weight decay on them fights the EMA update. This is standard for VQ-style models and is copied from BrainOmni's own pretraining recipe (model.py:84-102).

6. The "why this architecture for 4M" bit

Worth saying once: the reason BrainOmni won over μ-transform for the 4M production cache despite the linear probe being chance is token-count economics. μ-transform produces ~76,151 tokens per trial (one per (channel, sample)). BrainOmni produces 512. 4M's transformer cost is quadratic in sequence length, so 76k tokens per MEG trial is infeasible as a regular sequence; 512 is. The architecture's 16 latent × 8 time × 4 RVQ = 512 is by design tuned to the right order of magnitude for a multimodal transformer to consume.

End-of-turn

That's the full picture: BrainOmni is a SEANet-encoded, attention-bottlenecked, RVQ-quantized autoencoder; we freeze the SEANet kernels because they encode signal-level priors that transfer cleanly across MEG datasets, and we train the sensor embedding, the cross-attention bridge, and the codebooks because those are where the dataset-, layout-, and task-distribution differences live.

seanet_encoder ↔ seanet_decoder No Different classes (SEANetEncoder vs SEANetDecoder). Encoder downsamples (strided conv), decoder upsamples (transposed conv). The kernel shapes are mirrored but the weight tensors are independent. Even the kernel counts at each stage may differ.
backwardsolution ↔ forwardsolution No — and structurally cannot be They are different classes with different parameter layouts: BackWard has v: Linear(D, D) (K is computed elsewhere via k_proj), Forward has kv: Linear(D, 2D) (K and V from one combined matmul). Different shapes → can't tie.
encoder.k_proj ↔ anything in decoder No The decoder has no K projection. The role is asymmetric: in the encoder, K comes from sensor features; in the decoder, K and V both come from the latent-neuro stream via one kv matmul.
encoder.neuros ↔ decoder No The neuros tensor lives only inside the encoder. The decoder sees the quantized output that resulted from a cross-attention against neuros, not the parameter itself.
sensor_embed (encoder side) ↔ sensor_embed (decoder side) Yes Because self.sensor_embed is one module instance at the top of BrainTokenizer, and both forward paths call it. Same parameter tensors get used both times.
RVQ codebooks across the 4 layers No Each residual layer has its own (V, codebook_dim) matrix.

Why the asymmetry matters for our finetune

When we freeze SEANet encoder + SEANet decoder, we freeze two completely separate sets of conv weights. There's no risk of "I froze one and accidentally also froze the other via tying" — they're independent tensors and you'd have to call requires_grad = False on both, which is exactly what trainer.py:47-51 does:

for p in model.encoder.seanet_encoder.parameters(): p.requires_grad = False
for p in model.decoder.seanet_decoder.parameters(): p.requires_grad = False

Similarly, the cross-attention modules (backwardsolution and forwardsolution) are independently trainable — we are training two distinct sets of attention weights, one on each side of the bottleneck. That's the right setup, because the two attention layers have different jobs (271 sensors → 16 neuros vs 16 neuros → 271 sensors), and the data distribution of their queries/keys/values differs.

And because sensor_embed is the single shared module, when we train it, one gradient update improves both the encoder's and the decoder's sensor reasoning at once. That's why the sensor-embed parameter count counts once in the 959k trainable, not twice.

End-of-turn

Confirmed: SEANet encoder ↔ decoder do not share weights, and the two cross-attention modules can't share weights (different shapes). The only genuinely shared parameter set is sensor_embed, which is instantiated once at the top of BrainTokenizer and called from both the encoder and decoder forward passes.

Why frozen SEANet decoder still works — it never sees the 16 channels

Your intuition is reasonable but rests on a misreading of where the SEANet decoder sits in the data flow. Let me walk through the decoder forward with actual shapes; the answer falls out.

The decoder forward, with shapes

From external/BrainOmni/model_utils/module.py:191-205:

def forward(self, x, sensor_embedding):
    # x:                (B, 16, N=1, T=8, D)     ← 16 latent neuros, 8 time tokens
    # sensor_embedding: (B, 271, D)              ← per-sensor embedding

    x = self.forwardsolution(sensor_embedding, x)
    # ^^^ CROSS-ATTENTION HAPPENS HERE
    #     queries: sensor embeddings  (271 sensors)
    #     K/V:     the 16 latents
    #     output:  (B*N*T, 271, D)    ← already projected to 271-sensor space

    x = rearrange(x, "(B N T) C D -> (B C N) D T", ...)
    # ^^^ Reshape so each (sample, sensor) becomes its own (D, T=8) feature map

    x = self.seanet_decoder(x)
    # ^^^ Operates on (B*271, D, T=8) → (B*271, 1, L=512)
    #     Each of the 271 sensors' 8-timestep latent gets upsampled independently
    #     using THE SAME conv kernel

    return rearrange(x, "(B C N) 1 L -> B C N L", ...)
    # Output: (B, 271, 1, 512)

The critical fact: by the time data reaches seanet_decoder, the cross-attention has already converted the 16 latent channels into 271 sensor-space channels. The SEANet decoder never sees the 16-channel latent space at all.

What SEANet decoder actually does

It's a 1-D transposed convolution applied per-output-sensor, with the same kernel for every sensor. Its job is exactly:

Given an 8-timestep, D-dimensional feature map for one sensor channel, upsample it to a 512-sample waveform that "looks like MEG."

That's it. It's not learning anything about the 16 latents, and it's not learning anything sensor-specific. The sensor specialization is entirely the cross-attention's job (which uses sensor_embedding as queries to mix the right latents for each output sensor).

So the function the SEANet decoder represents is:

single-sensor latent (D, T=8)  →  single-sensor waveform (1, L=512)

This function is: - Sensor-agnostic — same kernel for every output sensor - Dataset-shift robust — the local temporal structure of an MEG voltage trace (its frequency content, smoothness, transient shapes) is similar across resting-state and event-related MEG. That's what the 656 h of pretraining captured.

Now the subtler version of your question

You might rephrase: "OK, the SEANet decoder doesn't see the 16 latents directly. But by finetuning the cross-attention + RVQ, I'm changing the feature distribution that gets fed into the frozen decoder. Doesn't that decoder need to adapt to the new feature distribution?"

That's a real concern. The answer is:

The frozen decoder acts as a fixed "language" that the trainable upstream layers have to speak. During finetune, the cross-attention and RVQ learn to emit features that the (frozen) SEANet decoder kernels can interpret as MEG waveforms. This is a feature, not a bug:

  1. It constrains the upstream layers to produce "MEG-shaped" features, because anything else would round-trip badly. The decoder is acting as a learned regularizer.
  2. It prevents catastrophic forgetting. If we unfroze the decoder, both ends could co-drift to memorize THINGS reconstruction while losing the generic-MEG prior — bad if you ever want to use this tokenizer on another MEG dataset, and bad for generalization.
  3. It is empirically sufficient. Channel Pearson 0.87 after adapt says the trainable layers successfully learned to feed the frozen decoder usable features. If the frozen decoder were the bottleneck, we would have plateaued well below 0.87.

Think of it like this: the SEANet decoder defines a manifold of "valid MEG-shaped reconstructions" in feature space. Finetuning the upstream layers is just teaching them to project THINGS data onto that manifold. We don't need to reshape the manifold; we just need to learn to land on it.

When you WOULD unfreeze the SEANet decoder

If you suspected the frozen kernels were systematically distorting THINGS waveforms — e.g., zero-shot 3a had reasonable cross-attention but high-frequency MEG transients were being smeared into smooth shapes — then the decoder's local temporal prior would be wrong and unfreezing helps. The diagnostic would be:

  • Time-domain MSE plateaus while spectral MSE in the gamma band (30–80 Hz) stays high
  • Per-channel residuals look like a fixed kernel-shape error rather than random misfit

Neither showed up here. PSD MSE dropped 67%, channel correlation went to 0.87 → the kernels were fine.

The asymmetric setup, finally

To make this concrete, here's the actual freeze structure relative to where the 16-channel bottleneck is:

Input (B, 271, 512)
   │
   ├── seanet_encoder ←──── FROZEN
   │     per-sensor 1-D conv stack, 512 → 8 timesteps
   │
   ├── backwardsolution ←─── TRAINABLE   (271 sensors → 16 latents)
   │     cross-attention; sensor embeddings as keys
   │
   ├── RVQ quantizer ←────── TRAINABLE   (the 16 latents get tokenized)
   │
   │    ╔═════════════ This is where the 16 latents live ═════════════╗
   │
   ├── forwardsolution ←──── TRAINABLE   (16 latents → 271 sensors)
   │     cross-attention; sensor embeddings as queries
   │
   ├── seanet_decoder ←───── FROZEN
   │     per-sensor 1-D conv stack, 8 → 512 timesteps
   │
Output (B, 271, 512)

Notice the symmetry: the trainable layers form a contiguous block around the 16-channel bottleneck, and the frozen SEANet stacks live entirely in 271-sensor space. The "fine-grain details of the 16 channels" you mentioned are exactly what backwardsolution, neuros, the RVQ codebooks, and forwardsolution are responsible for — and all four of those are trainable. The SEANet kernels never touch the 16-channel space.

End-of-turn

The SEANet decoder doesn't see the 16 latent channels — by the time data hits it, the cross-attention has already projected back to 271 sensor channels, and the decoder applies the same per-sensor upsampling kernel to all of them. So freezing it freezes a sensor-agnostic, dataset-shift-robust function; the dataset-specific work of decoding the 16-channel bottleneck is done by the trainable cross-attention right before it.
