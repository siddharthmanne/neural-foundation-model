2	
3	Stock 4M's in-loop eval can only validate the training task (its val loaders
4	derive masking from the train dataset of the same name), and it only runs at all
5	when the data config has a ``val:`` section. To also score the named tasks in
6	``configs/4m_things_val_tasks.yaml`` every ``eval_freq`` epochs — independently
7	of any ``val:`` section — we wrap a single stock-trainer global:
8	
9	  * ``train_one_epoch`` → after each epoch, at the eval cadence, run the suite on
10	                          the just-updated model and merge its per-task losses
11	                          into the returned train stats (which the trainer writes
12	                          to ``log.txt`` / wandb).
13

15	called with a ``val:`` section) keeps in-loop validation self-contained: it needs
16	only the ``in_loop_val_tasks`` field. The wrapper factory below is pure (no
17	fourm) and unit-tested; the fourm-dependent ``build_suite_fn`` reuses
18	``validate_4m.ValidationSuite`` wholesale — it does not re-implement validation.

35	def make_validating_train_one_epoch(
36	    orig_train_one_epoch: Callable,
37	    suite_fn: Callable[[Any, Any], dict],
38	    eval_freq: int,
39	    epochs: int | None,
40	) -> Callable:
41	    """Wrap ``train_one_epoch`` to run the suite at the eval cadence.
42	
43	    After the wrapped epoch, when ``(epoch+1) % eval_freq == 0`` (or it's the
44	    final epoch), ``suite_fn(model, device)`` runs on the just-updated live model
45	    and its stats are merged into the returned train stats. Suite errors are
46	    swallowed with a warning so a val hiccup never kills a long training run.
47	    Stock calls this as ``train_one_epoch(model=..., device=..., epoch=..., ...)``.
48	    """
49	
50	    def wrapped(*args: Any, **kwargs: Any):
51	        stats = orig_train_one_epoch(*args, **kwargs)
52	        epoch = kwargs.get("epoch")
53	        if epoch is None:
54	            return stats
55	        at_cadence = (epoch + 1) % eval_freq == 0 or (epochs and epoch + 1 == epochs)
56	        if at_cadence:
57	            try:
58	                suite_stats = suite_fn(kwargs.get("model"), kwargs.get("device"))
59	                stats = {**stats, **suite_stats}
60	                summary = "  ".join(
61	                    f"{k}={v:.4f}" for k, v in suite_stats.items() if k.endswith("] loss")
62	                )
63	                if summary:
64	                    print(f"[in-loop val] epoch {epoch}: {summary}", flush=True)
65	                # Tokens-seen for the checkpoint being scored. The token-accounting wrapper
66	                # (installed inside this one) already merged these into `stats`; report them
67	                # alongside the val losses so each validation moment is labeled with the
68	                # exact training tokens behind it.
69	                tokens = {k: stats[v] for k, v in TOKEN_STAT_KEYS.items() if v in stats}
70	                if tokens:
71	                    print(f"[in-loop val] epoch {epoch}: {format_tokens_seen(tokens)}", flush=True)
72	            except Exception as exc:  # noqa: BLE001 — never crash training on val
73	                print(f"[in-loop val] suite failed at epoch {epoch}: {exc!r}")
74	        return stats
75

139	def evaluate_task(model, loader, device, all_domains, n_in, n_out, loss_type, dtype, n_batches):
140	    """Average loss + per-modality loss over up to n_batches val batches."""
141	    model.eval()
142	    totals: dict[str, float] = {}
143	    count = 0
144	    for i, x in enumerate(loader):
145	        if n_batches and i >= n_batches:
146	            break
147	        mod_dict = {
148	            mod: {k: v.to(device) for k, v in d.items()}
149	            for mod, d in x.items()
150	            if mod in all_domains
151	        }
152	        autocast = torch.autocast(device_type=device.type, dtype=dtype, enabled=dtype != torch.float32)
153	        with autocast:
154	            loss, mod_loss = model(mod_dict, num_encoder_tokens=n_in,
155	                                   num_decoder_tokens=n_out, loss_type=loss_type)
156	        totals["loss"] = totals.get("loss", 0.0) + float(loss)
157	        for mod, l in mod_loss.items():
158	            totals[f"{mod}_loss"] = totals.get(f"{mod}_loss", 0.0) + float(l.mean())
159	        count += 1
160	    if count == 0:
161	        return {"loss": float("nan"), "n_batches": 0}
162	    stats = {k: v / count for k, v in totals.items()}
163	    stats["n_batches"] = count
164	    return stats

173	    callers share it, differing only in *where the model comes from* and *how the
174	    result is formatted*:

I now have the complete picture. Here's how validation works in your pipeline, and an important caveat about "fixed eval."

Example selection — deterministic ✓

Selection is fixed, not random. The eval loader is built deterministically:

  • fourm_dataloader.py:172-185 builds the pipe from wds.SimpleShardList(data_path) with no shuffle and no repeat (unlike the train loader). Shards are read in listed order, samples in tar order.
  • validate_4m.py:144-146: evaluate_task just does for i, x in enumerate(loader): if i >= n_batches: break — it takes the first n_batches batches every time.
  • The suite caches the loader (validate_4m.py:200-207) and uses num_workers=0, so there's no worker-ordering nondeterminism.

So every run (and every in-loop epoch) scores the same examples in the same order. Good.

Masking — random and unseeded ✗

This is the problem for your comparative-base goal. Two things are fixed, one is not:

  • Token counts are fixed. validate_4m.py:204 passes (n_in, n_in) and (n_out, n_out), so random.randint(n_in, n_in) always returns n_in (neural_masking.py:84). This is all 4M's fixed_eval flag ever meant — fixed budgets.
  • Which tokens get masked is random, per sample, every run. Inside the same __call__:
  • dir_idx = random.randint(...) / torch.multinomial(...) picks which Dirichlet mixture component → which modality split (neural_masking.py:79-82)
  • input_dirichlets[dir_idx].sample() splits the fixed budget across modalities randomly (masking.py:192)
  • noise = torch.rand(num_tokens); argsort picks which specific cells are input vs target (masking.py:248-249)

There is no seeding anywhere in the eval path (I grepped — none). Stock 4M even prints the tell at fourm_dataloader.py:289-291: "Eval stats may vary slightly as the masking applied on images is random."
