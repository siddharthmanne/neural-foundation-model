Good question — let me walk you through what is actually happening in probe.py, where the label space comes from, and what's making the numbers look like noise right now.

What the probe is actually doing today

It's a 3-way comparison on the same classification task with the same train/test split. All three classifiers predict the same labels:

Feature set Source Role
tokens Tokenizer output, pooled to (B, D) via codebook-embedding mean (or bag-of-codes if no embed) The thing being tested
raw Flattened (C·T,) MEG signal Upper bound — all info is there
random Uniform random tokens of the same shape Lower bound — no info

Each gets fed into a single nn.Linear head, trained with AdamW + cross-entropy (probe.py:153-163), then scored with top-1 and top-5 (probe.py:171-182). Top-k is already there and configurable via EvalConfig.probe_top_k (protocol.py:122).

The label space — this is where your "27 vs 1000+" confusion is real

The pipeline goes MEG trigger → image_id → concept_id:

  • 22,449 image-level trigger codes (one per unique image shown)
  • 1,854 THINGS object concepts — what the probe currently uses (modal_meg_eval.py:219-229)
  • 27 THINGS superordinate categories — what meg/CLAUDE.md §8 says should be the primary probe target, but is not implemented

There is then a "step (b)" densification (modal_meg_eval.py:227-229) that only keeps the concepts actually present in the eval sample — for n=3000 trials this is ~1,152 classes. So the eval JSON's top1_tokens = 0.0035 is 1/1152 ≈ chance. That's what's making the numbers look broken.

So the answer to your direct question: - Not 27. Not 22,449. Currently ~1,152-way (a thinned 1,854-way). - Tokens vs raw vs random are all judged on that same 1,152-way task. - Top-1 and top-5 are both reported.

Why the current setup is barely informative

  1. ~1,152 classes / ~3,000 trials ≈ 2.6 trials per class. A linear head cannot fit per-class weights from ~2 samples. Even the raw signal probe sits at 0.33% top-1 (CLAUDE.md §9) — that means the ceiling is broken, so we can't distinguish "tokenizer threw the signal away" from "task is too hard."
  2. No trial averaging. THINGS-MEG repeats some images 12×; averaging those repeats kills single-trial noise that the probe currently has no way to see through.
  3. Single random split. No cross-validation; the noise floor itself is noisy.
  4. Embed-and-mean over all 512 RVQ codes (probe.py:106-119) collapses all 4 RVQ layers and 8 temporal positions into one vector. Per-layer / per-position probes would tell you where the information lives if it's there at all.

Concrete ways to make it diagnostic

In order of impact:

  1. Switch primary label space to the 27 superordinate categories. This is the canonical "is the tokenizer keeping anything semantic?" question. 3,000 trials / 27 classes ≈ 110 per class — finally enough to fit. Keep 1,854-way as a secondary metric. The mapping needs to be added to meg/data.py (currently only has concept_id, not superordinate).
  2. Run on the full test split (--n-test 0) — already noted as pending in meg/CLAUDE.md §9. ~10k trials.
  3. Average repeats before probing. For the 200 test-images-×-12-reps subset, this gives 12× SNR for free.
  4. Add k-fold CV (e.g. 5-fold). Cheap; gives error bars.
  5. Report a "raw trial-averaged" ceiling alongside raw. If even that ceiling is at chance, the conclusion is "MEG doesn't decode objects with a linear head," not "the tokenizer is bad."
  6. Per-RVQ-layer probes for BrainOmni — does layer 0 carry semantic info that gets dropped later? Useful diagnostic, not strictly required.

A note on the system-level framing

The current probe answers the question "are tokens linearly decodable to a 1,854-way label?" — which is the right shape of question (linear, because that's what 4M's input embeddings see), but the label space is wrong for the sample size. The fix isn't to make the probe smarter — that would overstate quality, exactly the failure mode the probe.py docstring warns against. The fix is to make the label space match the statistical power of the trial count. Coarser labels, more trials per label, averaging — same probe.

def _balanced_accuracy(logits, labels, n_classes):

Three input shape families:

Source Natural shape (after batch) Dispatch target
BrainOmni tokens (RVQ-decoded) (16 latent, 8 time, D=512 embed) CNN2D — D as channels, conv over (latent × time)
μ-transform tokens (dequantized) (271 channels, 281 time) CNN1D — channels as input dim, conv over time
Raw signal (271 channels, 281 time) CNN1D — same as μ-transform
Random tokens matches real tokens shape same dispatch as tokens

Dispatch rule: 3D feats after batch ((B, C, T)) → CNN1D. 4D feats ((B, S1, S2, D)) → CNN2D.

CNN1D head (~33k–145k params depending on input)

Conv1d(C_in, 64, k=7, stride=2, padding=3) → BN → ReLU
Conv1d(64,    64, k=5, stride=2, padding=2) → BN → ReLU
AdaptiveAvgPool1d(1) → Flatten
Linear(64, n_classes)

Large initial kernel (7) + stride-2 captures local temporal pattern then downsamples. Two convs total; we're probing, not learning the whole task.

CNN2D head (~333k params for D=512)

Conv2d(D_embed, 64, k=(3,3), padding=1) → BN → ReLU
Conv2d(64,      64, k=(3,3), padding=1) → BN → ReLU
AdaptiveAvgPool2d((1,1)) → Flatten
Linear(64, n_classes)

3×3 spatial kernels over the (16 latent × 8 time) grid. Smaller field-of-view than CNN1D because the grid itself is small.

Why this is a fair "right inductive bias" probe

  • Temporal smoothness assumption (convs over time): consistent with what EEGNet/Conformer use on MEG/EEG
  • Latent locality on the 2D grid: weak but defensible (some latents in BrainOmni learn similar functions)
  • Small parameter count: ~150k vs MLP's 20M for raw — much less overfitting risk
  • No attention / recurrence: keeps the §5.3 spirit ("simplest probe that has the right inductive bias")

If we train BrainOmni on averaged MEG, the model's encoder learns the statistics of averaged signal (lower noise floor, different spectral profile, smoother spatial structure). But this creates a real design tension

Cross-subject + cross-rep averaging gives the cleanest noise reduction (one sample per unique image), but it also eliminates subject as a decodable variable — by definition, an averaged trial has no subject identity. Three implications:

Early-stopping design

# In the existing epoch loop, after computing val_avg:
if val_avg < best_val:
    best_val = val_avg
    epochs_since_best = 0
    torch.save(model.state_dict(), out_dir / "BrainTokenizer.pt")

else:
    epochs_since_best += 1

if epoch + 1 >= min_epochs and epochs_since_best >= patience:

Defaults: patience=2, min_epochs=3. Loose enough to absorb initial training volatility (one bad epoch ≠ stop), tight enough to catch overfitting fast on a small dataset. Min 3 epochs ensures we don't stop on cold-start noise.

Best checkpoint is already saved every time val improves — so the on-disk weights at the end will be the best-val ones regardless of whether we early-stopped or ran to full epochs.

What the val loss curve will tell us

If val_loss plateaus while train_loss keeps dropping → classic overfitting, patience triggers, we stop. Best val checkpoint is preserved.

If val_loss tracks train_loss throughout → not overfitting; runs the full schedule.
