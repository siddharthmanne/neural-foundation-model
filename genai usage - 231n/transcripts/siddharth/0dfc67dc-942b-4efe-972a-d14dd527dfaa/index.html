Short answer: yes, you need a split — but for reasons that have less to do with the μ-transform itself and more to do with what the §5 harness measures.

Why a split is still required, even though μ-transform has no learned weights

The μ-transform looks "stateless," but per meg/CLAUDE.md §5 it has a fit(train_dataset) → calibration.json step that estimates two real dataset-fit parameters:

  • s = max|x| (per-channel max-abs scaler)
  • q_lo, q_hi (clip percentiles, ~0.5 / 99.5)

Fitting these on the same trials you then evaluate on is leakage — small in magnitude (max|x| over 27k trials ≈ max|x| over 22k trials), but principled rot. More importantly, it makes the Phase 1 → Phase 2 comparison unfair, because Phase 2 will have a proper split and you'd be comparing apples (calibrated on test) to oranges (trained on train).

The real reason: §5 has axes that mandate a split regardless of the tokenizer

Look at the four axes in neural_tokenizers/CLAUDE.md §5:

Axis Needs a split? Why
5.1 Reconstruction (MSE / Pearson / spectral MSE) yes (mild) calibration leakage as above
5.2 Codebook utilization (perplexity, dead codes) yes (mild) same — bin edges fit on test data
5.3 Linear probe on THINGS labels YES, hard requirement trains a logistic regression on tokens; without a split the probe number is meaningless. §5.3 literally says "Report top-1/top-5 on a held-out split."
5.4 Token-sequence statistics (entropy, run-length) no dataset-level statistic

5.3 alone forces the split.

What meg/CLAUDE.md §9 already decided for you

Split policy: implement /project/data/things-meg/splits/by_image.json before training any tokenizer; trial-index splits leak THINGS image identity across train/val. Owner: whoever ships phase 1.

That's you. The doc has already committed to: - by-image split, not by-trial — because THINGS repeats each image across multiple trials, so a random trial split leaks image identity (same image in train and test → probe accuracy gets a free boost from memorization of image-specific noise patterns). - The split manifest lives at /project/data/things-meg/splits/by_image.json on the Modal Volume.

Concrete recipe for Phase 1

  1. Build splits/by_image.json first. Take the 22,449 unique image IDs, split ~80/10/10 by image, materialize as {train: [image_ids], val: [...], test: [...]}. Save to the Modal Volume.

  3. test_tokenizer.py: run on the test (or val) split. §5.1, §5.2, §5.4 on the held-out trials; §5.3 trains the logistic regression on train-split tokens and reports accuracy on test-split tokens.

Fair point — let me be explicit about where the GPU boundary lives.

Where numpy vs torch goes (and why)

There are three layers in the pipeline. Only the middle one matters for GPU throughput:

Layer Lib Why
Disk → RAM (MNE reading .fif) numpy MNE has no torch backend; it always returns float64 numpy. There's no way around this.
Math (calibration, tokenize, decode, harness) torch, GPU Hundreds of millions of float ops per pass — this is exactly where we want torch on a GPU.
Trial metadata (image_ids, subject_id, split index arrays) numpy Just int label arrays. They're never on GPU, never get tensor math applied to them — they're used as Python indices into the data tensor. Forcing torch here just costs .tolist() conversions.

In my current data.py, I already do torch.from_numpy(...).to(dtype) immediately on the MEG data path — X is torch from the moment it leaves MNE. The numpy you see is only for image_ids and trial-index arrays.

What the GPU shift simplifies for calibration

Your push is actually a useful simplification: instead of the streaming histogram approach I was vaguely planning for calibration, just subsample and do exact percentile on GPU. Here's the math:

  • Calibration needs q_lo, q_hi per channel (clip percentiles) — quantile estimation on ~24k trials × 281 samples per channel.
  • Subsample 1000 train trials → load as torch.Tensor (1000, 271, 281) ≈ 305 MB on GPU.
  • torch.quantile(X.permute(1,0,2).reshape(C,-1), [q_lo,q_hi]/100, dim=1) → (2, 271) on GPU, milliseconds.
  • The scaler s is derived from max(|q_lo_val|, |q_hi_val|) — no separate max|x| pass needed, since clipping caps it anyway.

So no streaming needed, no histogram pass — just one subsample + one GPU reduction. Simpler AND faster.

This is exactly the right system-level question to ask. Let me walk through what each tokenizer preserves vs throws away on the spatial and temporal axes — and crucially, what role the downstream 4M transformer plays.

What "spatiotemporal information" decomposes into for MEG

A trial is (C, T) = (271, 281). There are four kinds of information in this tensor:

What it captures Example
Per-sample amplitude the value at one (c, t) "channel 17, sample 142 = +3.2 pT"
Temporal context (per-channel) how a single channel evolves over time "channel 17 has an alpha oscillation; an N170 peak at t≈170ms"
Spatial context (per-time) which channels co-activate at one timepoint "at t=200ms, frontal sensors are negative and occipital are positive"
Spatiotemporal interactions patterns spanning both axes "a traveling wave moves from occipital to frontal over 150ms"

An ideal MEG tokenizer encodes as many of these into the token value itself as possible, so the downstream transformer doesn't have to rediscover them from scratch.

μ-transform: throws away BOTH spatial and temporal context

Reading the code path (encoder.py → quantizer.py):

token[b, c, t] = quantize(μ_law(x[b, c, t] / scaler[c]))

Each output token depends only on a single input sample. Not its neighbors in time, not its neighbors in space.

Info type μ-transform Why
Per-sample amplitude ✓ (preserved with bin-width quantization error) this is literally all the tokenizer does
Temporal context (per-channel) ✗ token at (c, t) doesn't know what (c, t-1) looked like
Spatial context (per-time) ✗ token at (c, t) doesn't know what (c+1, t) looked like
Spatiotemporal interactions ✗ can't see anything beyond the one sample

This is precisely why the §5.3 probe degenerated to ≈ random in our last run — each token carries no contextual information, so bag-of-codes histograms can't tell concepts apart.

That said, μ-transform doesn't destroy the information — it just doesn't encode it. The token position (c, t) is still preserved in the output grid. The 4M transformer can in principle attend across positions and learn cross-(c,t) patterns from scratch — but that costs more transformer layers and more training data.

Cho2026: recovers temporal context, still throws away spatial

From meg/CLAUDE.md §5 and the reference config:

encoder: GRU(rnn_n_units=128) → Dense → LayerNorm → V-way logits
decoder: bank of V 1-D conv kernels over time
applied: per-channel (the GRU has no channel-conditioning)

Each token at (c, t) is produced by a GRU that has consumed that channel's full time history up to t (causal) or the whole channel's time series (non-causal). The decoder kernels span multiple timesteps. So:

Info type Cho2026 Why
Per-sample amplitude ✓ reconstruction MSE objective
Temporal context (per-channel) ✓ GRU + multi-timestep conv kernels
Spatial context (per-time) ✗ GRU is per-channel; no channel-mixing layer
Spatiotemporal interactions ✗ same reason

So Cho2026 is a strict improvement over μ-transform on the temporal axis — each token encodes "what waveform pattern is this channel doing right now, given its recent history" — but it still treats the 271 channels as independent. No attention. No cross-channel convolution. The CLAUDE.md note about "weights can be reused on sensor data" is precisely because they're channel-agnostic.

What about cross-channel info?

Cho2026 leaves that to 4M to learn. The token grid is still (C, T) = (271, 281) and 4M's transformer will get position embeddings for both axes, so it can attend across channels at the same timepoint, or across time at the same channel, or anything in between. But the tokenizer doesn't pre-bake those patterns.

This is a deliberate design choice in Cho2026 — keeps the tokenizer cheap and modality-flexible (you can swap channels in/out at inference without retraining the tokenizer).

What an "ideal" MEG tokenizer would do

The CLAUDE.md Stage 2 plan (neural_tokenizers/CLAUDE.md §4) actually flags this:

"Backbone: a 1-D CNN over the time axis or a temporal-patch transformer is the right shape... Or LABRAM's architecture: temporal-patch transformer + channel embeddings + multi-objective loss."

The LABRAM-style approach gets you both axes: 1. Patch tokenization — each token represents a (small-channel-block × short-time-window) tile, like ViT patches over the (C, T) image. Token value encodes a local spatiotemporal pattern, not just one sample. 2. Channel embeddings — each channel index gets a learned embedding, so the transformer knows "this is a frontal sensor" vs "this is an occipital sensor". Cross-channel attention can then learn topographic patterns. 3. Attention-based encoder — captures arbitrary cross-(c,t) dependencies, not just local conv neighborhoods.

This is what would explicitly encode all four info types into the token. Phase 3 (BrainOmni) sounds like it goes further but the doc is light on details — we'd evaluate it once Phase 2 lands.

The summary table

Tokenizer Per-sample amplitude Temporal context Spatial context Spatiotemporal Who learns cross-(c,t)?
Phase 1: μ-transform ✓ ✗ ✗ ✗ the downstream 4M transformer, from scratch
Phase 2: Cho2026 ✓ ✓ ✗ ✗ 4M, but with a head start on temporal patterns
Phase 3: BrainOmni / LABRAM-style ✓ ✓ ✓ partially 4M, with much less to discover

Practical implication for our §5.3 probe results

The reason μ-transform ≈ random on the §5.3 probe with bag-of-codes: - Bag-of-codes throws away token position → throws away the only contextual information μ tokens carry - μ token values alone have no concept information

For Cho2026 the expected story is: - Token values DO carry temporal context → bag-of-codes might actually pick up some signal - AND Cho2026 exposes tokens_to_embedding (each token has a learned embedding vector) → the probe switches to _embed_and_mean (probe.py:106-119) which gives much richer per-trial features - We should expect tokens > random ≫ chance, finally

For an LABRAM-style spatial-aware tokenizer: - Each token encodes both axes → even pooled features carry concept info - Should be the clearest §5.3 winner
