First: what "registering a type" actually controls

There are two independent knobs, and people conflate them:

  1. type decides mechanics, not geometry:
  2. which masking branch runs (img → image_mask, scattered/parallel; seq_token → sequence_token_mask, contiguous + text sentinels + autoregressive),
  3. which decoder path in cat_decoder_tensors runs (img/other → parallel MaskGIT-style; seq* → left-to-right autoregressive),
  4. a few special cases (max_tokens auto-compute for img, mod_is_img in the budget math).

  5. type does not encode position at all.

  6. encoder_embedding / decoder_embedding is the nn.Module that turns token-ids → vectors and adds the positional embedding. This is where spatial structure lives:

  7. SequenceEncoderEmbedding → 1D sincos positions 0…L-1 (flat order),
  8. ImageTokenEncoderEmbedding → 2D sincos positions over an H×W grid (preserves grid adjacency),
  9. a custom module → any scheme you want (3D, factorized, RVQ-aware).

So "stay loyal to the spatial structure" is primarily an embedding choice, with a type picked to give it a compatible masking/decoder path.

What 4M gives you natively: 1D structure (seq*) and 2D structure (img, the only native spatial type, parallel decode). There is no native 3D.

What your code does today

NeuralTrialSampleTransform does arr[idx].reshape(-1) → (16,8,4) becomes a flat (512,), fed to SequenceEncoderEmbedding with 1D positions. So today it's structurally wrong twice: it (a) discards the 16×8 grid adjacency, and (b) lays the 4 RVQ levels out as if they were 4 sequential neighbors, when they're really 4 descriptions of one position.

Option How you'd register What the model "sees" Cost
A. Flat 1D (today) reshape→(512,), SequenceEmbedding, 1D pos. type = parallel-grid or seq_token 512 tokens in fixed flattened order; position = index. No grid, no RVQ awareness trivial
B. Fold to a 2D grid reshape (16,8,4)→ a 2D (H,W) of single tokens (e.g. (16, 32)), type: img + ImageTokenEmbedding a 2D positional grid + parallel MaskGIT decode small, but you pick a factorization and one folded axis mixes RVQ-depth with space

Which dim to collapse, and why

Axis Meaning Collapse?
16 — latent source variables (C') spatial (virtual brain sources) No — real structure
8 — temporal latent steps (W) time (SEANet downsampled) No — real structure
4 — RVQ layers (N_q) residual quantizer depth Yes — collapse, by summing

The 4 RVQ layers are not a fourth spatial axis — they're 4 codes describing the same (source, time) position, each refining the residual of the previous. The mathematically faithful way to combine them is exactly how RVQ reconstructs a vector: look up each layer's code in its own codebook and sum the 4 embedding vectors. So you "collapse" the 4 into the embedding, not into the sequence. That leaves a clean 16 × 8 spatiotemporal grid (128 positions), which 4M's transformer handles natively as a token grid.

So Option C becomes concrete and not that heavy: - Per position: emb = Σ_{q=0..3} Codebook_q[token[s,t,q]] (4 separate 512-entry codebooks, summed). - Positions: factorized/axial positional embedding — and here's a nuance worth respecting: the 16 sources have no inherent order (they're latent sources), so a learned positional embedding fits them; the 8 time steps are ordered, so sincos fits them.
