Fair ask. Let me make the actual argument rather than handwave. I'll separate "what could shift" from "what matters for this pipeline."

What could shift between train (a–w-ish) and val (last ~12%, ends at zucchini)

Concretely the val concepts will be alphabetically biased toward letters near the end of the alphabet. There are three places a distribution shift could bite you:

1. Image RGB pixel statistics — almost certainly safe

Pixel distributions (color histograms, edge statistics, object scale) don't track the first letter of the object's English name. "zebra", "yacht", "windmill", "watermelon" are not pixel-statistically distinguishable as a group from "apple", "boat", "car" as a group. The reason this would fail is if THINGS used a different photographer / lighting setup / image source for some alphabetical subrange — but THINGS was collected as one homogeneous corpus, so this isn't a thing.

2. Concept frequency / category coverage — partially biased

English noun frequency is skewed: s, c, p, b, a, m are dense letters; q, x, z are sparse. So val will be biased toward less common starting letters, which slightly correlates with more "exotic" concepts (zebra, xylophone, yacht). But these aren't visually weird — they're still concrete photographable objects with normal lighting. The categorical coverage of val (animals / food / tools / vehicles / etc.) is unlikely to be wildly skewed because each letter contains a mix.

3. MEG/EEG signal statistics — basically safe

This is the one that matters most for your downstream tokenizers. Raw MEG/EEG response variance is dominated by: - subject-level brain anatomy - session-level noise (sensor drift, eye blinks, muscle artifacts) - evoked-response statistics that are largely concept-independent at the channel/frequency level

The brain doesn't have systematically different sensor-level signal distributions for "things starting with z" vs "things starting with a." Concept-specific information is in the pattern of activation across channels — not in the marginal channel statistics that a tokenizer's reconstruction loss is sensitive to. For tokenizer training (reconstruction + RVQ), val should be in-distribution.

Where this split would break a thing

  • A 1854-way concept classifier trained on train, evaluated on val. Val concepts that don't appear in train get 0% by construction. You can't train such a head with this split. The §5.3 linear probe in meg/CLAUDE.md already uses its own image-level split for exactly this reason — don't conflate the two.
  • Anything that does k-shot concept retrieval val→train. Same issue.

The split you're creating is for 4M's masked-token pretraining, which has no concept-classification head — it predicts masked modality tokens given other modality tokens. That objective is concept-agnostic in the same way next-token prediction in LLMs is identity-agnostic about characters in a sentence.

Why the split is also kind of better than image-level random

The alternative (image-level random) puts aardvark_01.jpg in train and aardvark_02.jpg in val. For a model that learns image embeddings, val is then trivial — the model just recognizes "I've seen 13 other photos of this same aardvark." Val loss looks great, generalization claim is empty. Concept-level holdout (your choice) is the standard rigorous answer to "did the model generalize, or memorize?"

When you'd want to revisit

  • If §5.3 probe numbers on the 4M-trained representations look suspicious, you'd want to additionally evaluate on a within-distribution held-out image set (image-level holdout). That's a different artifact, not a fix to this split.
  • If you discover empirically that val loss is way off train loss in a way that doesn't match held-out generalization, that's a signal the split is bad — but you'd want to measure this rather than predict it.

Easy escape hatch

Because the manifest JSON enumerates every (shard → image_id) edge, re-doing this as image-level random later is one Modal job: read all 27 tars, reshuffle, write 23 train + 4 val. The catalog JSON is invariant to that operation. So you're not locked in.

Good — the goal flips my recommendation. Let me reason about this in terms of which question the eval is answering, because that's what determines the right split.

What "neural→image with 4M" actually requires at eval time

4M is trained with random masked-token prediction across modalities. At inference for neural→image, you: 1. Feed the full MEG/EEG token sequence as observed conditioning 2. Mask all RGB tokens 3. Sample/decode RGB tokens → run them through the RGB tokenizer's decoder to get pixels 4. Compare the generated image to the ground-truth stimulus

The eval question is: "Given MEG from a held-out trial, can the model reproduce the image the subject was actually looking at?"

The three splits answer different questions

Split Eval question Hardness for 4M Standard in literature?
Image-level random (each concept in both splits, different exemplars held out) "Can the model decode the specific image, having seen other exemplars of the concept?" Realistic — model has a concept template, must use MEG to pick the right exemplar / pose / variant Yes — MindEye, MindEye2, Takagi & Nishimoto, Scotti et al., BrainDiffuser, the THINGS-EEG decoding papers all use this
Shard-level concept-holdout (whole concepts never seen in train) "Zero-shot: can the model produce a zucchini given only MEG, never having seen a zucchini at all?" Very hard — model has no template; can at best produce shared low-level features (color, shape) Used as secondary eval ("zero-shot decoding"), never primary
Image-level random + concept-holdout (hybrid) Both, separately reported Two evals to run The most rigorous papers do both

Why image-level is the right primary split for your goal

  1. The model can't generate what it has no template for. With concept-holdout, val contains concepts the model has never paired with any neural signal. 4M's mechanism is associative — it learns (MEG_pattern, RGB_pattern) joints. Zero exposure to "zucchini-MEG ↔ zucchini-RGB" means the model has no path to a zucchini image, even with perfect MEG decoding. You'd be evaluating the model on a task it wasn't trained for.

  2. Comparison to literature. Every brain-decoding paper I know of (MindEye, Takagi, BrainDiffuser, the THINGS-EEG decoding line) uses image-level holdout for their primary reported number. If you ever benchmark against them, your eval needs to be apples-to-apples or your numbers aren't comparable.

  3. The "concept memorization" worry I flagged earlier doesn't apply to generation. I argued earlier that image-level holdout lets the model "trivially recognize the concept." For a classifier that's a problem. For a generator, that's exactly the desired behavior — you want the model to use the concept-level signal from MEG to produce a recognizable image. Memorization-of-concept is the mechanism, not a bug.

  4. MEG sample-level information is mostly concept-level anyway. The §5.3 linear probe results in meg/CLAUDE.md §9 show that neither μ-transform nor BrainOmni tokens currently carry image-specific information beyond chance. The realistic ceiling is "decode the concept, generate an exemplar." That requires having seen the concept.

What the cost difference actually is

  • Shard-level (your current pick): ~minutes of Modal time. os.rename 4 tar files. Volume state minimally disturbed.
  • Image-level random: ~30 min Modal job. Read all 27 tars (~5 GB), shuffle deterministically (seed=0), repack 23 train + 4 val tars (~5 GB writes), commit. The catalog and manifest JSONs are produced the same way — they don't care which images are in which shard, only which split each image_id belongs to.

The cost is real but not huge — and it's a one-time cost you pay now to avoid a misaligned eval for the entire lifetime of the project.

My recommendation

Switch to image-level random split (seed=0), 85/15 by image count. Concretely:

  • Total: 26,107 THINGS images
  • Train: ~22,191 images (85%) packed into 23 shards of 1000 (last one partial)
  • Val: ~3,916 images (15%) packed into 4 shards of 1000 (last one partial)
  • things_catalog.json is unchanged (image_id ↔ filename is split-invariant)
  • train_manifest.json / val_manifest.json list exactly which image_ids go in each shard

Within each split, shards are packed by sorted image_id so the image_id // 1000 = shard_id math still works within a split. (Cross-split, the math doesn't work but the manifest is the source of truth.)

Keep the concept-holdout split as an optional secondary eval. When you want to measure zero-shot concept transfer later, you can produce a third split (val_unseen_concept) by holding out a few alphabetical-shard's worth of concepts. Cheap to add later from the same catalog.

Why this module is import-clean (no `modal`, no `torch`):
  The Modal entrypoint in `modal_things_repack.py` runs in a container with

  building manifests, computing the split, planning the repack) is here

  not a dependency of the logic.

System-level note on the split policy:
  Splits are at the image level (each THINGS concept appears in both train
  and val, different exemplars held out) because the project target is
  neural-to-image decoding with 4M. Standard brain-decoding evals
  (MindEye, Takagi & Nishimoto, BrainDiffuser, the THINGS-EEG/MEG decoding
  line) hold out images, not concepts. Concept-level holdout is a
  secondary "zero-shot decoding" eval, not the primary one. If we ever
  want that, we can derive it from the same catalog without rerunning the
  repack — see `modal_things_repack.py` docstring.

THINGS shard layout (verified 2026-05-23 against /project/data/train/things/rgb):
  Each tar contains paired entries `<image_id>.jpg` + `<image_id>.txt`.
  - `image_id` is a 9-digit zero-padded alphabetical rank of the THINGS
    filename (e.g. `aardvark_01b.jpg` → `000000001`).
  - `<image_id>.txt` body is the canonical THINGS filename.
  - 26,107 unique images total across 27 shards × 1000 (last partial).
"""

from __future__ import annotations

import io
import os
import random
import re
import tarfile
from dataclasses import dataclass
from typing import Iterable


# ---------- constants -----------------------------------------------------

SHARD_NAME_RE = re.compile(r"^shard_(\d{3})\.tar$")
IMAGE_ID_TXT_RE = re.compile(r"^(\d{9})\.txt$")
IMAGE_ID_JPG_RE = re.compile(r"^(\d{9})\.jpg$")
IMAGES_PER_SHARD = 1000


# ---------- catalog construction ------------------------------------------

def extract_shard_contents(tar_path: str) -> dict[str, str]:
    """Read a single THINGS shard tar; return ``{image_id: filename}``.

    Reads only the `.txt` entries (filename strings); ignores the `.jpg`
    bytes. Cheap enough to call on all 27 shards back-to-back when building
    the catalog.
    """
    out: dict[str, str] = {}
    with tarfile.open(tar_path, "r") as tar:
        for member in tar:
            m = IMAGE_ID_TXT_RE.match(member.name)
            if not m:
                continue
            f = tar.extractfile(member)
            if f is None:
                continue
            out[m.group(1)] = f.read().decode("utf-8").strip()
    return out


def build_catalog(image_id_to_filename: dict[str, str]) -> dict:
    """Return the global THINGS catalog JSON payload.

    The catalog is split-invariant — any future modality (MEG, EEG, fMRI)
    consults this single file to resolve image_id ↔ filename, regardless
    of which split each image lives in.
    """
    ordered = dict(sorted(image_id_to_filename.items()))
    return {
        "version": "1",
        "n_images": len(ordered),
        "id_format": (
            "9-digit zero-padded alphabetical rank of the THINGS image filename"
        ),
        "image_id_to_filename": ordered,
    }


# ---------- split logic ----------------------------------------------------

def image_level_split(
    image_ids: Iterable[str],
    val_frac: float = 0.15,
    seed: int = 0,
) -> tuple[list[str], list[str]]:
    """Deterministic image-level shuffle into (train, val).

    Each THINGS concept's ~14 images get distributed across both splits.
    This matches the brain-decoding-eval convention (image-level holdout):
    the model sees the concept at train time and must use the held-out
    trial's neural signal to pick the right exemplar at inference.

    Determinism: sorts ids first so callers can pass any iterable and still
    get the same split for a given seed.
    """
    if not (0.0 < val_frac < 1.0):
        raise ValueError(f"val_frac must be in (0, 1); got {val_frac}")
    ids = sorted(image_ids)
    n_val = round(len(ids) * val_frac)
    rng = random.Random(seed)
    shuffled = list(ids)
    rng.shuffle(shuffled)
    val_set = set(shuffled[:n_val])
    train_ids = [i for i in ids if i not in val_set]
    val_ids = [i for i in ids if i in val_set]
    return train_ids, val_ids


def pack_into_shards(
    image_ids: list[str],
    images_per_shard: int = IMAGES_PER_SHARD,
) -> list[list[str]]:
    """Sort `image_ids` and chunk into fixed-size shards (last may be partial).

    Sorting (rather than keeping random order) preserves the
    `image_id // images_per_shard == shard_index` math within a split,
    which is useful for debugging and for any modality that wants to
    locate a shard from an image_id without consulting the manifest.
    """
    ordered = sorted(image_ids)
    return [
        ordered[i : i + images_per_shard]
        for i in range(0, len(ordered), images_per_shard)
    ]


def build_split_manifest(
    split_name: str,
    shards: list[list[str]],
    shards_subpath: str,
) -> dict:
    """Build the per-split manifest JSON.

    Shape:
      {
        "split": "train" | "val",
        "n_shards": int,
        "n_images": int,
        "images_per_shard": int,
        "shards_subpath": str,                    # e.g. "things/rgb"
        "shards": {"shard_NNN": {"n_images": ..., "image_ids": [...]}}
      }
    """
    shards_payload: dict[str, dict] = {}
    total = 0
    for n, ids in enumerate(shards):
        shard_id = f"shard_{n:03d}"
        shards_payload[shard_id] = {
            "n_images": len(ids),
            "image_ids": list(ids),
        }
        total += len(ids)
    return {
        "split": split_name,
        "n_shards": len(shards),
        "n_images": total,
        "images_per_shard": IMAGES_PER_SHARD,
        "shards_subpath": shards_subpath,
        "shards": shards_payload,
    }


# ---------- repack execution ----------------------------------------------

@dataclass(frozen=True)
class SourceLocation:
    """Where a single image_id currently lives in the source shard layout."""

    src_tar_path: str
    image_id: str


def index_source_shards(shard_paths: list[str]) -> dict[str, SourceLocation]:
    """Walk every source shard once; return ``{image_id: SourceLocation}``.

    Errors loudly if the same image_id appears in more than one source tar —
    that would mean the shards are inconsistent and the repack is unsafe.
    """
    locations: dict[str, SourceLocation] = {}
    for p in shard_paths:
        for image_id in extract_shard_contents(p):
            prior = locations.get(image_id)
            if prior is not None:
                raise ValueError(
                    f"image_id {image_id} found in both "
                    f"{prior.src_tar_path} and {p}"
                )
            locations[image_id] = SourceLocation(p, image_id)
    return locations


def write_shard_from_locations(
    out_path: str,
    image_ids_in_order: list[str],
    source_index: dict[str, SourceLocation],
) -> None:
    """Build a new tar at `out_path` from existing source shards.

    Copies `.jpg` and `.txt` bytes verbatim (no JPEG re-encoding). Groups
    reads by source tar so each source is opened at most once per call.
    Writes to `<out_path>.tmp` then atomically renames into place.
    """
    by_src: dict[str, list[str]] = {}
    for image_id in image_ids_in_order:
        src = source_index[image_id].src_tar_path
        by_src.setdefault(src, []).append(image_id)

    extracted: dict[str, tuple[bytes, bytes]] = {}  # id -> (jpg, txt)
    for src_path, ids in by_src.items():
        needed = set(ids)
        with tarfile.open(src_path, "r") as src:
            for member in src:
                if member.name.endswith(".jpg"):
                    iid = member.name[:-4]
                    if iid not in needed:
                        continue
                    f = src.extractfile(member)
                    if f is None:
                        continue
                    cur = extracted.get(iid, (b"", b""))
                    extracted[iid] = (f.read(), cur[1])
                elif member.name.endswith(".txt"):
                    iid = member.name[:-4]
                    if iid not in needed:
                        continue
                    f = src.extractfile(member)
                    if f is None:
                        continue
                    cur = extracted.get(iid, (b"", b""))
                    extracted[iid] = (cur[0], f.read())

    missing = [i for i in image_ids_in_order if i not in extracted]
    if missing:
        raise RuntimeError(
            f"write_shard_from_locations: {len(missing)} ids missing from "
            f"sources (first 5: {missing[:5]})"
        )

    tmp_path = out_path + ".tmp"
    with tarfile.open(tmp_path, "w") as out:
        for image_id in image_ids_in_order:
            jpg_bytes, txt_bytes = extracted[image_id]
            if not jpg_bytes or not txt_bytes:
                raise RuntimeError(
                    f"missing payload for {image_id}: "
                    f"jpg_bytes={len(jpg_bytes)}, txt_bytes={len(txt_bytes)}"
                )
            for name, data in (
                (f"{image_id}.jpg", jpg_bytes),
                (f"{image_id}.txt", txt_bytes),
            ):
                info = tarfile.TarInfo(name=name)
                info.size = len(data)
                out.addfile(info, io.BytesIO(data))
    os.replace(tmp_path, out_path)

    """Write a tar at `path` with paired <id>.jpg + <id>.txt entries."""
    with tarfile.open(path, "w") as tar:
        for image_id, filename in items.items():
            txt = filename.encode("utf-8")
            info = tarfile.TarInfo(f"{image_id}.txt")
            info.size = len(txt)
            tar.addfile(info, io.BytesIO(txt))
            jpg = b"\xff\xd8\xff\xe0FAKE" + image_id.encode()
            info = tarfile.TarInfo(f"{image_id}.jpg")
            info.size = len(jpg)
            tar.addfile(info, io.BytesIO(jpg))

    image=repack_image,
    volumes={PROJECT_MOUNT: project_volume},
    cpu=4.0,
    memory=16 * 1024,
    timeout=60 * 90,
)
def repack_remote() -> dict:
    """Phase 2: build new shards in staging, validate, then atomic swap."""
    if not os.path.exists(PLAN_PATH):
        raise FileNotFoundError(f"Run plan first; {PLAN_PATH} missing")
    with open(PLAN_PATH) as fh:
        plan_payload = json.load(fh)
    with open(TRAIN_MANIFEST) as fh:
        train_manifest = json.load(fh)
    with open(VAL_MANIFEST) as fh:
        val_manifest = json.load(fh)

    src_paths = plan_payload["src_paths"]
    missing_src = [p for p in src_paths if not os.path.exists(p)]
    if missing_src:
        # If staging already has everything we need, this is a resume after
        # source-delete crash — fall through to swap. Otherwise abort.
        n_staged = 0
        for d in (STAGING_TRAIN, STAGING_VAL):
            if os.path.isdir(d):
                n_staged += len(
                    [f for f in os.listdir(d) if f.endswith(".tar")]
                )
            else:
                n_staged += 0
        expected_staged = train_manifest["n_shards"] + val_manifest["n_shards"]
        if n_staged != expected_staged:
            raise FileNotFoundError(
                f"sources missing ({len(missing_src)} gone) and staging "
                f"incomplete ({n_staged}/{expected_staged}). Aborting."
            )
        print("[repack] sources gone but staging complete; jumping to swap")
    else:
        print(f"[repack] indexing {len(src_paths)} source shards...")
        src_index = things_manifest.index_source_shards(src_paths)
        print(f"[repack] indexed {len(src_index)} image_ids")

        os.makedirs(STAGING_TRAIN, exist_ok=True)
        os.makedirs(STAGING_VAL, exist_ok=True)

        for split_name, manifest, staging in (
            ("train", train_manifest, STAGING_TRAIN),
            ("val", val_manifest, STAGING_VAL),
        ):
            for shard_id, info in manifest["shards"].items():
                out_path = os.path.join(staging, f"{shard_id}.tar")
                if os.path.exists(out_path):
                    print(f"[repack] {split_name}/{shard_id}: exists, skip")
                    continue
                things_manifest.write_shard_from_locations(
                    out_path, info["image_ids"], src_index
                )
                print(
                    f"[repack] {split_name}/{shard_id}: "
                    f"{info['n_images']} imgs, "
                    f"{os.path.getsize(out_path) / 1e6:.1f} MB"
                )
        project_volume.commit()

    # Validate staging counts.
    n_train_staged = len(
        [f for f in os.listdir(STAGING_TRAIN) if f.endswith(".tar")]
    )
    n_val_staged = len(
        [f for f in os.listdir(STAGING_VAL) if f.endswith(".tar")]
    )
    if n_train_staged != train_manifest["n_shards"]:
        raise RuntimeError(
            f"staged {n_train_staged} train tars, manifest expects {train_manifest['n_shards']}"
        )
    if n_val_staged != val_manifest["n_shards"]:
        raise RuntimeError(
            f"staged {n_val_staged} val tars, manifest expects {val_manifest['n_shards']}"
        )
    print(f"[repack] staging OK: {n_train_staged} train + {n_val_staged} val shards")

    # Swap.
    print("[repack] swapping into place...")
    surviving_sources = [p for p in src_paths if os.path.exists(p)]
    for p in surviving_sources:
        os.remove(p)
        print(f"[repack]   rm {p}")
    os.makedirs(DST_TRAIN_RGB, exist_ok=True)
    os.makedirs(DST_VAL_RGB, exist_ok=True)
    for f in sorted(os.listdir(STAGING_TRAIN)):
        os.replace(
            os.path.join(STAGING_TRAIN, f), os.path.join(DST_TRAIN_RGB, f)
        )
    for f in sorted(os.listdir(STAGING_VAL)):
        os.replace(os.path.join(STAGING_VAL, f), os.path.join(DST_VAL_RGB, f))
    shutil.rmtree(STAGING_ROOT, ignore_errors=True)
    project_volume.commit()

    return {
        "train_n_shards": n_train_staged,
        "val_n_shards": n_val_staged,
        "sources_removed": len(surviving_sources),
    }
